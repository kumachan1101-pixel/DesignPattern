# 第0章　この本の読み方
―― デザインパターンは「考えた結果」に過ぎない

---

## なぜ「パターンを覚えても使えない」のか

ソフトウェア設計を学ぼうとすると、必ずと言っていいほど
「GoFのデザインパターン」に出会います。
本で学び、構造図を頭に入れ、いざ自分のコードに使おうとしたとき——

「どこに適用すればいいのか、わからない。」
「無理に使ってみたら、かえってコードが複雑になった。」

私自身、同じ壁に何度もぶつかりました。
パターンの名前と図は頭に入った。でも、
目の前の問題にどう当てはめればいいのか、
まるでわからなかったのです。

その感覚、うまく伝わっているでしょうか。

なぜ、実績のある優れた設計手法が、
時にはコードをより複雑にしてしまうのか。

理由はシンプルです。
**パターンを「最初から目指すべき答え」として扱っているから**です。

デザインパターンは、先人たちが泥臭い現場で問題に向き合い、
いくつかの選択肢を天秤にかけ、
「この状況ではこれが一番割に合う」と判断した
**決断の結果**として生まれたものです。

結果だけを真似ても、状況が違えばうまく機能しません。
大切なのは、その結果に至るまでの**思考のプロセス**を体験することです。

---

## 先人たちが使っていた「5ステップの思考プロセス」

先人たちも、私たちと同じように混沌としたコードの前で頭を抱えていました。
彼らが問題を解決するとき、意識的かどうかはともかく、
以下の5つのステップを踏んでいたと解釈することができます。

### ステップ1：現状把握

「今のシステムはどうなっていて、今何を求められているか」を整理する。

仕様・コードの構造・今日届いた変更要求を、
ひとつの「状況」としてまとめて把握することが出発点です。
コードを「読む」のではなく「観察する」感覚です。

### ステップ2：課題の発見

「変更要求を受けて、何が難しいのか」を具体化する。

変更要求を頭の中で試しに実装してみます。
「あ、これをやろうとするとここが壊れる」
「この引数がまた増える……」という
**詰まりの感覚**を言葉にするステップです。

まだ答えを出しません。
「難しい」という事実をはっきりさせることが目的です。

### ステップ3：原因特定

「なぜ難しいのか」の根本を突き止める。

詰まりの感覚には、必ず構造的な原因があります。
多くの場合、それは以下のいずれかです：

- 変わり続けるものと変わってほしくないものが同じ場所にいる
- 使う側が知らなくていいことまで知っている
- 状態と振る舞いの管理が1か所に集まっている
- 生成の責任と利用の責任が混在している

原因が特定できると、「ここを変えれば楽になる」という
仮説が立てられます。

### ステップ4：対策案の検討

「どう解決するか」を、自然な発想の順に試みる。

最初に思いつく素朴な案から始めます。
その限界に気づき、発想を転換し、
より良い構造へたどり着くプロセスが、
このステップの核心です。

答えを「知る」のではなく、「辿り着く」体験をします。

### ステップ5：天秤にかける・決断する

「どの案が、この状況で最も割に合うか」を判断する。

判断には基準が必要です。
何を重視するか（テストの容易さ？変更の局所性？チームの分担？）を
先に言葉にしてから比較します。

テストは「設計の質を測るセンサー」として使います。
テストが書きにくい案は、まだ原因が解消されていないサインです。

そして最後に「費用対効果」を考えます。
変化が今後も続くなら投資の価値がある。
変化が止まるなら、シンプルなままにしておく方が賢明かもしれない。
**設計に絶対の正解はありません。**

## このプロセスの結果として生まれたものが「パターン」

具体的に見てみましょう。
「新しい割引ルールを追加するたびに計算処理のコードが壊れる」
という問題に、このプロセスを適用した場合です。

| ステップ | 先人たちの思考 |
|:---|:---|
| **1. 現状把握** | ECサイトの決済計算モジュール。法人・一般それぞれに割引ルールがあり、今日また新ルール追加の依頼が来た |
| **2. 課題の発見** | 変更要求を実装しようとすると、calculateTotal関数の引数が増え続ける。B2Cを変えるとB2Bのテストまで不安になる |
| **3. 原因特定** | 「変わり続ける割引ルール」と「変わってほしい計算の骨格」が同じ関数の中にいる |
| **4. 対策案検討** | まず「引数で渡す」案を試みる → 複雑な条件を数値1つで表現できず限界に気づく → 「ルールごと渡す」発想へ転換 |
| **5. 天秤・決断** | テストで比較。ルールを独立させると各部品を単独でテストでき、変更も局所化される。このビジネスは今後もルールが増える → 投資の価値あり |

この思考の結果として生まれた構造に、
後の人たちが「**Strategyパターン**」という名前を付けました。

パターンは**最初から目指す形ではなく、このプロセスを踏んだ結果のラベル**です。

## この本の読み方

### どの章から読んでも大丈夫です

第1章以降、各章は完全に独立しています。
興味のあるパターンから読み始めてください。

各章では、毎回同じ5ステップの思考プロセスを体験します。
章の違いは「今回の変化の種類が何か」だけです。

### 各章で出会う「変化の種類」と原因の切り口

| 章 | パターン | 今回の変化の種類 | 原因の切り口 |
|:---|:---|:---|:---|
| 第1章 | Strategy | 実行する振る舞いが変わる | A：変化の混在 |
| 第2章 | Facade | 複雑な依存関係を隠したい | B：依存の過多 |
| 第3章 | State | 状態によって振る舞いが変わる | C：状態と振る舞いの混在 |
| 第4章 | Template Method | 処理の一部のステップが変わる | A：変化の混在 |
| 第5章 | Command | 実行する操作が変わる | A：変化の混在 |
| 第6章 | Decorator | 機能の組み合わせが変わる | A：変化の混在 |
| 第7章 | Observer | 通知を受け取る相手が変わる | A：変化の混在 |
| 第8章 | Factory Method | 作るオブジェクトが変わる | D：生成と利用の混在 |

「原因の切り口」が「A：変化の混在」に集中していますが、
それは現実の設計問題の多くがこの構造を持つからです。
B・C・Dが少ないことは「その種類の問題は特定の状況に現れる」
という情報でもあります。

### パターン名は「後から出てきます」

各章では、あなたが自分で考えを進めた後に、
「この構造を、先人たちは〇〇パターンと呼んでいます」という形で
名前が登場します。

名前を覚えることが目的ではありません。
**「次の現場で同じ問題に出会ったとき、同じ5ステップで考えてみよう」**
と思えることが、この本を読む意味です。

---

*間違えても大丈夫です。設計は一度決めたら終わりではなく、
状況が変わればまた考え直せばいい、という軽さで向き合ってほしいと思います。
一つの参考として受け取っていただければと思います。*
