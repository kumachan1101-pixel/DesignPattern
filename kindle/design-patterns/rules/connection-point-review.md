# 接続点を見直すための共通ルール

## 目的

本書では、設計を固定的な接続分類へ当てはめない。

着目するのは、変更要求が通過する**接続点**である。接続点とは、クラスや処理を
分けたときに、値・型・呼び出し・生成責任・処理順序などを受け渡す境界を指す。

## フェーズ4：原因分析で確認すること

1. 変わる理由が異なる判定条件・計算式・処理順序・生成判断が、同じクラスやメソッドに混在していないか
2. 呼び出し元が、相手のクラス名・生成方法・呼び出し順序をどこまで知っているか
3. 一方を変更したとき、変更しない側まで修正・再テストが必要になるのはなぜか

4-3の「確認すること」は、固定リストとして先に決めない。4-1/4-2で言語化した原因から、次の順に抽出する。

1. 原因文から、守りたい骨格を1文で書く。
2. 原因文から、変わる差分を1文で書く。
3. 差分を動かすために骨格側が知ってしまっている名前・条件・順序・型を拾う。
4. 原因タイプに応じて、接続点で見る抽象観点を決める。
5. 接続点に残す最小の約束を、値・型・操作・イベントとして書く。

原因によって抽象観点は変わる。

| 原因タイプ | 接続点で確認する抽象観点 |
|---|---|
| 判定/条件が漏れている | 条件・定数・選択基準 |
| 処理手順が漏れている | 呼び出し順・前後条件・失敗時分岐 |
| 生成判断が漏れている | 具体クラス名・生成条件・登録場所 |
| データ/状態が漏れている | 境界を流れる値・型・状態 |
| 通知/副作用が漏れている | 通知先・タイミング・成否の扱い |

旧来の4分類名は使わず、実際に抱えている判断・前提・処理順序と変更影響を文章で示す。

## フェーズ5：課題定義で確認すること

接続点は、クラス図の線やインターフェース名から探し始めない。フェーズ4の原因を、次の手順で「解くべき境界」に絞り込む。

1. 変更要求を1つ置く。
2. その要求で変えたい側を決める。
3. その要求では変えたくない側を決める。
4. 変えたい側と変えたくない側が、どのメソッド呼び出し・引数・戻り値・生成・イベントでつながっているかを見る。
5. そのつながりのうち、変更要求のたびに知識が漏れて修正が波及する場所を、解くべき接続点とする。

接続点ごとに次を整理する。

| 観点 | 問い |
|---|---|
| 受け渡すもの | 境界を越える値・型・操作・イベントは何か |
| 変わる側／守る側 | 境界のどちら側が変わり、どちら側を安定させたいか |
| 変化の種類 | その仕様は、どの機能・仕様変更で変わるか。担当者や関係者が分かる場合は補助情報として使う |
| 安定する契約 | 呼び出し元が本当に必要とする最小限の約束は何か |
| 漏れている判断や前提 | クラス名、生成方法、処理順序、条件分岐のうち何が漏れているか |
| 変更影響 | 要求を1つ当てたとき、どのファイルを修正・再テストするか |
| 受け入れ条件 | 変更要求を再適用したとき、修正がどの範囲（役割の言葉）に収まれば解けたと言えるか |

フェーズ5では対策の形を書かない。「どこへ移すか」「何を新設するか」は、フェーズ6の配置の決定である。受け入れ条件も「クラス1つと登録だけ」のような構造の形では書かない。

## フェーズ6：対策検討で確認すること

フェーズ6の決定は、分離（どの知識をどの境界で切るか）・配置（切り出した知識をどこへ置くか）・組み立て（誰が生成・所有・注入して繋ぎ、安定側は何を呼ぶか）の3つで構成する。候補はフェーズ5の受け入れ条件で評価し、読者が思いつく単純案も含めて比較する。

対策は固定分類から選ばない。小さな改善から順に試し、残る問題を確認する。

第0章の段階的進化アプローチを標準フローとする。ただし、ステップは一本道の作業手順ではなく、対策案を比較するための候補である。章の題材に合わない案を省略したり、順序を入れ替えたり、複数の接続点ごとに分岐させたりする場合は、次を本文で説明する。

- フェーズ5のどの課題を解くために、その順序で案を並べるのか
- 省略した案が、今回の接続点では論点外・効果不足・導入コスト過多のどれに当たるのか
- 複数接続点がある場合、どの接続点から解くのか、または接続点ごとに案を分岐させるのか
- 標準フローから外れても、各案の比較観点が変わっていないか

1. 名前を付けて関数へ切り出す
2. 変わる判定条件・処理・生成判断を別クラスへ移す
3. 複数実装を交換する必要がある場合だけ、安定した契約を定義する
4. 呼び出し元に複雑な手順を知らせたくない場合だけ、窓口や仲介役へ移す
5. 機能を組み合わせる必要がある場合だけ、コンポジションを導入する
6. 生成方法が変わる場合だけ、生成責任を組み立て箇所や生成メソッドへ移す

各案は「分類名」ではなく、次の形式で説明する。

- 何をどこへ移すか
- 接続点で何を受け渡すか
- 呼び出し元が知らなくてよくなること
- 代わりに増えるクラス・設定・実行時コスト
- 今回の変更要求を当てたときの修正箇所

## 禁止事項

- 旧来の4分類名
- 固定的な分類マトリクスや到達点
- パターンを固定的な接続分類へ無理に割り当てる説明
- 接続点を見ず、インターフェースや仲介役の有無だけで設計を評価すること
- 「知識」を具体化せず、曖昧なまま対策へ進むこと
