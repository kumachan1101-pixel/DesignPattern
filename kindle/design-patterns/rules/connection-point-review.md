# 接続点を見直すための共通ルール

## 目的

本書では、設計を固定的な接続分類へ当てはめない。

着目するのは、変更要求が通過する**接続点**である。接続点とは、クラスや処理を
分けたときに、値・型・呼び出し・生成責任・処理順序などを受け渡す境界を指す。

## フェーズ4：原因分析で確認すること

1. 変わる理由が異なる判定条件・計算式・処理順序・生成判断が、同じクラスやメソッドに混在していないか
2. 呼び出し元が、相手のクラス名・生成方法・呼び出し順序をどこまで知っているか
3. 一方を変更したとき、変更しない側まで修正・再テストが必要になるのはなぜか

4-3の「確認すること」は、固定リストとして先に決めない。4-1/4-2で言語化した原因から、次の順に抽出する。

1. 原因文から、守りたい骨格を1文で書く。
2. 原因文から、変わる差分を1文で書く。
3. 差分を動かすために骨格側が知ってしまっている名前・条件・順序・型を拾う。
4. 原因タイプに応じて、接続点で見る抽象観点を決める。
5. 接続点に残す最小の約束を、値・型・操作・イベントとして書く。

原因によって抽象観点は変わる。

| 原因タイプ | 接続点で確認する抽象観点 |
|---|---|
| 判定/条件が漏れている | 条件・定数・選択基準 |
| 処理手順が漏れている | 呼び出し順・前後条件・失敗時分岐 |
| 生成判断が漏れている | 具体クラス名・生成条件・登録場所 |
| データ/状態が漏れている | 境界を流れる値・型・状態 |
| 通知/副作用が漏れている | 通知先・タイミング・成否の扱い |

旧来の4分類名は使わず、実際に抱えている判断・前提・処理順序と変更影響を文章で示す。

## フェーズ5：課題定義で確認すること

接続点は、クラス図の線やインターフェース名から探し始めない。フェーズ4の原因を、次の手順で「解くべき境界」に絞り込む。

1. 変更要求を1つ置く。
2. その要求で変えたい側を決める。
3. その要求では変えたくない側を決める。
4. 変えたい側と変えたくない側が、どのメソッド呼び出し・引数・戻り値・生成・イベントでつながっているかを見る。
5. そのつながりのうち、変更要求のたびに知識が漏れて修正が波及する場所を、解くべき接続点とする。

接続点ごとに次を整理する。

| 観点 | 問い |
|---|---|
| 受け渡すもの | 境界を越える値・型・操作・イベントは何か |
| 変わる側／守る側 | 境界のどちら側が変わり、どちら側を安定させたいか |
| 変化の種類 | その仕様は、どの機能・仕様変更で変わるか。担当者や関係者が分かる場合は補助情報として使う |
| 漏れている判断や前提 | クラス名、生成方法、処理順序、条件分岐のうち何が漏れているか |
| 変更影響 | 要求を1つ当てたとき、どのファイルを修正・再テストするか |

成果物は `課題ID・接続点 / 接続するデータ / 変わる側 / 守る側` の一表と、全課題を解いたシステム全体の到達状態を表す一文である。課題IDはP1から連番にし、同じ課題を別表で再定義しない。

フェーズ5では対策の形を書かない。「どこへ移すか」「何を新設するか」は、フェーズ6の配置の決定である。引き渡し表、受け入れ条件表、課題カードを追加して同じ内容を繰り返さない。

## フェーズ6：対策検討で確認すること

フェーズ6はフェーズ5の接続点表から直接始め、システム全体について次の三つを決める。

| 観点 | 決めること |
|---|---|
| 分離方法 | 変わりやすい判断・処理・生成判断を、守る処理の骨格からどの契約で切るか |
| 配置場所 | 切り出した責任を、どの既存クラスまたは新しい具体クラスへ置くか |
| 組み立て方法 | 誰が具体クラスを生成・所有・登録・選択・注入し、安定側はどの契約だけを呼ぶか |

三つの決定を同じ粒度のクラス図で示し、課題の中心を色で強調する。`main()` はクラスとして描かず、Composition Root での生成・所有・登録・注入は注記、対応表、コードで具体化する。対策検討時と対策完成後は同じMermaid定義を使う。

その後、課題に関係する変更前コードだけをおさらいし、各課題IDについて「なぜこの箇所を最初に変えるか → 変更後コード → 減った変更 → 次に必要な変更」を示す。部分対応を採用候補として比較するためではなく、最終構造をコードへ安全に到達させるための段階である。

三つの観点から完成構造が一つに定まるなら比較しない。全課題を解き、責任配置や依存関係が異なる完成可能な構造が複数残る場合だけ、次を比較する。

- システム全体の課題をどこまで満たすか
- 変更要求を再適用したときの修正・再テスト範囲
- 増えるクラス、設定、登録、実行時コスト

## 禁止事項

- 旧来の4分類名
- 固定的な分類マトリクスや到達点
- パターンを固定的な接続分類へ無理に割り当てる説明
- 接続点を見ず、インターフェースや仲介役の有無だけで設計を評価すること
- 「知識」を具体化せず、曖昧なまま対策へ進むこと
