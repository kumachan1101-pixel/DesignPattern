# rules/checklist.md
# 品質チェックリスト（生成後に必ず確認）
# CLAUDE.md から参照される。

---

## 構造チェック

- [ ] 章の冒頭に `## この章を読むと得られること` セクションがあるか
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか（パターン名を前面に出していないか）
- [ ] このドメインがドメイン割り当て表と一致しているか
- [ ] ステップ0〜7、整理、振り返り、パターン解説のH2見出しが揃っているか
- [ ] `## 整理` に「8ステップとこの章でやったこと」の対応表と最終責任テーブルがあるか
- [ ] `## パターン解説` に抽象GoF構造のクラス図・章固有マッピング図・散文説明があるか
- [ ] 変更前クラス図（ステップ1）と変更後クラス図（ステップ7）が両方あるか

## ステップ5・6チェック（rules/step5-6-model.md 参照）

- [ ] ステップ5の冒頭で「原因を第0章の4分類に当てはめ、手札候補を絞り込む」表が示されているか
- [ ] 候補が複数の場合：「手段①」がコードつきで示されているか（「失敗例」ではなく「限界のある有効案」として）
- [ ] 候補が複数の場合：手段①の後に「残る課題」が明示されているか（手段②への橋渡し）
- [ ] 候補が複数の場合：「手段②」がコードつきで示されているか（①の課題への応答として）
- [ ] 手段②（または唯一の手段）の後に変更後クラス図があるか
- [ ] 手段②（または唯一の手段）の後にパターン名が初登場しているか（それより前に出ていないか）
- [ ] 複数の手札を組み合わせる場合：「1つ目適用→残る課題→2つ目が必要」の順序で示されているか
- [ ] 候補が複数の場合：ステップ6の冒頭で評価軸が先に宣言されているか（比較より前）
- [ ] 候補が複数の場合：ステップ6に「手段①vs②の比較表」があるか（宣言した評価軸で測っているか）
- [ ] 候補が複数の場合：採用の決定が比較結果から明示されているか
- [ ] 耐久テストがX.3ヒアリングのリスクと対応しているか（伏線回収）
- [ ] 「使う/使わない場面」に【過剰コード】例があるか

## コード品質チェック

- [ ] コードに専門知識が混入していないか（ファイルI/O・通信・フレームワーク禁止）
- [ ] templateメタプログラミング・lambda・スマートポインタを使っていないか
- [ ] ステップ1の起点コードにmain()と実行結果が載っているか
- [ ] ステップ7の最終コードにBatchApplication＋main()の形があるか
- [ ] テスト（スタブを使った動作保証）があるか
- [ ] X.1の直後に「要するに〜するパターン」の核心一文があるか

## 責任チェック

- [ ] ステップ1に各クラスの責任を1文で定義する表があるか
- [ ] ステップ1に「責任チェック表」があるか
- [ ] 責任外の知識が「誰の責任か」まで明示されているか

## 関係者ヒアリングチェック

- [ ] X.3に関係者ヒアリングの会話シーンがあるか
- [ ] ヒアリング結果が変動/不変テーブルの根拠になっているか
- [ ] ヒアリングで言及したリスクが耐久テストで登場しているか（伏線回収）
- [ ] インターフェース名がビジネス責任で命名されているか

## 執筆品質チェック（rules/writing-rules.md 参照）

- [ ] 禁止表現（「まず〜」「次章では〜」「悪いコード」等）がないか
- [ ] `// 💭` コードコメントを使っていないか
- [ ] 次章予告が書かれていないか
- [ ] ★ 指摘が残っていないか

## Kindle品質チェック

- [ ] コードが1行80文字以内か
- [ ] 表の列数が4列以内か
- [ ] Mermaid図に説明文が添えられているか

## 著者の人格チェック

- [ ] 「〜すべき」「〜べきです」がないか
- [ ] 口癖が3〜5箇所散りばめられているか
