## 読者体験チェック

章やフェーズを評価するときは、最初に「この箇所を置く目的は何か」を確認し、その次に「読者がこの箇所を読み終えたとき、何を自分の言葉で説明できるべきか」を確認する。構成・図・表・実行結果の要否は、目的と読者体験から逆算して判断する。

### 章全体
- [ ] 冒頭の `### この章を読むと得られること` が、章末の振り返りで回収されているか
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか（パターン名を前面に出していないか）
- [ ] 読者が章を読み終えたとき、この章のシステム・変更要求・解いた課題・採用した構造を自分の言葉で説明できる流れになっているか
- [ ] 章の途中で、まだ知らなくてよい設計上の問題やパターン名を先取りしていないか
- [ ] このドメインがドメイン割り当て表と一致しているか
- [ ] `templates/chapter-template.md` の各標準見出し直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` があり、章本文もその目的と基準を満たす構成になっているか
- [ ] 各項目の本文・図・表・コード・実行結果が、目的を達成するために必要な素材として説明できるか
- [ ] 本文で使うID、状態名、フラグ、入力値、金額、エラー条件が、コードや実行結果に出る前に仕様または動作例で説明されているか
- [ ] 「一般的」「業界標準」「現場ではよくある」など、根拠を示せない一般化をしていないか。必要なら「この章のシステムでは」に限定しているか
- [ ] 過去の `★` 指摘を消した場合、指摘内容・判断・対応先が `review-tasks.md` 等に記録されているか

### フェーズ1（現状把握）
- [ ] フェーズ1を読み終えた読者が「このシステムが何をするか」を自分の言葉で説明できるか
- [ ] 1-1を読み終えた読者が、システムの目的・利用者・扱うデータ・業務上の制約を説明できるか
- [ ] 動作例テーブルを見て、任意の代表入力に対する出力を予測できるか
- [ ] 動作例テーブルと掲載コードの `main()`・実行結果が対応しているか
- [ ] 動作例テーブル、`main()`、実行結果のケース数・対象データ・計算式・エラー理由が同じ粒度で対応しているか
- [ ] 1-3を読み終えた読者が、クラス名を見て責任を1文で言えるか
- [ ] 1-4を読み終えた読者が、動作例テーブルと現状コードの対応を追えるか
- [ ] 1-4で定義した主要クラスが、定義だけで終わらず `main()` または本文説明で使い道を回収されているか
- [ ] 1-5を読み終えた読者が、現状仕様と変更後仕様の差分を説明できるか
- [ ] クラス構成図の前に、登場クラスの責任が1文で説明されているか
- [ ] クラス構成図に出るクラスが、図の前に登場クラス表または本文で説明されているか
- [ ] フェーズ1は現状の仕様・クラス・コードの事実に限定され、変更後仕様・原因分析・設計上の痛みを含んでいないか
- [ ] フェーズ1を読み終えた時点で「なぜこの設計が問題なのか」はまだ結論づけていないか

### フェーズ2〜5（問題を定める）
- [ ] フェーズ2を読み終えた読者が、変わる可能性のある仕様と安定している仕様を区別できるか
- [ ] 2-1を読み終えた読者が、フェーズ1の事実から変わりそうな部分を仮説として挙げられるか
- [ ] 2-1は担当チーム名だけで断定せず、フェーズ1の事実から仮説を立てる書き方になっているか
- [ ] 2-3を読み終えた読者が、仮説が裏付いたか修正されたかを説明できるか
- [ ] ヒアリングは2-1の仮説を質問として回収し、後続の変動/不変テーブルまたは将来リスク表の根拠になっているか
- [ ] 2-5を読み終えた読者が、今回の変更と将来リスクを分けて説明できるか
- [ ] フェーズ2の出口で、変わるもの/変わらないもの、今回確定したこと/将来リスクが混ざらず整理されているか
- [ ] フェーズ3を読み終えた読者が、変更を試みるとどこへ影響が広がるかを説明できるか
- [ ] 3-1を読み終えた読者が、変更要求を現状コードへ当てるとどこを触る必要があるか追えるか
- [ ] 変更後コードを示す場合、追加・変更箇所がコメント、表、差分説明などで読者に分かるようになっているか
- [ ] 3-2を読み終えた読者が、変更の波及範囲を図で説明できるか
- [ ] 3-3を読み終えた読者が、痛みを感想ではなく修正箇所・確認範囲として説明できるか
- [ ] フェーズ4を読み終えた読者が、痛みの原因を「どの知識がどの責任へ漏れているか」で説明できるか
- [ ] 4-1を読み終えた読者が、痛みを生む構造を依存・分岐・責任の混在として説明できるか
- [ ] 4-2を読み終えた読者が、分離すべき知識と保ちたい骨格を区別できるか
- [ ] 4-3を読み終えた読者が、呼び出し元が相手の何を知りすぎているか説明できるか
- [ ] フェーズ5を読み終えた読者が、解くべき接続点・非機能制約・クライアント影響を説明できるか
- [ ] フェーズ5を読み終えた読者が、「何をどこから切り離し、接続点には何を残すか」を課題文として言えるか

### フェーズ6〜7（対策を選び、実装する）
- [ ] フェーズ6を読み終えた読者が、各案がどの知識をどこへ移す案なのか説明できるか
- [ ] フェーズ6の各ステップで、移した知識・残った痛み・次のステップへ進む理由が説明されているか
- [ ] フェーズ6の対策案の根拠が、フェーズ4・5の原因・課題と対応しているか
- [ ] 採用ステップの決断で、今回の文脈で採用する案と受け入れるコストを説明しているか
- [ ] フェーズ7を読み終えた読者が、最終コードで責任がどこへ移ったか説明できるか
- [ ] 7-1を読み終えた読者が、最終コードの責任分担と組み立て箇所を追えるか
- [ ] 7-4を読み終えた読者が、将来変更時に触る場所・触らない場所・確認範囲を予測できるか
- [ ] フェーズ7の実行結果が、変更後の動作例・受入条件と対応しているか

## 構造チェック

- [ ] S1〜S7、整理、振り返り、あなたのコードで考える、パターン解説のH2見出しが揃っているか
- [ ] `## 整理` に「フェーズとこの章でやったこと」の対応表と最終責任テーブルがあるか
- [ ] `## パターン解説` に抽象GoF構造のクラス図・章固有マッピング図・散文説明があるか
- [ ] 変更前クラス図（フェーズ1）と変更後クラス図（フェーズ7）が両方あるか

## S5〜S8チェック（rules/design-decision-guide.md 参照）

### S5チェック（課題定義）
- [ ] S5の冒頭で接続点（どのクラス間の結合が問題か）が明示されているか
- [ ] S5に非機能制約（パフォーマンス・メモリ・変更頻度）の記述があるか
- [ ] S5に課題まとめ表があるか

### 共通チェック
- [ ] S1はコードの事実（仕様表・クラス図・責任一覧）のみで、仮説を含んでいないか
- [ ] S2で仮説テーブルを立て、ヒアリングで確定する2段構えになっているか
- [ ] S6の冒頭に接続点の現状と、各案で移す知識があるか
- [ ] 対策案の根拠がS4・S5の原因・課題と対応しているか
- [ ] パターン名がS6以降に初登場しているか（それより前に出ていないか）
- [ ] 耐久テストがX.3ヒアリングのリスクと対応しているか（伏線回収）
- [ ] 「使う/使わない場面」に【過剰コード】例があるか

### 案が1つの場合（案Nのみ）
- [ ] コード→変更後クラス図→パターン名の順で示されているか
- [ ] 天秤（評価軸の宣言・比較表）が存在しないか（案1つでは不要）

### 案が複数の場合（案0〜案N）
- [ ] 各案のトレードオフが対比されているか（どれが正解とは言っていないか）
- [ ] S7の評価軸が先に宣言されているか（比較より前）
- [ ] S7に「コスト天秤表」があるか（宣言した評価軸で測っているか）
- [ ] 採用の決定が比較結果から明示されているか（文脈依存の選択として説明されているか）

## コード品質チェック

- [ ] コードに専門知識が混入していないか（ファイルI/O・通信・フレームワーク禁止）
- [ ] templateメタプログラミング・lambda・スマートポインタを使っていないか
- [ ] フェーズ1の起点コードにmain()と実行結果が載っているか
- [ ] フェーズ7の最終コードにアプリケーション境界＋main()の形があるか
- [ ] テスト（スタブを使った動作保証）があるか
- [ ] X.1の直後に「要するに〜するパターン」の核心一文があるか

## 責任チェック

- [ ] S1に各クラスの責任を1文で定義する表があるか
- [ ] S1に「責任チェック表」があるか
- [ ] 責任外の知識が「誰の責任か」まで明示されているか

## 関係者ヒアリングチェック

- [ ] X.3に関係者ヒアリングの会話シーンがあるか
- [ ] ヒアリング結果が変動/不変テーブルの根拠になっているか
- [ ] ヒアリングで言及したリスクが耐久テストで登場しているか（伏線回収）
- [ ] インターフェース名がビジネス責任で命名されているか

## 執筆品質チェック（rules/writing-rules.md 参照）

- [ ] 禁止表現（「まず〜」「次章では〜」「悪いコード」等）がないか
- [ ] `// 💭` コードコメントを使っていないか
- [ ] 次章予告が書かれていないか
- [ ] `★` 指摘が残っていないか

## Kindle品質チェック

- [ ] コードが1行80文字以内か
- [ ] 表の列数が4列以内か
- [ ] Mermaid図に説明文が添えられているか

## 著者の人格チェック

- [ ] 「〜すべき」「〜べきです」がないか
- [ ] 口癖が3〜5箇所散りばめられているか
