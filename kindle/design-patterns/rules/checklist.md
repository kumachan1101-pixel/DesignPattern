## 読者体験チェック

章やフェーズを評価するときは、最初に「この箇所を置く目的は何か」を確認し、その次に「読者がこの箇所を読み終えたとき、何を自分の言葉で説明・予測・判断できるべきか」を確認する。
構成、図、表、コード、実行結果の要否は、目的と読者体験から逆算して判断する。

### 章全体

- [ ] 冒頭の `### この章を読むと得られること` が、章末の振り返りで回収されているか。
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか。
- [ ] 読者が章を読み終えたとき、この章のシステム、変更要求、解いた課題、採用した構造を自分の言葉で説明できる流れになっているか。
- [ ] 章の途中で、まだ知らなくてよい設計上の問題やパターン名を先取りしていないか。
- [ ] `templates/chapter-template.md` の各標準見出し直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` があるか。
- [ ] 章本文、図、表、コード、実行結果が、それぞれの目的と達成基準を満たすために置かれているか。
- [ ] 第0章の7フェーズ説明と `templates/chapter-template.md` の標準項目・目的・達成基準が矛盾していないか。
- [ ] 本文で使うID、状態名、フラグ、入力値、金額、エラー条件が、コードや実行結果に出る前に仕様または動作例で説明されているか。
- [ ] 「一般的」「業界標準」「現場ではよくある」など、根拠を示せない一般化をしていないか。必要なら「この章のシステムでは」に限定しているか。
- [ ] 過去の `★` 指摘を消した場合、指摘内容・判断・対応先が `review-tasks.md` 等に記録されているか。

## 図解チェック

- [ ] 入力、加工、出力の流れが重要な箇所は、文章だけでなく図または表で示しているか。
- [ ] クラス間の関係、変更影響、改善前後の差分、責任移動は、必要に応じて図または表で先に見えるようになっているか。
- [ ] 図の前に、読者がその図を見る目的を1文で書いているか。
- [ ] 図に出るクラス名・値・状態名・操作名は、本文または表で説明済みか。
- [ ] 図の後に、図から読み取るべき結論を1〜3文で回収しているか。
- [ ] 図だけで新しい仕様や判断を初出ししていないか。
- [ ] 図が細かくなりすぎる場合、全体図と補足図に分けているか。

## フェーズ1（現状把握）

- [ ] フェーズ1を読み終えた読者が「このシステムが何をするか」を自分の言葉で説明できるか。
- [ ] 1-1を読み終えた読者が、システムの目的・利用者・扱うデータ・入力・加工・出力・業務上の制約を説明できるか。
- [ ] 1-1に入力→加工→出力の仕様構造図または表があるか。
- [ ] 動作例テーブルを見て、任意の代表入力に対する出力を予測できるか。
- [ ] 動作例テーブルと掲載コードの `main()`・実行結果が対応しているか。
- [ ] 1-3を読み終えた読者が、クラス名を見て責任を1文で言えるか。
- [ ] クラス構成図の前に、登場クラスの責任が説明されているか。
- [ ] 1-4を読み終えた読者が、動作例テーブルと現状コードの対応を追えるか。
- [ ] 1-5を読み終えた読者が、現状仕様と変更後仕様の差分を説明できるか。
- [ ] フェーズ1は現状の仕様・クラス・コードの事実に限定され、原因分析や設計上の痛みを含んでいないか。

## フェーズ2（仮説立案）

- [ ] フェーズ2を読み終えた読者が、変わる可能性のある仕様、今回確実に変わる仕様、当面安定している仕様、将来リスクを区別できるか。
- [ ] 2-1は担当チーム名だけで断定せず、フェーズ1の事実から仮説を立てる書き方になっているか。
- [ ] ヒアリングは2-1の仮説を質問として回収し、後続の確定変更または将来リスクの根拠になっているか。
- [ ] フェーズ2の出口で、今回確定したことと将来リスクが混ざらず整理されているか。

## フェーズ3（問題特定）

- [ ] フェーズ3を読み終えた読者が、変更を試みるとどこへ影響が広がるかを説明できるか。
- [ ] 3-1を読み終えた読者が、変更要求を現状コードへ当てるとどこを触る必要があるか追えるか。
- [ ] 変更後コードを示す場合、追加・変更箇所がコメント、表、差分説明などで読者に分かるようになっているか。
- [ ] 3-2を読み終えた読者が、直接変更と巻き込まれた変更を図で説明できるか。
- [ ] 3-3を読み終えた読者が、痛みを感想ではなく修正箇所・確認範囲・再テスト範囲として説明できるか。

## フェーズ4（原因分析）

- [ ] フェーズ4を読み終えた読者が、痛みの原因を「どの判断や前提がどの責任へ漏れているか」で説明できるか。
- [ ] 4-1を読み終えた読者が、痛みを生む構造を依存・分岐・責任の混在として説明できるか。
- [ ] 4-2を読み終えた読者が、分離すべき判定・処理・生成判断と、保ちたい処理の骨格を区別できるか。
- [ ] 4-3を読み終えた読者が、呼び出し元が相手のクラス名・生成方法・条件・処理順序の何を知りすぎているか説明できるか。
- [ ] 「知識」という語を使う場合、判定条件、計算式、処理順序、生成判断、クラス名、型、操作名などへ具体化しているか。

## フェーズ5（課題定義）

- [ ] フェーズ5を読み終えた読者が、解くべき接続点・非機能制約・クライアント影響を説明できるか。
- [ ] フェーズ5を読み終えた読者が、「何をどこから切り離し、接続点には何を残すか」を課題文として言えるか。
- [ ] 接続点で流れる値・型・操作・イベントが図または表で示されているか。
- [ ] 課題がフェーズ4の原因から導かれており、対策名を先に決めていないか。

## フェーズ6（対策検討）

- [ ] フェーズ6を読み終えた読者が、各案がどの判断・処理・生成責任をどこへ移す案なのか説明できるか。
- [ ] 各案で、接続点、減る変更、残る変更、導入コストが説明されているか。
- [ ] 対策案の根拠が、フェーズ4・5の原因・課題と対応しているか。
- [ ] 採用する形の決定で、今回の文脈で採用する案と受け入れるコストを説明しているか。
- [ ] パターン名を、対策を決める前の選択肢ラベルとして使っていないか。

## フェーズ7（対策実施）

- [ ] フェーズ7を読み終えた読者が、最終コードで責任がどこへ移ったか説明できるか。
- [ ] 7-1を読み終えた読者が、最終コードの責任分担と組み立て箇所を追えるか。
- [ ] 7-2に動作シーケンス図があり、改善後の呼び出し順を確認できるか。
- [ ] 7-3に改善後の変更影響グラフがあり、改善前より何が減ったか説明しているか。
- [ ] 7-4を読み終えた読者が、将来変更時に触る場所・触らない場所・確認範囲を予測できるか。
- [ ] フェーズ7の実行結果が、変更後の動作例・受入条件と対応しているか。

## 後段チェック

- [ ] `## 整理` に「フェーズとこの章でやったこと」の対応表と最終責任テーブルがあるか。
- [ ] `## 振り返り` で、冒頭の「得られること」を回収しているか。
- [ ] `## あなたのコードで考える` が、章固有の設計判断を読者の現場へ持ち帰れる問いにしているか。
- [ ] `## パターン解説` に抽象GoF構造のクラス図・章固有マッピング図・散文説明があるか。
- [ ] パターン解説が章の本体を圧倒していないか。

## コード品質チェック

- [ ] コードに専門知識が混入していないか（ファイルI/O・通信・フレームワーク禁止）。
- [ ] templateメタプログラミング・lambda・スマートポインタを使っていないか。
- [ ] フェーズ1の起点コードに `main()` と実行結果が載っているか。
- [ ] フェーズ7の最終コードにアプリケーション境界と `main()` の形があるか。
- [ ] テストまたは実行結果で、変更前後の振る舞いが確認されているか。

## 執筆品質チェック

- [ ] 禁止表現（「まず〜」「次章では〜」「悪いコード」等）がないか。
- [ ] `// 💭` コードコメントを使っていないか。
- [ ] 次章予告が書かれていないか。
- [ ] `★` 指摘が残っていないか。
- [ ] AIへの指示文や引用マーカーが本文へ混入していないか。

## Kindle品質チェック

- [ ] コードが1行80文字以内か。
- [ ] 表の列数が4列以内か。
- [ ] Mermaid図に説明文が添えられているか。
