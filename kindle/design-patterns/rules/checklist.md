## 構造チェック

- [ ] 章の冒頭に `## この章を読むと得られること` セクションがあるか
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか（パターン名を前面に出していないか）
- [ ] このドメインがドメイン割り当て表と一致しているか
- [ ] フェーズ1〜7、整理、振り返り、パターン解説のH2見出しが揃っているか
- [ ] `## 整理` に「7フェーズとこの章でやったこと」の対応表と最終責任テーブルがあるか
- [ ] `## パターン解説` に抽象GoF構造のクラス図・章固有マッピング図・散文説明があるか
- [ ] 変更前クラス図（フェーズ1）と変更後クラス図（フェーズ7）が両方あるか

## フェーズ5〜7チェック（rules/design-decision-guide.md 参照）

### フェーズ5チェック（課題定義）
- [ ] フェーズ5の冒頭で接続点（どのクラス間の結合が問題か）が明示されているか
- [ ] フェーズ5に非機能制約（パフォーマンス・メモリ・変更頻度）の記述があるか
- [ ] フェーズ5に課題まとめ表があるか

### 共通チェック
- [ ] フェーズ1（現状把握）はコードの事実（仕様表・クラス図・責任一覧）のみで、仮説を含んでいないか
- [ ] フェーズ2で仮説テーブルを立て、ヒアリングで確定する2段構えになっているか
- [ ] フェーズ6の冒頭に2×2マトリクス接続図（mermaid）があるか
- [ ] 対策案の根拠がフェーズ4・フェーズ5の原因・課題と対応しているか
- [ ] パターン名がフェーズ6以降に初登場しているか（それより前に出ていないか）
- [ ] 耐久テストがフェーズ2（2-3）ヒアリングのリスクと対応しているか（伏線回収）
- [ ] 「使う/使わない場面」に【過剰コード】例があるか

### 対策案を1つだけ提示する場合（比較なし）
- [ ] コード→変更後クラス図→パターン名の順で示されているか
- [ ] 天秤（評価軸の宣言・比較表）が存在しないか（案1つでは不要）

### 案が複数の場合（案0〜案4を比較する場合）
- [ ] 各案のトレードオフが対比されているか（どれが正解とは言っていないか）
- [ ] フェーズ6（対策選定）の評価軸が先に宣言されているか（比較より前）
- [ ] フェーズ6（対策選定）に「コスト天秤表」があるか（宣言した評価軸で測っているか）
- [ ] 採用の決定が比較結果から明示されているか（文脈依存の選択として説明されているか）

## コード品質チェック

- [ ] コードに専門知識が混入していないか（ファイルI/O・通信・フレームワーク禁止）
- [ ] templateメタプログラミング・lambda・スマートポインタを使っていないか
- [ ] フェーズ1の起点コードにmain()と実行結果が載っているか
- [ ] フェーズ7の最終コードにBatchApplication＋main()の形があるか
- [ ] テスト（スタブを使った動作保証）があるか
- [ ] フェーズ1（1-8の直後）に核心一文があるか（形式：「要するに、[フェーズ1の観察]から[構造の問題]が見えてくる」。「〜するパターン」形式はNG）

## 責任チェック

- [ ] フェーズ1に各クラスの責任を1文で定義する表があるか
- [ ] フェーズ1に「責任チェック表」があるか
- [ ] 責任外の知識が「誰の責任か」まで明示されているか

## 関係者ヒアリングチェック

- [ ] フェーズ2（2-3）に関係者ヒアリングの会話シーンがあるか
- [ ] ヒアリング結果が変動/不変テーブルの根拠になっているか
- [ ] ヒアリングで言及したリスクが耐久テストで登場しているか（伏線回収）
- [ ] インターフェース名がビジネス責任で命名されているか

## 執筆品質チェック（rules/writing-rules.md 参照）

- [ ] 禁止表現（「まず〜」「次章では〜」「悪いコード」等）がないか
- [ ] `// 💭` コードコメントを使っていないか
- [ ] 次章予告が書かれていないか
- [ ] ★ 指摘が残っていないか

## Kindle品質チェック

- [ ] コードが1行80文字以内か
- [ ] 表の列数が4列以内か
- [ ] Mermaid図に説明文が添えられているか

## 著者の人格チェック

- [ ] 「〜すべき」「〜べきです」がないか
- [ ] 口癖が3〜5箇所散りばめられているか
