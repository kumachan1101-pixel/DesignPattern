## 構造チェック

- [ ] 章の冒頭に `## この章を読むと得られること` セクションがあるか
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか（パターン名を前面に出していないか）
- [ ] このドメインがドメイン割り当て表と一致しているか
- [ ] ステップ0〜8（S0〜S8）、整理、振り返り、パターン解説のH2見出しが揃っているか
- [ ] `## 整理` に「8ステップとこの章でやったこと」の対応表と最終責任テーブルがあるか
- [ ] `## パターン解説` に抽象GoF構造のクラス図・章固有マッピング図・散文説明があるか
- [ ] 変更前クラス図（ステップ1）と変更後クラス図（S8）が両方あるか

## S5〜S8チェック（rules/step5-6-model.md 参照）

### S5チェック（課題定義）
- [ ] S5の冒頭で接続点（どのクラス間の結合が問題か）が明示されているか
- [ ] S5に非機能制約（パフォーマンス・メモリ・変更頻度）の記述があるか
- [ ] S5に課題まとめ表があるか

### 共通チェック
- [ ] S6の冒頭に2×2マトリクス接続図（mermaid）があるか
- [ ] 手札候補の根拠がステップ4の原因と対応しているか
- [ ] パターン名がS6以降に初登場しているか（それより前に出ていないか）
- [ ] 耐久テストがX.3ヒアリングのリスクと対応しているか（伏線回収）
- [ ] 「使う/使わない場面」に【過剰コード】例があるか

### ケース①（候補手札が1つ）
- [ ] コード→変更後クラス図→パターン名の順で示されているか
- [ ] 天秤（評価軸の宣言・比較表）が存在しないか（ケース①では不要）

### ケース②（複数手札・組み合わせで解決）
- [ ] 統合実装として1つのコードで示されているか（別々に試していないか）
- [ ] クラス図で各手札の適用箇所が明示されているか
- [ ] 天秤が存在しないか（ケース②でも天秤不要）

### ケース③（複数手札・別々に試す必要がある）
- [ ] 手段①がコードつきで示されているか（「手札表から選んだ正当な候補」として。"naive な試み"ではない）
- [ ] 手段①の後に「残る課題」が明示されているか（手段②への橋渡し）
- [ ] 手段②がコードつきで示されているか（①の残る課題への応答として）
- [ ] 手段②の後に変更後クラス図があるか
- [ ] S7の評価軸が先に宣言されているか（比較より前）
- [ ] S7に「コスト天秤表（手段①vs②）」があるか（宣言した評価軸で測っているか）
- [ ] 採用の決定が比較結果から明示されているか

## コード品質チェック

- [ ] コードに専門知識が混入していないか（ファイルI/O・通信・フレームワーク禁止）
- [ ] templateメタプログラミング・lambda・スマートポインタを使っていないか
- [ ] ステップ1の起点コードにmain()と実行結果が載っているか
- [ ] S8の最終コードにBatchApplication＋main()の形があるか
- [ ] テスト（スタブを使った動作保証）があるか
- [ ] X.1の直後に「要するに〜するパターン」の核心一文があるか

## 責任チェック

- [ ] ステップ1に各クラスの責任を1文で定義する表があるか
- [ ] ステップ1に「責任チェック表」があるか
- [ ] 責任外の知識が「誰の責任か」まで明示されているか

## 関係者ヒアリングチェック

- [ ] X.3に関係者ヒアリングの会話シーンがあるか
- [ ] ヒアリング結果が変動/不変テーブルの根拠になっているか
- [ ] ヒアリングで言及したリスクが耐久テストで登場しているか（伏線回収）
- [ ] インターフェース名がビジネス責任で命名されているか

## 執筆品質チェック（rules/writing-rules.md 参照）

- [ ] 禁止表現（「まず〜」「次章では〜」「悪いコード」等）がないか

- [ ] 次章予告が書かれていないか
- [ ] ★ 指摘が残っていないか
