## 読者体験チェック

章やフェーズを評価するときは、最初に「この箇所を置く目的は何か」を確認し、その次に「読者がこの箇所を読み終えたとき、何を自分の言葉で説明・予測・判断できるべきか」を確認する。
構成、図、表、コード、実行結果の要否は、目的と読者体験から逆算して判断する。

### 章全体

- [ ] 冒頭の `### この章を読むと得られること` が、章末の振り返りで回収されているか。
- [ ] 「得られること」が「〜できるようになる」形式で3〜4項目あるか。
- [ ] 読者が章を読み終えたとき、この章のシステム、変更要求、解いた課題、採用した構造を自分の言葉で説明できる流れになっているか。
- [ ] 各項目が「前の項目で何が分かったか」「だから次に何を見るのか」でつながっているか。つながらない項目は順序変更・別フェーズ移動・削除を検討しているか。
- [ ] 章の途中で、まだ知らなくてよい設計上の問題やパターン名を先取りしていないか。
- [ ] 現状仕様、変更要求、最終コードの動作を同じ表や図に混ぜていないか。混ぜる場合は「現状」「変更後」「最終確認」の区分が明記されているか。
- [ ] `templates/chapter-template.md` の各標準見出し直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` があるか。
- [ ] 章本文、図、表、コード、実行結果が、それぞれの目的と達成基準を満たすために置かれているか。
- [ ] 第0章の7フェーズ説明と `templates/chapter-template.md` の標準項目・目的・達成基準が矛盾していないか。
- [ ] 専門用語・略語・英語由来の呼び名を使う場合、初出で意味を説明しているか。説明せずに使うより、一般的な日本語へ言い換えられないか。
- [ ] 本文に「まだ判断しない」「事実としてそろえる」など、執筆者向けの進行管理を読者へ言う文が出ていないか。必要な場合は「次に何を確認するか」へ言い換えているか。
- [ ] 本文で使うID、状態名、フラグ、入力値、金額、エラー条件が、コードや実行結果に出る前に仕様または動作例で説明されているか。
- [ ] 題材のコア処理が、単なる分岐・ハンドリング・分類箱に薄められていないか。読者が現場で気にする処理順序、必要データ、状態、失敗、非同期、外部連携が、章の論点に必要な粒度で示されているか。
- [ ] 複雑度ストレス条件（順次実行、非同期/ポーリング、イベント、部分失敗、ジョブ/スレッド、再試行など）が、章の中心課題に関係する範囲で示されているか。複雑さを足した場合、フェーズ4の原因、フェーズ5の課題、フェーズ6の対策比較に接続しているか。
- [ ] 「一般的」「業界標準」「現場ではよくある」など、根拠を示せない一般化をしていないか。必要なら「この章のシステムでは」に限定しているか。
- [ ] 過去の `★` 指摘を消した場合、指摘内容・判断・対応先が `review-tasks.md` 等に記録されているか。

## 図解チェック

- [ ] 入力、加工、出力の流れが重要な箇所は、文章だけでなく図または表で示しているか。
- [ ] クラス間の関係、変更影響、改善前後の差分、責任移動は、必要に応じて図または表で先に見えるようになっているか。
- [ ] 図の前に、読者がその図を見る目的を1文で書いているか。
- [ ] 図に出るクラス名・値・状態名・操作名は、本文または表で説明済みか。
- [ ] 1-1の仕様図は、システムの目的・利用者・扱うデータ・業務上の前提を文章で説明した後に置かれているか。
- [ ] 1-1の仕様図は、仕様説明の代わりではなく、一通り説明した仕様を整理して俯瞰する位置に置かれているか。
- [ ] 仕様図の前に、代表入力・判定条件・加工内容・正常出力を文章または表で説明しているか。異常系は正常系図の直後に別表で整理しているか。
- [ ] 図の後に、図から読み取るべき結論を1〜3文で回収しているか。
- [ ] 図だけで新しい仕様や判断を初出ししていないか。
- [ ] 図が細かくなりすぎる場合、全体図と補足図に分けているか。
- [ ] 状態・通知先・承認者・設定値などの保存データが動作を決める章では、入力→加工→出力図の前に「どのデータがどこに記憶され、誰がアクセスし、どう使うか」の保存データ構造図を置いているか。
- [ ] 仕様説明の最初の処理フロー図は正常系を先に示し、エラー条件は図に混ぜず直後の表で分けて説明しているか。

## 仕様粒度チェック

- [ ] 1-1の仕様が、動作例・変更要求・実行結果に出る入力値、状態名、フラグ、ID、金額、判定条件、出力名、エラー条件を先に説明しているか。
- [ ] 仕様が情報の列挙で終わらず、入力・判定・加工・正常出力の各要素が、後続の動作例・コード・変更要求・接続点でどこに出るか追えるか。異常系は別表で、どの判定で止まるか追えるか。
- [ ] 仕様が、題材の中心となる実処理を説明しているか。決済なら認証・入金待ち・失敗、ワークフローなら保存状態・承認者・通知、ファイル処理なら形式差分と解析手順のように、読者が最も知りたい処理を「後で省略」として消していないか。
- [ ] 1-1の冒頭が、担当チーム、責任分担、変更背景、設計上の困りごとではなく、システムの仕様そのものから始まっているか。
- [ ] 1-1〜1-3のシステム説明に、時系列の経緯ストーリー（「リリース当初は…次々と追加された」「サービスの拡大に伴い…増えた」等）や、後続フェーズの結論を先取りする評価的伏線（「後のフェーズで問題の核心になる」「設計上の問題として浮かび上がる」「頭の片隅に置いておいて」等）が混ざっていないか。フェーズ1は事実として読む節であり、判断・伏線はフェーズ4以降へ送る。中立的な前方参照（「後のフェーズで…名前として使う」）は可。
- [ ] 1-1の仕様表に、分類名だけでなく、代表入力値、判定条件の具体例、加工後の値・状態、正常出力の具体例、後続での使い道が書かれているか。異常出力は別表で扱っているか。
- [ ] 入力→加工→出力の図が、分類だけで終わらず、各要素が後でどの動作例・コード・変更要求・接続点で使われるかを示しているか。
- [ ] 入力→加工→出力の図が3箱で終わらず、代表入力がどの判定・計算・変換を通り、まず正常出力へ至るかを示しているか。エラーは別表で扱っているか。
- [ ] 仕様図が代表ケースの場合は「代表ケース」と明記し、全種類を示す必要があるID、状態、設定値、手数料、形式、通知先などは別表で網羅しているか。
- [ ] 正常系の仕様図で、入力・判定・加工・正常出力が同じ四角で表されておらず、形状または色で区別されているか。異常出力を同じ図へ混ぜていないか。
- [ ] 仕様の「加工」が、業務上の判定・計算・変換・保存・通知として書かれ、1-3以降でどのクラス責任に対応するか追えるか。
- [ ] 種類ごとに必要データや制約が異なる場合、都合よく同じプリミティブ引数に潰していないか。要求オブジェクト、結果オブジェクト、状態、イベントなどで差分を表し、共通契約として何を残すか説明しているか。
- [ ] ファイル形式、入力フォーマット、状態遷移、計算ルールなど、文章だけでは差分が見えにくい仕様に、実データ例・比較表・状態例が添えられているか。
- [ ] 割引、料金、ポイント、税などの計算では、加算・逐次適用・優先適用・排他などの計算ポリシーを仕様として明記しているか。
- [ ] 動作例テーブルは、仕様図の代表ケースになっているか。仕様図にない入力・状態・出力を突然出していないか。
- [ ] 1-2の動作例テーブルに、1-5以降の変更要求で初めて登場する操作（履歴、Undo、再実行、通知など）を混ぜていないか。
- [ ] 1-3で登場クラスとクラス構成を説明する前に、具体的なクラス名、メソッド名、状態定数、変数名、コード風の出力文字列を出していないか。
- [ ] クラス図に出ていない関数名・変数名・状態定数を、構成説明の前に出していないか。
- [ ] 実装コードを提示する前に、その中の条件分岐・計算・生成・例外処理などのロジックを説明していないか。
- [ ] 仕様図に置いた入力・加工・出力のうち、後続で使われない要素が混ざっていないか。後続で必要な要素が仕様図から漏れていないか。
- [ ] 掲載コードで `print`、固定データ、簡略化した `main()` などを使う場合、実際のシステムでは何に相当するかを全体図または本文で示しているか。
- [ ] 実システムでAPI、DB、ファイル、通知、描画ライブラリなどを呼ぶ箇所が、掲載コードでも境界クラス/インターフェースを呼ぶ形になっているか。`print` は境界の先のスタブ実装に閉じているか。
- [ ] 認証、外部APIの連続呼び出し、非同期完了、失敗照会、リトライなどが題材の痛みに関係する場合、それを簡略化で消さず、境界クラス・状態・結果で表しているか。論点外なら、後続の原因・課題・対策に影響しない理由を書いているか。
- [ ] 章末または整理で、追加した複雑さ、見えた原因、定めた課題、採用した構造の対応を示し、複雑度が上がっても同じ7フェーズの論理で対策できたことを確認できるか。
- [ ] 簡略化範囲の説明は、仕様図の直後ではなく、登場クラスとクラス構成を説明した後、1-4のコードへ入る直前に置いているか。
- [ ] DB、画面、外部API、ファイル出力、通知などを章の都合で省略する場合、省略した範囲と理由を明記しているか。
- [ ] 簡略化しても、設計上の論点と変更影響の説明が変わらないことを説明しているか。
- [ ] 論点から外す処理について、実際はどう動くか、掲載コードでは何で代替するか、なぜ割愛するかを補足しているか。
- [ ] 省略した外部境界（DB・画面・API・通知）が実運用で失敗・非同期になりうる場合、失敗を扱うコードを最低1つ示すか、補足で本番の扱い（例外・リトライ・タイムアウト・非同期）と境界を流れる値の形への影響を明記しているか。
- [ ] DB保存、API呼び出し、メール/チャット通知、ファイル出力などが成功した場合に何が保存・送信されるか、失敗した場合に状態更新や通知をどう扱うかを、章の論点に必要な粒度で示しているか。
- [ ] 外部処理の処理時間・タイムアウト・リトライ・非同期化が読者の疑問になりやすい題材では、それを本文で扱うのか、論点外として補足に留めるのかを明記しているか。
- [ ] 「論点ではない」として割愛した処理が、後続の原因分析・課題定義・対策検討で暗黙の前提になっていないか。

## 実システム化ルーブリック（R1〜R5）

第8章の★指摘から抽出した、題材のコア処理が薄くないと言える条件。各章でこの5点を満たすか、満たさない場合は理由を明記する。薄いときの症状も併記する。

- [ ] R1（要求／結果オブジェクト）：種類ごとの入力差分を、真偽フラグや同一プリミティブ引数へ潰さず、要求オブジェクト・結果オブジェクトで表しているか。戻り値が金額や成否の裸の値だけでなく、内訳・適用内容・失敗理由を持つ結果オブジェクトになっているか。（症状：`pay(amount)`、`bool` フラグの羅列、`int` を返すだけ）
- [ ] R2（順序を持つ手順）：処理を、認証→確定などの順序を持つ手順として示しているか。単なる分岐・分類箱になっていないか。
- [ ] R3（保存データ＋境界）：状態・設定・履歴・価格・通知先などを、利用側が毎回渡す入力ではなく、Repository / Catalog / Database などの保存データ＋境界として表しているか。更新や取り消しの副作用が、集計だけでなく保存データにも届いているか。（症状：利用側が毎回状態や通知先を渡して見える）
- [ ] R4（成功／保留／失敗・非同期）：外部境界の成功・保留・失敗・非同期を、結果オブジェクトや状態で扱っているか。章の論点外にする場合は、その理由と本番での扱い（例外・リトライ・タイムアウト・非同期）を明記しているか。（症状：成功前提だけで本番懸念が宙に浮く）
- [ ] R5（代表ケースと全量表の分離）：図が代表ケースなら「代表ケース」と明記し、全ID・状態・設定値・価格・形式・通知先などは別表で網羅しているか。（症状：図の一例が全量か読者が判断できない）

## システム構成の多様化チェック（ARCH）

現場の実装は同期呼び出しやポーリングだけではない。章の題材に無理がない範囲で、システム構成が1つの型に固定化していないかを確認する。実スレッドは掲載コードの制約（`-pthread` なしでのコンパイル・実行、lambda/スマートポインタ不使用）に反するため、非同期やワーカーは実スレッドを使わず、キュー・状態・結果オブジェクトで表す。

- [ ] ARCH-01（イベント駆動）：外部からの非同期リクエスト（Webhook等）を受け取り、種別に応じた処理オブジェクトを生成・ルーティングする構成を、題材に合う章で扱えているか。扱わない場合、同期呼び出し前提が骨格へ漏れていないか。
- [ ] ARCH-02（別スレッド／ワーカー）：メインフローからジョブをキューへ積み、ワーカーが順に取り出して実行する構成を、実スレッドを使わずキューと `JobResult` などで表せているか。投入順・実行順・部分失敗時に後続を続けるかの契約が明示されているか。
- [ ] ARCH-03（構成の固定化回避）：システム構成（同期／ポーリング／イベント／別スレッド）が題材の都合だけで固定化していないか。現場の実態に即した構成バリエーションを選び、選んだ理由を説明しているか。
- [ ] ARCH-04（ロールバック／補償と中断）：処理の途中でエラーやキャンセルが起きた場合に、すでに完了した状態や外部API呼び出しを安全に取り消す（補償）流れを、題材の痛みに関係する章で扱えているか。中断時にどこまで戻すかが結果オブジェクトや状態で追えるか。

## フェーズ間整合チェック

各フェーズ単体の達成基準を満たしていても、フェーズ間で対象がずれていれば章としては不合格とする。
詳細は `rules/phase-consistency-check.md` を参照する。

- [ ] 1-4の実装コード説明では、構造分析や問題提起をしていないか。「混在している」「分けるべき」「変更に弱い」「影響を受ける」などの評価は、フェーズ3以降またはフェーズ4に送っているか。
- [ ] フェーズ2で「混在」「適切/不適切」「違反」と断定していないか。変わりそうな仕様候補を、仕様図・動作例・コード上の場所に対応づける表になっているか。
- [ ] 1-1の入力・加工・出力が、1-2の動作例テーブルに出ているか。
- [ ] 1-1の入力・加工・出力が、1-3以降で登場するクラス責任・コード上の変数・条件・メソッド・出力名へ段階的に対応づけられているか。
- [ ] 1-2の代表ケースが、1-4の `main()` またはテストで実行されているか。
- [ ] `main()` やテストの出力が、動作例テーブルの期待結果と一致しているか。
- [ ] 実行結果の直前に、実行対象コード、対応する動作例ケース、確認したいことが明記されているか。
- [ ] 実行結果だけを見ても、どの入力条件・状態・フラグ・操作に対する結果なのか分かるか。
- [ ] 実行結果が、入力（誰が・何を・いくら）と結果（小計→支払金額など）まで示し、最終値の羅列になっていないか。施策や操作の効果（割引前→割引後）が読み取れるか。
- [ ] コード抜粋の前に、どのクラスのどの関数か、何を省略しているか、どこに着目するかを書いているか。
- [ ] `main()` の各ケースの入力（条件・データ）と実行結果が、対応表などで近くで照合できるか（長い `main()` と出力を行き来させていないか）。
- [ ] 実行結果にケース名または番号、入力、出力、区切りがあり、どこからどこまでが1つの実行結果か分かるか。略語を使う場合は意味を説明し、基本は「ケース1：予約して支払う」のように読める名前にする。
- [ ] 複数の実行結果を並べる場合、フェーズ1の現状コード・フェーズ3の変更途中コード・フェーズ7の最終コードのどれに対する結果かが区別されているか。
- [ ] 「現状コード」とだけ書くと対象がぶれる箇所では、「フェーズ1の現状コード」「フェーズ3の変更途中コード」「フェーズ6の検討案コード」「フェーズ7の最終コード」のように対象フェーズを付けているか。
- [ ] 仕様にあるエラー条件が、動作例・コード・実行結果のどこで扱われるか説明されているか。
- [ ] コードに登録されるID、状態、フラグ、設定値、手数料、形式、通知先などが、仕様表・動作例・実行結果と一致しているか。代表図に出ない値も、別表で説明されているか。
- [ ] 1-3の登場クラス表にあるクラスが、1-4のコードに存在するか。
- [ ] 1-4の主要クラスが、1-3の登場クラス表またはクラス構成図に存在するか。
- [ ] クラス構成図の依存関係が、コード上のメンバ、引数、生成、呼び出しと矛盾していないか。
- [ ] 1-5の変更要求が、2-3の確定変更、3-1の変更作業、7-4の変更シナリオ表と同じ要求を扱っているか。
- [ ] 3-2の変更影響グラフが、3-1で実際に触った変更箇所と一致しているか。
- [ ] 3-3の痛みが、3-1の変更作業と3-2の変更影響から導かれているか。
- [ ] 4の原因分析が、3の痛みと対応しているか。
- [ ] 5の課題文が、4の原因分析から導かれているか。
- [ ] 6の各対策案が、5の課題文へ応答しているか。
- [ ] 7-1の最終コードが、6で採用した案を実装しているか。
- [ ] 7-1の最終コード直後に、実クラス名の完成後クラス図があり、章末の抽象的なパターン骨格図と役割を比較できるか。
- [ ] 7-2の動作シーケンス図、7-3の改善後影響グラフ、7-4の変更シナリオ表が、7-1の最終コードと矛盾していないか。

## フェーズ1（現状把握）

- [ ] フェーズ1を読み終えた読者が「このシステムが何をするか」を自分の言葉で説明できるか。
