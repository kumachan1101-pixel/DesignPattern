## 基本姿勢

テンプレートの項目を箇条書きで埋めるだけでは足りない。
各項目は、読者が「ここまで読んだので次へ進める」と感じる水準まで展開する。

各項目では、次の流れを意識する。

1. なぜ今これを確認するか。
2. 何を材料に見るか。
3. 何が分かったか。
4. 次の項目で何に使うか。

## 目的と達成基準

各標準見出しの直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` を置く。

- 目的：この項目は読者の理解を何のために進めるのか。
- 達成基準：読み終えた読者は何を自分の言葉で説明・予測・判断できるべきか。

目的を達成しない説明、図、表、コード、実行結果は置かない。
達成基準に関係しない一般論や先取りは削る。

## 短段落で積み重ねる

1段落は1〜3文に収める。
長い段落は分割する。
「この文で何かを伝えたら、次の段落で別のことを伝える」という積み重ね方をする。

## 章の独立性

第0章を読んでいれば、どの章から読んでも理解できる完全自己完結の構成にする。

- 「前章で学んだように」は使わない。
- 「第〇章と同様に」は使わない。
- 次章予告は書かない。

## 図解ルール

文章だけで全体像を説明しきろうとしない。
次の箇所は、できるだけ図または表で示す。

- 仕様の入力→加工→出力。
- クラス責任と依存関係。
- 変更影響の広がり。
- 接続点で流れる値・型・操作・イベント。
- 改善前後の差分。
- 責任移動。

図を置くときは、次を守る。

- 図の前に、読者が図を見る目的を1文で書く。
- 図に出るクラス名・値・状態名は、本文または表で説明済みにする。
- 図の後に、図から読み取るべき結論を1〜3文で回収する。
- 図だけで新しい仕様や判断を初出ししない。

## 仕様説明ルール

現状の仕様を文字だけで箇条書きにしない。
必ず以下のいずれかで構造を見せる。

1. 入力→加工→出力の図。
2. 入力→加工→出力の3列テーブル。
3. 動作例テーブル。

加工は実装詳細ではなく、業務上の変換・判定・保存・通知として書く。

## コード提示ルール

コードブロックの前に、読む目的と注目点を書く。
コードブロックの後に、コードから分かったことを1〜3文で回収する。

コードは責任の固まり（入力データ・計算・組み立て・実行など）ごとに分割して掲載する。
各コードブロックの直後に、そのブロックが仕様のどの部分を担い、何をしているかを箇条書きで補う。
プログラミングに不慣れな読者が、ブロック単位で「どの仕様か・何をするか」を追える水準にする。
長いメソッド（計算・判定）は、処理の固まりにコメントや区切りを入れ、各固まりを仕様のルールへ対応づける。

現状コードの説明では、原因や責任外の判断を先取りしない。
特に1-4の実装コード説明では、構造分析や問題提起をせず、コード上の事実と仕様との対応だけを説明する。
原因分析の後で初めて、どの行にどの判断や前提が漏れているかを説明する。

### フェーズ6のコード提示

フェーズ6は、フェーズ3・4の現在グラフと同じ要求で理想の変更影響グラフを描くところから始める。現在→理想で消した経路を、切る境界・守る契約・完了条件を持つ課題カードへ落とし、その後で完了条件に効く候補を出す。候補から課題カードを逆算しない。

課題IDは独立した課題へ一つだけ割り当て、各グラフ・表の中では重複させず、P1から欠番なく連番にする。同じIDを理想グラフ、課題カード、候補、関連コードで再利用するのは、同じ課題を追跡するときだけにする。

候補の後には、課題カードの着目コードに該当するフェーズ3の関連部分だけをおさらいとして置く。フェーズ1の現行コード全体は再掲しない。

以降は課題ごとに関連部分へ絞り、次の順で示す。

1. 着目するクラス・メソッドと、フェーズ3のどの痛みを解くか。
2. 変更前の関連コード。
3. 最小変更後の関連コード。
4. 減った変更、残った問題、次へ進む理由。

中間コードの外で省略範囲を説明し、コードブロック内に `...` や「以下省略」を入れない。クラス分離が必要になった段階では、分離先、利用側、生成・所有・注入箇所を、その判断を追うために必要な関連コードとして示す。

採用後の全体コードは7-1へ置く。フェーズ6の最初に完成形を提示して、後から理由を説明する順序にしない。

コードへの気づきはコード外の散文で説明する。
`// 💭` は使わない。

## 実行結果ルール

実行結果は、最終値だけを並べない。
読者が「何を・いくらで・誰が」入力し、「どう変わって」その値になったかを追えるようにする。

- 入力（対象データ・金額・会員種別など）と結果（小計→支払金額、状態遷移など）をセットで出力する。
- 内部の件数・残高・在庫・予約数を変更するときは、対象IDと「変更前→変更後」（必要なら上限・単位）をログへ出す。結果名だけで数値変化を隠さない。
- システムが後続して行う昇格・通知・補償・履歴記録を、`main()` や運用者による別操作として実装しない。起点となるユースケースから自動的に呼ぶ。
- 状態が変わらないエラーは状態遷移図の矢印に含めず、エラー条件表と実行結果で示す。
- 割引・加算・状態変化など、施策や操作の効果が変更前後で読み取れる形にする（例：`小計 10000円 → 支払金額 8000円`）。
- 実行結果の直前に、実行対象コード・対応する動作例ケース・確認したいことを書く。
- 複数ケースを並べるときは、ケースの区切りと、どのコード（現状／変更後／最終）の結果かを示す。
- 最終値の羅列だけにして、「何が起きたか」を読者の推測に委ねない。

## コードラベルのルール

| ラベル | 用途 |
|:---|:---|
| `// ← 知らなくていい` | 原因分析後に、呼び出し元へ漏れている判断や前提を示す |
| `// ← ここだけ変わる` | 改善後に、変更影響が局所化された行を示す |
| 【深化コード】 | より難しい変化への対応 |
| 【過剰コード】 | 使いすぎた失敗例 |

`// ← 知らなくていい` は、フェーズ4以降の説明根拠があるコードで使う。
フェーズ1の現状コードには入れない。

## パターン名の扱い

パターン名は、対策を決める前の選択肢ラベルとして使わない。
章タイトルでは、読者の検索性と期待値合わせのためにパターン名を出してよい。ただし「得られること」、ステップ名、要約表、本文中の判断理由では、パターン解説前は本文で導いた構造名を使う。
問題を解いた後、その構造をチームで共有する語彙として紹介する。

## この章を読むと得られること

- 「〜できるようになる」または「〜がわかるようになる」形式で3〜4項目にする。
- パターン名を前面に出さない。
- 問題を発見できる目、使う/使わない判断、影響範囲を読む目を含める。

## 禁止表現

| 禁止する表現 | 代替 |
|:---|:---|
| 文頭・段落冒頭の「まず〜」 | 「最初の試みとして〜」 |
| 「ここで答えは出さない」等 | 削除。問いを立てるだけでよい |
| 「次章では〜」等の次章予告 | 書かない |
| 「悪いコード」「スパゲッティ」 | 「この構造では〇〇が難しい」等の事実で書く |
| 「〜すべき」「〜べきです」 | 「〜が合うかもしれません」等 |

## 数字根拠ルール

「〇箇所」「〇ファイル」という数字を出すときは根拠を添える。
方法は、ファイル名列挙、差分表、または検索例とする。

## 依存注入コードルール

インターフェースを使う実装では、依存注入のコードを示す。

1. コンストラクタでの受け取り。
2. 本番コードでの組み立て例。
3. テストでの差し替え例。

## テスト文法説明ルール

スタブ・モック・アサーションマクロが初めて登場する箇所に、1〜2行の日本語コメントを添える。

```cpp
class IPayrollFacade {
public:
    virtual int calculateNetPay(int grossPay) = 0;
    virtual ~IPayrollFacade() = default;
};

// スタブ：本物のAPIを呼ばずに、あらかじめ決めた値を返す差し替えクラス。
class StubPayrollFacade : public IPayrollFacade {
public:
    int calculateNetPay(int grossPay) override {
        return grossPay - 1200;
    }
};

class PayrollCalculator {
    IPayrollFacade& payroll;
public:
    explicit PayrollCalculator(IPayrollFacade& payroll)
        : payroll(payroll) {}

    int calculate(int grossPay) {
        return payroll.calculateNetPay(grossPay);
    }
};

StubPayrollFacade stub;
PayrollCalculator calc(stub); // コンストラクタでスタブを注入する。

// EXPECT_EQ(期待値, 実際の値)：等しければテスト通過。
EXPECT_EQ(8800, calc.calculate(10000));
```

## Kindle品質ルール

- コードブロック：1行80文字以内。
- Mermaid図：テキストでも意味が伝わる説明を必ず添える。
- 仕様図は正常系を先に描き、入力・判定・加工・正常出力を形状または色で区別する。異常系は同じ図へ混ぜず、エラー条件表で扱う。エラー条件表は正常系の各種仕様表をすべて示したあと、1-1の最後に置く。
- 脚注：使わない。
- 表：列数は最大4列まで。
- 1段落の長さ：5〜7行以内。
- コードブロックの前後：必ず説明文を置く。
