# フェーズ間整合チェック

このルールは、各フェーズ単体の達成基準とは別に、章全体で同じ仕様・同じクラス・同じ変更要求を追えているかを確認するためのものです。
読者は、前のフェーズで見た材料が次のフェーズでどう使われたかを追えなければならない。

## 基本方針

各フェーズの基準を満たしていても、フェーズ間で対象がずれていれば章としては不合格とする。

たとえば、次はすべて `error` として扱う。

- 仕様にない入力値が、動作例や `main()` に突然出る。
- 動作例にあるケースが、実行結果にない。
- クラス構成図にあるクラスが、コードに存在しない。
- コードに存在する主要クラスが、登場クラス表やクラス構成図にない。
- 変更要求で示した受入条件が、変更後コードや実行結果で確認されていない。
- フェーズ4で原因として挙げた判断や前提が、フェーズ5の課題やフェーズ6の対策に接続していない。

## 整合性台帳

章を書いたりレビューしたりするときは、本文とは別に次の台帳を頭の中または作業メモで確認する。
読者向け本文にそのまま出す必要はないが、本文・図・表・コード・実行結果はこの台帳と矛盾してはならない。

| 台帳 | 照合するもの |
|---|---|
| 仕様台帳 | 入力値、状態名、計算式、判定条件、出力、エラー条件 |
| 動作台帳 | 動作例テーブル、`main()`、実行結果、テストケース |
| クラス台帳 | 登場クラス表、クラス構成図、コード定義、依存関係、シーケンス図 |
| 変更台帳 | 変更要求、受入条件、変更後仕様、変更後コード、変更シナリオ |
| 影響台帳 | 変更箇所、変更影響グラフ、再テスト範囲、改善後影響グラフ |
| 原因台帳 | 痛み、原因、漏れている判断や前提、接続点、課題文、対策案 |

## 必須の照合ライン

### 仕様とコード粒度

- 1-1の仕様は、動作例・変更要求・実行結果に出る入力値、状態名、フラグ、ID、金額、判定条件、出力名、エラー条件を説明しているか。
- 1-1の入力→加工→出力図は、1-2の動作例で確認する代表ケースを含んでいるか。クラス名・メソッド名・変数名との対応は1-3以降で確認する。
- 仕様の「加工」は、業務上の判定・計算・変換・保存・通知として書かれ、どのコード上の責任に対応するかが後続で追える粒度になっているか。
- 仕様で扱う対象外ケースや簡略化した範囲がある場合、本文または図で「この章では扱わない」「この図では省略している」と明示しているか。
- 実際のシステムならDB、画面、外部API、ファイル出力などがあるが、掲載コードでは `print` や固定データで代替している場合、その簡略化が全体図で示されているか。
- 簡略化した入出力が、設計上の論点を変えていないことを説明しているか。
- 論点から外す処理について、実際の動き、掲載コードでの代替表現、割愛する理由、設計論点への影響が説明されているか。
- 割愛した処理が、後続フェーズで原因・課題・対策の前提として突然戻ってきていないか。

### 仕様と動作

- 1-1の入力・加工・出力が、1-2の動作例に出ているか。
- 1-2の代表ケースが、1-4の `main()` またはテストで実行されているか。
- `main()` やテストの出力が、動作例テーブルの期待結果と一致しているか。
- 実行結果の直前に、実行対象コード、対応する動作例ケース、確認したいことが明記されているか。
- 実行結果ブロックにケース名または番号、入力、出力、区切りがあり、どこからどこまでが1つの結果か分かるか。略語だけのケース名で読者を迷わせていないか。
- 現状コード、変更後コード、最終コードの実行結果が並ぶ場合、それぞれの対象コードがラベルで区別されているか。
- エラー条件が仕様にある場合、動作例・コード・実行結果のどこで扱うかが説明されているか。
- 内部数値を更新するコードと実行結果で、対象ID・変更前→変更後・上限/単位が対応しているか。
- 業務上の後続処理は起点ユースケースから自動で接続され、`main()` の手動呼び出しで補っていないか。
- 状態不変のエラーは状態遷移図へ混ぜず、エラー条件表・コード・実行結果で状態不変を確認しているか。

### クラスとコード

- 1-3の登場クラス表にあるクラスが、1-4のコードに存在するか。
- 1-4の主要クラスが、1-3の登場クラス表またはクラス構成図に存在するか。
- クラス構成図の依存関係が、コード上のメンバ、引数、生成、呼び出しと矛盾していないか。
- フェーズ3〜6の中間コードや説明用スニペット（コンパイル対象外でも）が、1-4で定義したクラス名・フィールド名・公開メソッドと対応しているか。存在しないフィールド（例：`Order` にない `customerType`）を使っていないか。
- 中間コードが差分抜粋なら、フェーズ1から維持するDB・Repository・Registry・Gateway・履歴・検証・副作用を「中間コードの継続条件」として具体名で示しているか。コードに出ないことを、削除や仕様変更として扱っていないか。
- シグネチャが変わる場合、同じ責任の単なる表記揺れか、公開操作から委譲先契約・要求/結果オブジェクト・操作オブジェクト・組み立て責任への移動かを区別しているか。後者なら、値と操作の移動先を説明し、入力や副作用が失われていないことを追えるか。
- フェーズ6冒頭に、フェーズ1の現行コード全体と、フェーズ3で変更した関連部分の痛みの差分が続けて置かれているか。全体構造の振り返りだけで対策の起点をフェーズ1へ巻き戻していないか。
- フェーズ5の各課題が、フェーズ6で課題ID、着目クラス・メソッド、最小変更、完了条件へ対応づけられているか。
- フェーズ6の各段階で、関連部分の変更前・変更後コードから「減った変更」「残った問題」「次へ進む理由」を追えるか。残った問題と無関係な構造を追加していないか。
- フェーズ6の関連コードは、対策前全体コードから変更していない行の順序・インデント・改行を維持しているか。コード整形を構造変更へ混ぜず、読者が同じ行を視線で対応づけられるか。
- クラスを分離する段階で、分離先だけでなく生成場所・所有者・生存期間・依存注入・利用側の呼び出しまで、判断に必要な関連コードがあるか。呼び出し元が具体クラスを知る範囲と、組み立て後に抽象だけを見る範囲を追えるか。
- 採用後の全体コードがフェーズ6で先出しされず、7-1で初めて全体として統合されているか。
- 7-1の最終コードにある主要クラスが、7-2の動作シーケンス図と7-3の改善後影響グラフに反映されているか。

### 変更要求と変更後の確認

- 1-5の変更要求が、2-3の確定変更と同じ内容を扱っているか。
- 2-5の将来リスクを、フェーズ7で扱う変更シナリオや使う/使わない判断と対応づけているか。
- 3-1で試した変更が、1-5の変更要求から来ているか。
- 3-2の変更影響グラフが、3-1の変更箇所と一致しているか。
- 7-4の変更シナリオ表が、1-5の変更要求、2-5の将来リスク、フェーズ6の採用判断と対応しているか。

### 痛み・原因・課題・対策

- 3-3の痛みが、3-1の変更作業と3-2の変更影響から導かれているか。
- 4-1の原因が、3-3の痛みに対応しているか。
- 4-2で分離すべき判定・処理・生成判断が、4-3の漏れている判断や前提と対応しているか。
- 5の課題文が、4の原因分析から導かれているか。
- 6の各対策案が、5の課題文へ応答しているか。
- 7の最終コードが、6で採用した案を実装しているか。

## 判定基準

| 重大度 | 条件 |
|---|---|
| error | 仕様・動作例・コード・図・実行結果の対象が食い違い、読者が同じものを追えない |
| error | 実行結果がどのコード・どの動作例ケースに対応するか分からない |
| warning | 実行結果の区切りが曖昧で、複数ケースの境目を読者が推測する必要がある |
| error | 仕様が動作例・変更要求・実行結果に出る値・状態・判定条件・出力と紐づく粒度で定義されていない |
| error | 実装を簡略化しているのに、省略した外部要素や代替表現が説明されていない |
| error | 論点から外す処理の実際の動き・割愛理由・設計論点への影響が説明されていない |
| error | 割愛した処理が後続フェーズの原因・課題・対策に影響している |
| error | クラス名、入力値、状態名、変更要求、受入条件がフェーズ間で別物になっている |
| error | 原因、課題、対策の対応が切れている |
| warning | 対応はあるが、本文で橋渡しされておらず読者が推測する必要がある |
| warning | 図や表には出ているが、本文で読み取る結論が回収されていない |

## レビュー時の出力

指摘するときは、次の形式にする。

```text
対象:
不整合:
照合元:
照合先:
修正方針:
```

例:

```text
対象: 1-3 登場クラスと 1-4 現状コード
不整合: クラス構成図に OrderValidator があるが、掲載コードに定義がない
照合元: 1-3 クラス構成図
照合先: 1-4 現状コード
修正方針: 図から削除するか、コードと責任表に OrderValidator を追加する
```
