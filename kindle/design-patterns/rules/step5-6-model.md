# ステップ5〜7 モデル定義

## ステップ3：課題特定 ―― 変更シミュレーションで痛みを発見する

変更要求を今のコードに当てはめてみる。このとき：
- 「置換しようとしたが密結合で差し替えられない」
- 「新しいケースを追加しようとしたら既存コードを大量に変えることになった」

こうした「置換・拡張のやりづらさ」がステップ3で発見する課題の典型例。

---

## ステップ4：原因特定 ―― 「分けるべき場所があるか」を特定する

痛みの根本にある問いは1つ。

> 「この塊の中に、独立して変わる部分があるか？」

| 観察した状況 | 判断 |
|:---|:---|
| 変わる理由（決定者）が異なる | → **分ける** |
| いつも一緒に変わる | → **分けない** |

「分ける」と判断した場合、「単体」と「接続点」が生まれる。接続点の形はステップ5で決める。

---

## ステップ5：対策検討 ―― 分け方と接続の形を段階的に決める

### 接続の2軸

接続点の形を決める軸は2つある。

- **具体 vs 抽象**：特定クラスへの依存か、インターフェースへの依存か
- **直接 vs 間接**：直接呼び出すか、中間層を挟むか

### 「なぜ分けたか」と接続の形の対応

| なぜ分けたか | 接続の形 |
|:---|:---|
| 責任を整理したい | 具体×直接 |
| 実装を差し替えたい | 抽象×直接 |
| 知らせたくない・複雑さを隠したい | 具体×間接 |
| 差し替えたい かつ 知らせたくない | 抽象×間接 |

接続点が複数あれば、それぞれ独立に判断する。

### 対策案の作り方

**必須の2案：**

| 案 | 内容 |
|:---|:---|
| **案0：分けない** | コード構造はそのまま。`if` 文追加・定数変更などで対応 |
| **案N：完璧に対応** | ステップ4で特定した分け方＋必要な接続の形をすべて適用 |

**中間案（内容に応じて追加）：**

中間案は「1つの接続点の形を変えるごとに1案」として作る。

```
例：案N で 具体×直接 → 抽象×直接 → 抽象×間接 と変える場合

案0：分けない（構造を変えない）
案1：分けるが具体×直接（シンプルな分離）
案2：分けて抽象×直接（インターフェース導入）
案N：分けて抽象×間接（中間層も導入）
```

意味のある段階がある場合だけ中間案を追加する。段階を刻む意味がない場合は案0と案Nの2案だけでよい。

---

## ステップ6：比較・決定 ―― コスト天秤で選ぶ

対策案を「現在の対応コスト」と「未来の対応コスト」で比較する。

| 対策案 | 現在の対応コスト | 未来の対応コスト |
|:---|:---|:---|
| 案0：分けない | 低 | 高 |
| （中間案 ※内容次第） | 中 | 中 |
| 案N：完璧に対応 | 高 | 低 |

### コストの算出根拠

未来の対応コストを見積もるときの5要素：

- **置換のしやすさ**：実装を差し替えるとき、変更箇所は最小限か
- **拡張のしやすさ**：新しいケースを追加するとき、既存コードを触らずに済むか
- **テストのしやすさ**：依存を切り離してテストを書けるか
- **読みやすさ**：変更前に構造を理解する工数
- **メモリ・パフォーマンスへの影響**：接続の形によって異なるオーバーヘッドがある

| 接続の形 | パフォーマンス上の特徴 |
|:---|:---|
| 具体×直接 | オーバーヘッド最小 |
| 抽象×直接（virtual 関数） | vtable ルックアップ、ヒープ確保コスト |
| 抽象×直接（template） | コンパイル時間増大、バイナリサイズ増大 |
| 具体×間接 | 間接参照コスト |
| 抽象×間接 | vtable ルックアップ＋間接参照 |

### 判断パターン

| 現在コスト | 未来コスト | 判断 |
|:---|:---|:---|
| 低い | 低い | 迷わず採用 |
| 高い | 低い | 長期重視なら採用。納期優先なら案0を暫定対応＋リファクタ計画 |
| 低い | 高い | 案0の暫定対応として採用。リファクタ計画を必ず明記 |
| 高い | 高い | その案を再検討 |

「今すぐ納期を間に合わせたいか、将来も低コストで完成させたいか」——この問いが判断の本質。
