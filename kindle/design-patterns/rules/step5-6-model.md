# rules/step5-6-model.md
# ステップ5・6の思考モデル（最重要）
# CLAUDE.md から参照される。このファイルが唯一の正。

---

## 概要

読者が「なぜその手段に辿り着いたのか」を納得しながら読み進めるために、
Step5・Step6は必ず以下の**人間の思考回路**の順序で書く。

```
原因（ステップ4で言語化）
  ↓
【方向性】原因の種類から「大まかな解消方向」が自然に出る
  例：「変わるものと変わらないものが同居」→「切り出す・分ける」方向
  例：「依存が分散している」→「統合する・窓口を作る」方向
  例：「状態と振る舞いが分散」→「集める・1か所にまとめる」方向
  ↓
【手段①】方向性の中で最初に人が思いつくシンプルな案
  → コードを試してみる
  → 「残る課題」を明示する（次の発想の転換への橋渡し）
  ↓
【手段②】手段①の残課題を踏まえた、より本質的な案
  → コードを試してみる
  → 変更後クラス図を示す
  → ここで初めてパターン名を登場させる
  ↓
（ステップ6へ）
【評価軸の宣言】比較の前に「この状況で何を重視するか」を先に宣言する
  → 後から基準を決めると結論ありきになる
  ↓
【手段①vs②の比較】宣言した評価軸で両方を測る
  → 「今回の状況ではなぜ②が優れるか」を表で示す
  ↓
【採用の決定】評価軸と比較結果から手段②を採用
  ↓
【耐久テスト】採用した手段②が、ヒアリング（ステップ2）で挙がった
  「将来の変化」に実際に耐えられるかを確認する
  ↓
【使う/使わない判断】このパターンを使うべき状況・使わないべき状況
```

---

## 原因の種類と「方向性」の対応表

| 原因の種類（ステップ4の言語化） | 自然に出てくる方向性 |
|---|---|
| 変わるものと変わらないものが同居 | 切り出す・分ける |
| 呼び出し元が実装詳細を知りすぎている | 隠す・境界を引く |
| 状態と振る舞いの対応関係が分散している | 集める・1か所にまとめる |
| 処理の骨格と変わる中身が混在している | 骨格を固定し、変わる中身を委ねる |
| 依存が複数箇所に分散している | 統合する・窓口を作る |
| 「実行するもの」と「実行するタイミング」が同居 | 実行内容をオブジェクトとして切り出す |

---

## 手段①・手段②の書き方ルール

- **手段①** は「人が自然に最初に思いつく案」として書く。失敗例・間違いではない。「限界のある有効な選択肢」として描く。
- 手段①の後に必ず「残る課題」を明示する。これが手段②への橋渡しになる。
- **手段②** は手段①の残課題を受けた「発想の転換」として書く。「いきなり正解が出てくる」ではなく、①の課題への応答として登場させる。
- 手段①と②を「大きく2つのアプローチがあります」と並列に並べてはいけない。必ず①→②の順序の流れとして書く。

---

## ステップ6の構成要素（必須）

| 要素 | 内容 |
|---|---|
| 評価軸の宣言 | 比較より前に「この状況で何を重視するか」を表で宣言 |
| 手段①vs②の比較表 | 宣言した評価軸ごとに両手段を評価 |
| 採用の決定 | 比較結果から「今回の状況では②」と明示 |
| 耐久テスト | ステップ2ヒアリングで挙がったリスクを実際に変化させて確認 |
| 使う/使わない判断 | 【過剰コード】例 + 使うべき状況/使わないべき状況の表 |
