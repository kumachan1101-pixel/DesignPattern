# ステップ5・6の思考モデル（最重要）
# CLAUDE.md から参照される。このファイルが唯一の正。

## 概要

読者が「なぜその手段に辿り着いたのか」を納得しながら読み進めるために、
Step5・Step6は必ず以下の**人間の思考回路**の順序で書く。

**この思考モデルの根拠は第0章「問題を発見した後の手札」にある。**
各章のステップ5・6は、第0章の4分類＋手札一覧から「この状況に合う候補」を選び出すプロセスそのものである。

```
原因（ステップ4で言語化）
  ↓
【第0章の4分類への当てはめ】
  原因は4分類のどれに該当するか？
    1. 分ける（分離）  2. まとめる（統合・隠蔽）
    3. 間に挟む（中間層）  4. モノ化する（実体化）
  ↓
【手札候補の絞り込み】
  該当する分類の手札一覧（第0章）から、この状況に当てはまる候補を列挙する。
  例：「1. 分ける」に該当 → 候補：インターフェース抽出／骨格・ステップ分離
  ↓
【候補の数で分岐する】（★ここが核心）

  ── 候補が1つの場合 ──────────────────────────
  その手札を直接適用する（ステップ6の天秤は不要）。
  → コードを試してみる
  → 変更後クラス図を示す
  → ここで初めてパターン名を登場させる
  → ステップ6：耐久テスト → 使う/使わない（天秤はスキップ）

  ── 候補が2つ以上の場合 ──────────────────────
  【手段①】候補の中で最初に人が思いつくシンプルな案
    → コードを試してみる
    → 「残る課題」を明示する（次の発想の転換への橋渡し）
  ↓
  【手段②】手段①の残課題を踏まえた、より本質的な案
    → コードを試してみる
    → 変更後クラス図を示す
    → ここで初めてパターン名を登場させる
  ↓
  （ステップ6へ）
  【評価軸の宣言】比較の前に「この状況で何を重視するか」を先に宣言する
    → 後から基準を決めると結論ありきになる
  ↓
  【手段①vs②の比較】宣言した評価軸で両方を測る
    → 「今回の状況ではなぜ②が優れるか」を表で示す
  ↓
  【採用の決定】評価軸と比較結果から手段②を採用

  ── 候補数に関わらず必須 ─────────────────────
  【耐久テスト】採用した手段が、ヒアリング（ステップ2）で挙がった
    「将来の変化」に実際に耐えられるかを確認する
  ↓
  【使う/使わない判断】このパターンを使うべき状況・使わないべき状況
```

## 第0章 4分類と手札の対応（原因を分類するための参照表）

| 分類 | 原因の種類（ステップ4の言語化） | 手札ラベル（第0章より） |
|---|---|---|
| **1. 分ける** | 任意の振る舞いが変わる（アルゴリズム・ルール等） | インターフェース抽出 |
| **1. 分ける** | 共通の骨格に対する部分的なステップが変わる | 骨格・ステップ分離 |
| **1. 分ける** | 巨大なクラスが複数の異なる役割を抱えている | クラス分割 |
| **2. まとめる** | 外部システムのAPI仕様や呼び出し順序が変わる | 窓口集約 |
| **2. まとめる** | 機能の組み合わせ（ログ・リトライ等）が変わる | コンポジション転換 |
| **3. 間に挟む** | 通知先や依存方向が変わる | 依存の逆転 |
| **3. 間に挟む** | オブジェクト間の多対多の複雑なやり取りが変わる | 仲介者の導入 |
| **4. モノ化する** | 要求の発生と実行のタイミングがずれる | 振る舞いのデータ化 |
| **4. モノ化する** | オブジェクトの状態とそれに伴う振る舞いが変わる | 状態クラス化 |

## 手段①・手段②の書き方ルール（候補が2つ以上の場合のみ適用）

- **手段①** は「人が自然に最初に思いつく案」として書く。失敗例・間違いではない。「限界のある有効な選択肢」として描く。
- 手段①の後に必ず「残る課題」を明示する。これが手段②への橋渡しになる。
- **手段②** は手段①の残課題を受けた「発想の転換」として書く。「いきなり正解が出てくる」ではなく、①の課題への応答として登場させる。
- 手段①と②を「大きく2つのアプローチがあります」と並列に並べてはいけない。必ず①→②の順序の流れとして書く。

## ステップ6の構成要素

| 要素 | 候補1つの場合 | 候補2つ以上の場合 |
|---|---|---|
| 評価軸の宣言 | 不要 | **必須**（比較より前に宣言） |
| 手段①vs②の比較表 | 不要 | **必須** |
| 採用の決定 | 不要（直接適用のため） | **必須** |
| 耐久テスト | **必須** | **必須** |
| 使う/使わない判断 | **必須** | **必須** |

**「使う/使わない判断」の必須要素：** 【過剰コード】例 ＋ 使うべき状況/使わないべき状況の表
