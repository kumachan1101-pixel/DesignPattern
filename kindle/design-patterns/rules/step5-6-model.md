# ステップ5・6の思考モデル（最重要）

## 用語の定義（混同防止）

| **用語** | **定義**                                                                                               |
| ------ | ---------------------------------------------------------------------------------------------------- |
| **手札** | 第0章の「4つの物理操作」一覧に載っている設計操作（①分割、②隠蔽、③規格化、④間接化）。 <br><br>※「⑤置換」「⑥拡張」は手札（操作）ではなく、手札を使った結果得られる「評価・能力」である。 |
| **手段** | ステップ5で実際にコードとして試みるアプローチ（構造的対策案）。候補が複数あって段階的に試す場合に手段①・手段②と番号を振る。                                      |
| **対策** | 第0章で「構造的対策（手札）」として登場する。各章では「手札」で統一し「対策」は使わない。                                                        |

> **重要：** 手段①・手段②は、「naive（素人っぽい）な間違い → 正しい正解」という構図ではない。手段①は「分離・隠蔽」を尽くした手続き型としての理想形であり、手段②はそこからさらに「規格化・間接化」を足して「置換・拡張」の評価を獲得するステップアップである。

## ステップ6 の構成

|**要素**|**ケース①**|**ケース②**|**ケース③（基本形）**|
|---|---|---|---|
|**評価軸の宣言**|不要|不要|必須（比較より前。「置換」「拡張」等の軸）|
|**手段①vs②の比較表**|不要|不要|必須|
|**採用の決定**|不要|不要|必須|
|**耐久テスト**|必須|必須|必須|
|**使う/使わない判断**|必須|必須|必須|

> 「使う/使わない判断」の必須要素： 【過剰コード】ラベルの例 ＋ 使うべき状況/使わないべき状況の表

## 手段①・手段②の書き方ルール

- **手段①は最大限洗練させる：** 手段①は「適当に書いた悪いコード」ではなく、手札①（分割）や②（隠蔽）を極限まで適用した「手続き型としての理想形」として書く。
    
- **「評価」を壁にする：** 手段①の後に、「整理整頓はできたが、新しい仕様が来た際に既存コードを触らずに追加（＝⑥拡張）できるか？」という評価の壁を提示する。これが手段②へ進む根拠になる。
    
- **並列に扱わない：** 手段①と②を「2つのアプローチがあります」と並列に並べてはいけない。「まず①・②の操作で要素を整理した。しかし『拡張』という評価には届かない。だから③の規格化（インターフェース化）が必要なんだ」という直列のストーリーで書く。
    

## 手札が見つからない場合

第0章の「4つの手札一覧」に適切な手札が見当たらない場合：

- ステップ4に戻り、原因の次元（要素・関係）の言語化を精緻にする。
    
- それでも見当たらなければ著者に報告する。
    

## 第0章 構造的対策（4つの操作）と事後評価の参照表

手段となる「物理操作（手札）」と、その結果得られる「評価の軸」を明確に分けた形です。

**【手段：基本となる物理操作（手札）】**

|**次元**|**物理操作（手札）**|**原因の種類（コードの臭い）**|**構造的対策案（具体）**|
|---|---|---|---|
|**要素**|**① 分割する**|クラスが巨大、責任が混在、共通処理が散在|クラスや関数の分割、処理の共通化|
|**要素**|**② 隠蔽する**|生データの露出、状態の直接保存、手順の複雑さ|値オブジェクト化、窓口集約によるカプセル化|
|**関係**|**③ 規格化する**|型や引数による分岐の増殖、具象への直接依存|インターフェース抽出によるつなぎ目の統一|
|**関係**|**④ 間接化する**|型の不一致、多対多の複雑な通信、タイミングのズレ|翻訳層・仲介者・代理人の導入|

**【目的・結果：設計の事後評価（得られる能力）】**

| **次元** | **評価の軸（目的）** | **解決したい課題（コードの限界）**       | **達成される構造（手段を組み合わせた結果）**              |
| ------ | ------------ | ------------------------- | ------------------------------------- |
| **発展** | **⑤ 置換できるか** | 状況による振る舞いの変化をif文で縛っている    | 既存コードに触れず、規格化された部品を動的に差し替え可能になる       |
| **発展** | **⑥ 拡張できるか** | 新機能追加のたびに既存のコアロジックを修正している | 既存コードを壊さず、新しい振る舞いや機能をプラグインのように追加可能になる |
