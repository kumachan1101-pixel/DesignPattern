# フェーズ5〜7 設計判断ガイド

このガイドは、接続を固定的な種類へ分類するためのものではない。
変更要求が通過する接続点を見つけ、どの知識を移すかを決めるために使う。

詳細な共通ルールは `connection-point-review.md` を参照する。

## フェーズ5：課題定義

### 接続点の特定

分けたい責任の境界で、実際に受け渡されているものを確認する。

- 値と型
- 呼び出す操作
- イベント
- 生成済みオブジェクト
- 処理順序

### 知識の所在

| 観点 | 確認内容 |
|---|---|
| 変化の種類 | 境界の両側は、それぞれどの機能・仕様変更で変わるか。担当者や関係者が分かる場合は補助情報として使う |
| 漏れている知識 | クラス名、生成方法、条件、処理順序 |
| 安定する約束 | 呼び出し元が本当に必要とする操作・データ |
| 変更影響 | 修正・再テストが必要になるファイル |
| 制約 | 性能、メモリ、納期、チームの習熟度 |

### 受け入れ条件

課題ごとに、変更要求を再適用したときの影響範囲として受け入れ条件を定める。
「新しいクラス1つと登録だけ」のような構造の形では書かず、役割の言葉で書く。

> 変更要求【X】を再適用したとき、修正が【許す範囲（役割の言葉）】に収まり、
> 【守る範囲】が修正・再テスト対象に入らなければ、解けたと言える。

### 課題の書き方

「AをBへ変える」のように手段を先に決めない。移す先（配置）はフェーズ6で決める。

> 接続点Aでは、【変わる側】と【守る側】の間で【値・型・操作・イベント】が流れている。
> 呼び出し元へ【知識】が漏れており、変更のたびに修正が波及する。

## フェーズ6：対策検討

フェーズ6では3つを順に決める。**分離**（どの知識をどの境界で切るか）、**配置**（切り出した知識をどの既存・新設クラスへ置くか）、**組み立て**（誰が生成・所有し、どの方法で注入・登録・継承・遷移して繋ぎ、安定側はどの契約だけを呼ぶか）。この3つが揃って初めて対策が決まったと言える。

候補はフェーズ5の受け入れ条件で評価する。読者が思いつく単純な案（関数化・リスト保持など）も候補に含め、採用しない場合は理由と（必要なら章末コラムで）コードイメージを示す。対策は必要な段階だけを比較する。

1. 関数へ切り出す
2. 別クラスへ責任を移す
3. 交換する実装が複数ある場合、共通の契約を定める
4. 複雑な手順を隠す必要がある場合、窓口や仲介役へ移す
5. 動的に組み合わせる必要がある場合、コンポジションを使う
6. 生成方法が変わる場合、生成責任を移す

各案について、次を示す。

| 比較項目 | 内容 |
|---|---|
| 移す判断・処理・生成責任 | 判定条件、計算式、処理順序、生成判断などをどの責任へ移すか |
| 接続点 | 何を受け渡すか |
| 減る変更 | 今回の要求で触らなくなる箇所 |
| 残る変更 | 組み立て・登録・設定など必要な変更 |
| 導入コスト | クラス数、設定、学習、実行時コスト |

パターン名は、対策を決める前の選択肢ラベルとして使わない。

## フェーズ7：対策実施

1. 採用案の全体コードを示す
2. 動作例を実行して、振る舞いを維持できたことを確認する
3. フェーズ2の変更要求を再度当てる
4. 修正箇所と再テスト範囲を、改善前と比較する
5. 組み立て箇所や登録処理など、残る変更も明記する
6. 最後に、その構造の既知の名前としてデザインパターンを紹介する

「既存コードを一切変更しない」「新クラス1つだけ」と断定せず、
対象変更シナリオと組み立て箇所を含めて説明する。
