# 第9章 総合演習 ―― 複数のパターンを組み合わせて複雑な課題を解く

## ステップ0：システムを把握し、仮説を立てる

**この章で扱うシステム：多機能自動券売機**

時間帯割引、会員割引、キャンペーン割引など複雑な割引計算が必要で、かつ「待機中」「硬貨投入中」「発券中」といった状態遷移も複雑な券売機システムです。

**変動と不変の仮説**

| 分類 | 仮説 | 根拠 |
|---|---|---|
| 🔴 **変動する** | 割引計算のアルゴリズム | キャンペーンや施策によって頻繁に変わるため |
| 🔴 **変動する** | 状態ごとのボタン操作の可否 | 新しい支払い手段（QRコード決済など）が増えると状態遷移が変わるため |
| 🟢 **不変** | 券売機の基本的なハードウェア制御 | 画面表示や発券の物理機構は安定しているため |

---

## ステップ1：実装コードを読む ―― 巨大なif文のジャングル

### 9.1 現状のコード

```cpp
void TicketMachine::onButtonPress(Button btn) {
    if (state == IDLE) {
        if (btn == START) {
            double price = basePrice;
            if (isEvening) price *= 0.8; // 割引ロジックが混在
            // ...
            state = ACCEPTING_COINS;
        }
    } else if (state == ACCEPTING_COINS) {
        // ...
    }
}
```

**問題の発見：**
1つのメソッドの中に、「状態遷移の判断」と「複雑な割引計算」という、全く異なる2つの「変わる理由」が混在しています。

---

## ステップ2：仮説を確定する ―― 関係者ヒアリング

**関係者の声：**
- **マーケティング：** 「来週から期間限定のコラボ割引を導入したいです。」
- **保守チーム：** 「QRコード決済の導入により、新しい待機状態が増える予定です。」

---

## ステップ3：課題分析 ―― 変更が来たとき、どこが辛いか

「新しい割引」を追加するだけで、状態遷移のコードも触らなければならず、券売機が動かなくなる（デグレード）リスクが常にあります。

---

## ステップ4：原因分析 ―― 困難の根本にある問題を言語化する

**真因：**
1. 割引ロジック（Strategy）が券売機のメインロジックに埋め込まれている。
2. 状態ごとの振る舞い（State）が、巨大な `if-else` で分断されている。

---

## ステップ5：対策案の検討 ―― 方向性を決め、手段を順に試す

### 9.5.1 方向性の特定

原因が2つあるため、解決も2段階で行います。

**原因1への方向：** 割引計算という「アルゴリズム」を切り出して外部に委ねる（Strategy）。
**原因2への方向：** 状態ごとの振る舞いを1箇所に集める（State）。

---

### 9.5.2 手段①：フラグとメソッド分割（直感的な案）

割引計算を `calculateDiscount()` メソッドに切り出し、状態遷移はフラグで管理する方法です。

**残る課題：**
メソッドに分けても、`TicketMachine` クラス自体の肥大化は止まらず、新しい割引や状態が増えるたびにこのクラスを修正し続ける必要があります。

---

### 9.5.3 手段②：StrategyとStateの組み合わせ（発想の転換）

第0章の手札：「アルゴリズムを切り替える」＋「状態をクラスにする」。

**「割引計算をStrategyとして、券売機の状態をStateとして、それぞれ独立したオブジェクトにする。」**

---

## ステップ6：天秤にかける ―― 手段①と手段②を比べる

### 9.6.1 評価軸の宣言

1. **変更の局所性**：割引の変更が状態遷移に影響しないか
2. **テストの容易さ**：割引ロジックだけを単体テストできるか

### 9.6.2 手段①vs手段②の比較

| 評価軸 | 手段①（メソッド化） | 手段②（複合パターン） |
|---|---|---|
| 変更の局所性 | ❌ TicketMachineクラスの修正が必須 | ✅ 新しいStrategy/Stateクラスを追加するだけ |
| テストの容易さ | ❌ 券売機全体を動かさないとテスト困難 | ✅ 割引クラス単体で計算結果を検証可能 |

**結論：手段②を採用します。**

---

## ステップ7：決断と、手に入れた未来

### 9.7 解決後のコード（構造）

```cpp
class TicketMachine {
    IState* state;
    IDiscountStrategy* discount;
public:
    void setDiscount(IDiscountStrategy* d) { discount = d; }
    void press() { state->handle(*this); }
};
```

## 整理

この最終章では、これまで学んできた「原因分析」から「手札の選定（Strategy, State）」、そして「天秤にかける」プロセスを複合的に適用しました。

### 振り返り：3つの哲学

- **哲学1（変わるものをカプセル化）**：割引と状態、2つの変動要因を別々のクラスに閉じ込めた。
- **哲学2（インターフェースに対してプログラム）**：券売機本体は2つのインターフェースだけを知っている。
- **哲学3（コンポジション優先）**：2つのパーツを組み合わせることで、複雑な要件をシンプルに解いた。
