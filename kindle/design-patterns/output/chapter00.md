―― デザインパターンは「考えた結果」に過ぎない

## この本のコードについて

この本のコード例はすべて **C++** で書いています。
ただし、テンプレート・スマートポインタ・ラムダ式などの
高度なC++機能は使いません。
クラス・継承・ポインタという基本的な構造だけで書いています。

C++を選んだ理由は、ポインタを通じて
「このクラスが何を知っていて、何を知らないか」が
コード上で明示的に見えるからです。

**考え方は言語に依存しません。**
Java・C#・Python・TypeScript——どの言語でもそのまま適用できます。

## なぜ「パターンを覚えても使えない」のか

ソフトウェア設計を学ぼうとすると、必ずと言っていいほど
「GoFのデザインパターン」に出会います。
本で学び、構造図を頭に入れ、いざ自分のコードに使おうとしたとき——

「どこに適用すればいいのか、わからない。」
「無理に使ってみたら、かえってコードが複雑になった。」

私自身、同じ壁に何度もぶつかりました。
パターンの名前と図は頭に入った。でも、
目の前の問題にどう当てはめればいいのか、
まるでわからなかったのです。

その感覚、うまく伝わっているでしょうか。

なぜ、実績のある優れた設計手法が、
時にはコードをより複雑にしてしまうのか。

理由はシンプルです。
**パターンを「最初から目指すべき答え」として扱っているから**です。

デザインパターンは、先人たちが泥臭い現場で問題に向き合い、
いくつかの選択肢を天秤にかけ、
「この状況ではこれが一番割に合う」と判断した
**決断の結果**として生まれたものです。

結果だけを真似ても、状況が違えばうまく機能しません。
大切なのは、その結果に至るまでの**思考のプロセス**を体験することです。

## まず伝えたいこと：「分けない」という選択肢

この先、「変わるものと変わらないものを分ける」という考え方を繰り返し使います。
その前に、一つだけ伝えておきたいことがあります。

**迷ったら、分けない。**

分けることには必ずコストが伴います。
クラスが増える。ファイルが増える。間接参照が増える。
読むコードの量が増える。

「将来変わるかもしれないから、今のうちに分けておこう」
——この判断で失敗した経験は、私にも数え切れないほどあります。

分けるのは、**痛みが出てから**で十分です。
「今、ここが困っている」という具体的な痛みがないのに
予防的に分けるのは、たいていの場合、過剰設計になります。

この本は「分け方」を教えますが、同時に
**「分けなくていい場面を見極める力」**も伝えたいと思っています。

## すべてのパターンを貫く「1つの原則」と「2つの技法」

GoFの23のデザインパターンは、一見バラバラに見えます。
でも、すべてのパターンは、たった1つの原則を
それぞれの状況に具体化したものに過ぎない——と、私は整理しています。

その原則とは：

> **「変わるものと変わらないものを見分け、分ける」**

これがすべての出発点です。
そして、この原則を実現するための技法が2つあります。

```
原則：変わるものと変わらないものを見分け、分ける
├── 技法1：分けた後、インターフェースで繋ぐ
└── 技法2：繋ぐとき、継承よりコンポジション
```

3つは対等ではありません。
原則が根本にあり、技法1と技法2は原則を実現する手段です。

### 原則：変わるものをカプセル化せよ

「変わりやすい部分」と「変わってほしくない部分」を、同じ場所に書かない。

GoF 23パターンのほぼすべては、この1つの哲学から導けます。
Strategyは「アルゴリズムの実装」を隔離し、
Observerは「通知先の種類」を隔離し、
Facadeは「外部サービスの詳細」を隔離します。
やっていることは全部同じで、「変わりやすいものを取り出して別の場所に閉じ込める」だけです。

**この哲学を使うための問い——この本を通じて使い回せる1つの問い：**

> 「このコードの中に、**『変わる理由』が異なる2つのものが、同じ場所に混在していないか？**」

パターンが違っても、問いはこれひとつです。
「何と何が混在しているか」の組み合わせが違うだけで、
本質的に問いかけていることは変わりません。

**コードで確かめる：**

```cpp
// NG：計算ロジックと出力形式が同じクラスに混在している
//      計算が変わっても、出力形式が変わっても、このクラスを変える羽目になる
class ReportService {
    void generate() {
        double value = calcAmount();      // 計算ロジックが変わる理由
        std::string text = format(value); // 出力形式が変わる理由
        writeToPdf(text);
    }
};

// OK：変わる理由ごとに分離する
//     計算が変わっても ReportFormatter は変わらない。出力が変わっても Calculator は変わらない
class Calculator    { virtual double calcAmount() = 0; };
class ReportFormatter { virtual std::string format(double v) = 0; };

class ReportService {
    Calculator*     calc_;
    ReportFormatter* fmt_;
public:
    void generate() { fmt_->format(calc_->calcAmount()); }
};
```

#### 「変わるもの」をどうやって見分けるか

コードを見て「これは変わりそうだ」と感じることは誰にでもできます。
問題は、**その直感が正しいとは限らない**ことです。

見分けるための問いは1つだけです：

> 「このコードを変更するとき、**誰に話を聞きに行くか？**」

答えが2人以上になるなら、そのコードには「変わる理由」が2つ以上混在しています。
各章で繰り返し使う「共通の問い」は、この見分け方を具体的に適用したものです。

**コードで確かめる：**

```cpp
// 「誰に聞きに行くか？」で判断する
void processOrder(Order order) {
    double total = order.amount;
    total *= 0.9;           // ← 営業に聞く（割引率）
    sendEmail(order.email); // ← インフラに聞く（通知手段）
    saveToDB(order);        // ← DBAに聞く（保存先）
}
// → 3人に聞く = 3つの理由で変わる = 分けるべき
```

ただし、**「変わるかもしれない」だけで分けるのは危険です。**

```cpp
// やりすぎた例：変わる予定がないものまで隔離した
class ITaxCalculator {
public:
    virtual double calc(double amount) = 0;
    virtual ~ITaxCalculator() {}
};

class TaxCalculatorImpl : public ITaxCalculator {
public:
    double calc(double amount) {
        return amount * 1.1; // 消費税。当面変わらない
    }
};
// → 経理担当と「統一ルール」と合意済み
// → 変わらないものを分けても、複雑さが増えるだけ
```

| 変わりそう？ | 関係者と確認した？ | 判断 |
|---|---|---|
| Yes | Yes（変わると確認） | **分ける** |
| Yes | No（未確認） | **確認してから判断** |
| No | — | **分けない** |

カプセル化は「念のため」の道具ではありません。
**「変わることが確認できたから、影響を閉じ込める」**ための道具です。

### 技法1：分けた後、インターフェースで繋ぐ

「何をするか（契約）」と「どうやるか（実装）」を分ける。

呼び出す側が `EmailNotifier` という具体クラスを知っていたら、
Emailの送り方が変わるたびに呼び出す側も変わります。
でも `INotifier` というインターフェースだけを知っていれば、
Email→SMS→Pushに切り替わっても呼び出す側はまったく変わりません。

この哲学は「変わる理由の異なるものを分離した後、その境界を何でつなぐか」の答えです。
インターフェースが、安定した呼び出し側と不安定な実装側の間に立つ「緩衝材」になります。

**コードで確かめる（依存の方向）：**

```cpp
// NG：具体クラスに直接依存している
//     EmailSender が変わるたびに Notifier も変わる
class Notifier {
    EmailSender* sender_; // 具体クラスを知っている
public:
    void notify(std::string msg) { sender_->send(msg); }
};

// OK：インターフェースに依存する
//     IMessageSender の実装が Email→SMS→Push に変わっても Notifier は変わらない
class IMessageSender {
public:
    virtual void send(std::string msg) = 0;
    virtual ~IMessageSender() {}
};

class Notifier {
    IMessageSender* sender_; // 契約（インターフェース）だけを知っている
public:
    void notify(std::string msg) { sender_->send(msg); }
};
```

ただし、インターフェースが守れないものがひとつあります。**引数の型が変わるとき**です。

同じ引数の型（例：`int userId`）を複数のインターフェースで共有しているとき、
「`int` を `string` に変えてほしい」という要求が来ると、
そのすべてのインターフェースのシグネチャが変わります。
どんな設計構造があっても、この変更からは逃げられません。

こういう状況に直面したとき、立てるべき問いは「なぜこのパターンでは守れないのか」ではなく、
**「この型はどこまで安定していると言えるか？チームで合意できているか？」** です。
型を決める前に、システムの所有者や連携する部署の担当者に確認することが、最初の一手になります。

**コードで確かめる（型の安定性）：**

```cpp
// ① 型を合意・固定する
//    シンプルだが、型が変われば全インターフェースのシグネチャを変更しなければならない
class IUserService {
public:
    virtual void process(int userId) = 0;
};

// ② 独自型でくるむ
//    インターフェースのシグネチャは変わらない。型の変更は UserId の中だけに留まる
struct UserId {
    std::string value;  // int → string に変わってもここだけ直す
};
class IUserService {
public:
    virtual void process(UserId id) = 0;
};

// ③ void* で型情報をインターフェースに持たせない
//    インターフェースは絶対に変わらない。代わりに型安全を失う
class IUserService {
public:
    virtual void process(void* context) = 0;  // 呼び出し側でキャストが必要
};
```

| 選択肢 | インターフェース変更 | 型安全 | コードの複雑さ |
|---|---|---|---|
| ①合意・固定 | 型が変われば変わる | ✅ 高い | シンプル |
| ②独自型でくるむ | **変わらない** | ✅ 高い | 型定義が1つ増える |
| ③void\* | **絶対変わらない** | ❌ 低い | キャストが必要 |

どれが正解かは状況次第です。各章では、パターンが直面したこの問題と、関係者との確認を経て選んだ判断を示します。

#### すべてにインターフェースを付けるべきか

「実装ではなくインターフェースに」と聞くと、
全クラスにインターフェースを付けたくなるかもしれません。
私自身、一時期そうしていました。

答えは「No」です。インターフェースが必要なのは、
**呼び出し元を変更から守りたい境界**だけです。
変わらないもの、1箇所からしか使わないもの、
チーム内で完全にコントロールできるものには不要です。

```cpp
// インターフェースが不要な例
// TaxCalculator は経理と合意済みの固定ルール
// 使う場所も1箇所だけ
class TaxCalculator {
public:
    double calc(double amount) {
        return amount * 1.1;
    }
};

// 直接使って何の問題もない
class BillingService {
    TaxCalculator tax_;
public:
    double calculate(double amount) {
        return tax_.calc(amount);
    }
};
```

| 状況 | インターフェース | 理由 |
|---|---|---|
| 外部サービスの仕様が変わる | 必要 | 変更から呼び出し元を守る |
| 複数の実装を切り替えたい | 必要 | 差し替えの仕組みが要る |
| テストで差し替えたい | 必要 | スタブに置き換える |
| 固定ルールで変わらない | 不要 | 守る必要がない |
| 1箇所からしか使わない | 検討 | 変更頻度による |

### 技法2：繋ぐとき、継承よりコンポジションを優先する

機能を「is-a（〜は〜である）」ではなく「has-a（〜を部品として持つ）」で組み合わせる。

**「優先せよ」は「常にコンポジションを使え」という意味ではありません。**
迷ったらコンポジション。継承を選ぶなら、明確な理由があるときだけ。
これが「優先」の意味です。

なぜコンポジションが優先なのか。
継承は親クラスの実装に依存するため、親が変わると子も変わります。
コンポジションは部品をインターフェース経由で持つため、
部品の実装が変わっても持ち主は変わりません。
哲学1（変わるものをカプセル化せよ）と
哲学2（インターフェースに対してプログラムせよ）を
自然に実現できるのがコンポジションです。

**コードで確かめる（継承の脆さ）：**

```cpp
// NG：継承（is-a）で機能を拡張する
//     LoggingNotifier は Notifier の実装に依存
class LoggingNotifier : public Notifier {
    void notify(std::string msg) override {
        log(msg);
        Notifier::notify(msg); // 親の実装に依存
    }
};
// → Notifier::notify() の中身が変わると
//    LoggingNotifier の動作も変わってしまう

// OK：コンポジション（has-a）で組み合わせる
//     inner_ を差し替えるだけで通知先を変えられる
class LoggingNotifier {
    IMessageSender* inner_;
public:
    explicit LoggingNotifier(IMessageSender* inner)
        : inner_(inner) {}
    void notify(std::string msg) {
        log(msg);
        inner_->send(msg); // 差し替え可能
    }
};
```

#### では、継承はいつ使うのか

「コンポジション優先」と聞くと、
「継承は使ってはいけないのか」と不安になるかもしれません。
私も同じ不安を感じました。

答えは明確です。**継承が適している場面は確かにあります。**

**継承が適している場面①：
is-a関係が概念として成立するとき**

「犬は動物である（Dog is-a Animal）」のように、
子クラスが親クラスの**すべての契約を完全に引き継げる**なら、
継承は自然な選択です。

判断の問い：
「親クラスが使われているすべての場所で、
子クラスに置き換えても正しく動くか？」

```cpp
// 良い継承の例：Shape と Circle
// Circle は Shape のすべての契約を引き継げる
class Shape {
public:
    virtual double area() = 0;
    virtual ~Shape() {}
};

class Circle : public Shape {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}
    double area() override {
        return 3.14159 * radius_ * radius_;
    }
};

class Rectangle : public Shape {
    double w_, h_;
public:
    Rectangle(double w, double h)
        : w_(w), h_(h) {}
    double area() override { return w_ * h_; }
};
// → Shape が使われている場所に
//    Circle を渡しても Rectangle を渡しても
//    正しく動く
```

**継承が適している場面②：
親クラスが「拡張される前提」で設計されているとき**

親クラスが「ここだけ子クラスで変えてよい」と
拡張点を明示しているなら、継承は安全です。

```cpp
// 親クラスが拡張点を明示している例
class DataExporter {
public:
    void exportData() {
        prepareHeader();   // 固定
        writeBody();       // ← ここだけ変えてよい
        finalizeFooter();  // 固定
    }
protected:
    virtual void writeBody() = 0;
private:
    void prepareHeader()  { /* 固定処理 */ }
    void finalizeFooter() { /* 固定処理 */ }
};

class CsvExporter : public DataExporter {
protected:
    void writeBody() override {
        // CSV形式で出力
    }
};
// → 親が「変えてよい場所」を明示しているから安全
```

#### 継承が壊れる場面

is-a関係ではなく、
「既存の機能を再利用したい」だけで継承すると問題が起きます。

```cpp
// 悪い継承の例：機能を足すために継承する
class LoggingNotifier : public Notifier {
    void notify(std::string msg) override {
        log(msg);
        Notifier::notify(msg);
    }
};

class EncryptedNotifier : public Notifier {
    void notify(std::string msg) override {
        msg = encrypt(msg);
        Notifier::notify(msg);
    }
};

// では「ログ付き暗号化通知」は？
// → LoggingEncryptedNotifier を作る？
// → 組み合わせが増えるたびにクラスが爆発する
```

コンポジションなら、組み合わせは組み立て時に自由に決められます。

```cpp
// コンポジションなら組み合わせは自由
class LoggingNotifier : public IMessageSender {
    IMessageSender* inner_;
public:
    explicit LoggingNotifier(IMessageSender* s)
        : inner_(s) {}
    void send(std::string msg) override {
        log(msg);
        inner_->send(msg);
    }
};

class EncryptedNotifier : public IMessageSender {
    IMessageSender* inner_;
public:
    explicit EncryptedNotifier(IMessageSender* s)
        : inner_(s) {}
    void send(std::string msg) override {
        inner_->send(encrypt(msg));
    }
};

// 組み合わせは組み立て時に決める
EmailSender email;
EncryptedNotifier encrypted(&email);
LoggingNotifier logged(&encrypted);
logged.send("Hello");
// → ログ → 暗号化 → Email送信
// 新しいクラスを作る必要はない
```

#### 継承とコンポジションの判断基準

迷ったときは、以下の問いで判断できます。

| 問い | Yes → 継承 | No → コンポジション |
|---|---|---|
| 子は親の**全契約**を引き継げるか？ | ✅ | ❌ |
| 親は**拡張前提**で設計されているか？ | ✅ | ❌ |
| 部品の**組み合わせ**が将来増えるか？ | ❌ | ✅ |

3つの問い全てに Yes なら継承、
1つでも No ならコンポジションが安全です。

「迷ったらコンポジション」——この原則で間違えることは、
ほとんどありません。
継承を選ぶのは、上の問いに明確に Yes と答えられるときだけです。

## 先人たちが使っていた「8ステップの思考プロセス」

先人たちも、私たちと同じように混沌としたコードの前で頭を抱えていました。
彼らが問題を解決するとき、意識的かどうかはともかく、
以下の8つのステップを踏んでいたと解釈することができます。

> **これはレシピではなく、地図です。**
> 現場の思考は直線的には進みません。ステップ5で対策を考えている最中に
> ステップ2に戻ることもあります。各ステップは「思考を整理するための道具」であり、
> 「この順番で実行せよ」という手順書ではありません。

ただし、**初めて使うときは順番に進めることをお勧めします。**
各ステップの出力が、次のステップの入力になっています。
一通り体験した後は、必要なステップに自由に戻ってください。

### ステップ0：現状の共有と視点のチューニング

コードを読む前に、「何のシステムか」を共有してから、「設計のレンズ」をセットする。

**順序が重要です。**
「このコードの中に〜」という問いを立てるには、まず「このコード」を知っていなければなりません。
だから各章は必ず、システムの仕様・クラス構造・起点コードを先に示してから、問いをセットします。

> 「このコードの中に、『変わる理由』が異なる2つのものが、同じ場所に混在していないか？」

このレンズがセットされた状態で初めて、コードが「動いている処理の羅列」ではなく「責任の配置図」として見え始めます。

**飛ばすとどうなるか：** コードの理解が曖昧なまま「何となくここが問題っぽい」で進んでしまう。対策を立てた後で「そもそもシステムの前提が違っていた」と手戻りになります。

### ステップ1：現状把握 ―― 責任外の関心を探す

セットした問いを使って「各クラスの責任」を定義し、「責任範囲外の知識を持っていないか」を確認する。

コードを「読む」のではなく「観察する」感覚です。
「このクラスの責任は何か」を1文で定義したうえで、コードの各行が本当にその責任の中にあるかを問います。
依存関係を図（依存グラフ）で可視化し、問題の広がりをマクロで確認します。

**なぜステップ0の後なのか：** システムの全体像（ステップ0）がなければ、「このクラスの責任は何か」を定義できません。

**飛ばすとどうなるか：** 「何が問題か」を特定せずに対策に飛びつく。結果として「問題でない場所」にパターンを適用し、無駄な複雑さを生みます。

### ステップ2：変動と不変の峻別

ステップ1で把握した現状のコードをもとに、「変わりそうなもの」と「変わらないもの」の仮説を立て、ヒアリングで確定する。

実際のコードを観察して仮説を立て、さらに関係者と照合した上で、「変わる」「変わらない」を表として明示します。
この表が、以降の設計判断の根拠になります。

**なぜステップ1の後なのか：** 責任チェック（ステップ1）で「どこに問題があるか」を把握してから、「その問題の中で何が変わり、何が変わらないか」を仕分けます。問題が特定されていなければ、何を仕分けるかも分かりません。

**飛ばすとどうなるか：** 「変わるかもしれない」という推測だけで設計を進める。ヒアリングなしの仕分けは、「変わらないものをカプセル化してしまう」過剰設計の原因になります。

### ステップ3：課題分析

「変更要求を受けて、何が難しいのか」を具体化する。

変更要求を頭の中で試しに実装してみます。
「あ、これをやろうとするとここが壊れる」
「この引数がまた増える……」という
**詰まりの感覚**を言葉にするステップです。

**なぜステップ2の後なのか：** 「何が変わるか」（ステップ2）が分かったから、「その変更を今の構造に適用したとき何が起きるか」をシミュレートできます。

**飛ばすとどうなるか：** 「困っている」という感覚はあるのに、何が困難なのか言葉にできない。チームに説明できないまま、リファクタリングの提案が通らなくなります。

### ステップ4：原因分析

「なぜ難しいのか」の根本を突き止める。

詰まりの感覚には、必ず構造的な原因があります。
多くの場合、それは以下のいずれかです：

- 変わり続けるものと変わってほしくないものが同じ場所にいる
- 使う側が知らなくていいことまで知っている
- 状態と振る舞いの管理が1か所に集まっている
- 生成の責任と利用の責任が混在している

原因が特定できると、「ここを変えれば楽になる」という
仮説が立てられます。

**なぜステップ3の後なのか：** 困難の症状（ステップ3）を具体的に知っているから、「なぜその症状が起きるのか」を根本まで掘れます。症状を飛ばして原因を探ると、推測に基づく分析になります。

**飛ばすとどうなるか：** 原因を特定せずに対策を考える。「なぜか分からないけど、このパターンが使えそう」——これが「パターンを覚えても使えない」の正体です。

### ステップ5：対策案の検討

「どう解決するか」を、「理想の契約」から逆算して構造を作る。

解決の順序は常に同じです。インターフェース（契約）を先に定義し、
実装は後から充てる。最初に思いつく素朴な案から始め、その限界に気づき、
発想を転換し、より良い構造へたどり着くプロセスが、このステップの核心です。

**なぜステップ4の後なのか：** 根本原因（ステップ4）が分かっているから、「その原因を取り除く構造」を設計できます。原因が曖昧なまま対策を考えると、症状を隠すだけの設計になります。

**飛ばすとどうなるか：** このステップを飛ばすことはありませんが、「素朴な案を試さずにいきなり完成形を目指す」と、なぜその構造が良いのかを説明できなくなります。

### ステップ6：天秤にかける

「この柔軟性は、将来発生する変更コストに見合うものか」を判断する。

評価軸を先に宣言してから比較します（後から基準を決めると「結論ありきの比較」になります）。
テストは「設計の質を測るセンサー」として使います。
テストが書きにくい案は、まだ原因が解消されていないサインです。

**なぜステップ5の後なのか：** 対策案（ステップ5）が複数あるから、比較できます。案が1つしかなければ天秤は不要ですが、それは対策案の検討が足りないサインかもしれません。

**飛ばすとどうなるか：** 「パターンを適用した方が良い」という結論ありきで進む。過剰設計の温床になります。パターンを使わない方が良い場面を見落とします。

### ステップ7：決断と、手に入れた未来

「費用対効果」を考えて決断し、解決後の構造を確認する。

変化が今後も続くなら投資の価値があります。
変化が止まるなら、シンプルなままにしておく方が合うかもしれません。
**設計に絶対の正解はありません。**

決断したら、変更前に感じていた「設計のツッコミ」が消えたかを確認します。
消えていなければ、原因がまだ残っています。

**なぜ最後なのか：** ステップ0〜6の全ての分析と比較があって初めて、根拠のある決断ができます。「直感」ではなく「積み上げた根拠」で選ぶからこそ、チームに説明できる設計になります。

## このプロセスの結果として生まれたものが「パターン」

先人たちは、このプロセスをさまざまな現場の問題に適用しました。
そこで繰り返し現れた「構造の解法」に、後の人たちが名前をつけたものが
GoFのデザインパターンです。

| 先人たちが向き合った問題 | 同じ場所にいた「2つのもの」 | 結果のラベル |
|:---|:---|:---|
| 処理の手段が増えるたびに制御処理が膨らんでいく | 「何をするかという制御」× 「どうやるかという手段」 | Strategy |
| 外部APIの仕様変更が、呼び出し元のコード全体に波及する | 「呼び出し元の要求」× 「外部サービスの詳細」 | Facade |
| 状態によって同じ操作が別の振る舞いをする | 「操作のインターフェース」× 「状態ごとの振る舞い」 | State |
| 処理の一部だけが変わるたびに、処理全体を複製している | 「処理の骨格（変わらない）」× 「ステップの実装（変わる）」 | Template Method |
| 実行する操作の種類が増えるたびに、実行する側のコードが膨らむ | 「処理を実行する側」× 「何の処理をするかという操作の種類」 | Command |
| 機能の組み合わせが増えるたびに、クラスが爆発的に増える | 「コアの機能」× 「後から追加したい機能の組み合わせ」 | Decorator |
| 通知先が増えるたびに通知元のコードを書き換えなければならない | 「自分の状態管理」× 「誰にどう通知するか」 | Observer |
| 使うオブジェクトの種類が変わるたびに、利用側のコードも変わる | 「オブジェクトを使う側」× 「どのクラスをどう生成するか」 | Factory Method |

8つすべてに共通するのは、「変わる理由の異なる2つのものが同じ場所にいた」という構造です。

**パターンは最初から目指す形ではなく、このプロセスを踏んだ結果のラベルです。**

### なぜこの8パターンなのか

GoFのデザインパターンは23種類ありますが、この本では8つに絞っています。
「残りの15パターンは学ばなくていいのか」と思うかもしれません。

この8パターンを選んだ理由は3つあります。

**理由1：現場での遭遇頻度が高い。**
この8パターンは、現実のプロジェクトで最も頻繁に直面する問題に対応しています。

**理由2：4種類の原因パターンをカバーしている。**
設計の問題は、突き詰めると4種類の原因に分類できます。
この8パターンで4種類すべてを体験できます。

| 原因の種類 | 扱うパターン |
|---|---|
| 変化の混在 | Strategy / Template Method / Command / Decorator / Observer |
| 依存の過多 | Facade |
| 状態と振る舞いの混在 | State |
| 生成と利用の混在 | Factory Method |

**理由3：この8つで考え方を身につければ、残りのパターンも自分で導ける。**
これが最も重要な理由です。

### 残りのGoFパターンも同じ哲学で説明できる

実は、GoFの23パターンのほぼすべてが、
「変わる理由の異なる2つのものが同じ場所にいた」
という構造で説明できます。

本書で扱わないパターンについても、「同じ場所にいた2つのもの」を示します。

| 向き合った問題 | 同じ場所にいた「2つのもの」 | 結果のラベル |
|:---|:---|:---|
| 既存クラスのインターフェースが合わない | 「呼び出し元の期待」×「既存クラスの実際の形」 | Adapter |
| 抽象と実装が密結合している | 「抽象の階層」×「実装の階層」 | Bridge |
| 個と集合で操作を統一したい | 「個の操作」×「集合の操作」 | Composite |
| アクセス制御と実処理が混在 | 「アクセスの制御」×「実際の処理」 | Proxy |
| 製品群全体の切り替えが必要 | 「製品群を使う側」×「どの製品群を作るか」 | Abstract Factory |
| 構築手順と表現が混在 | 「構築の手順」×「構築の表現」 | Builder |
| 要素のアクセス方法と内部構造が混在 | 「順次アクセスの方法」×「コレクションの内部構造」 | Iterator |

すべてのパターンに共通しているのは、同じ哲学です。
「変わるものと変わらないものを見分け、分ける」——
この思考プロセスを8つの章で体験すれば、
残りのパターンに出会ったときも「あの問いで考えてみよう」と動けるはずです。

> **補足：Singletonについて**
> GoF 23パターンの中で、Singletonだけは少し性質が異なります。
> 「変わるものと変わらないものの分離」ではなく、
> 「インスタンスのライフサイクル管理」が主な関心です。
> 設計原則とは異なる次元の話なので、この本では扱いません。

## この本の読み方

### 第一部：各パターンの考え方を学ぶ（第1章〜第8章）

第1章以降、各章は完全に独立しています。
興味のあるパターンから読み始めてください。

各章では、毎回同じ8ステップ（ステップ0〜7）の思考プロセスを体験します。
章の違いは「今回、何と何が混在しているか」だけです。

### 各章で出会う「変化の種類」と原因の切り口

| 章 | パターン | 今回の変化の種類 | 原因の切り口 |
|:---|:---|:---|:---|
| 第1章 | Strategy | 実行する振る舞いが変わる | 変化の混在 |
| 第2章 | Facade | 複雑な依存関係を隠したい | 依存の過多 |
| 第3章 | State | 状態によって振る舞いが変わる | 状態と振る舞いの混在 |
| 第4章 | Template Method | 処理の一部のステップが変わる | 変化の混在 |
| 第5章 | Command | 実行する操作が変わる | 変化の混在 |
| 第6章 | Decorator | 機能の組み合わせが変わる | 変化の混在 |
| 第7章 | Observer | 通知を受け取る相手が変わる | 変化の混在 |
| 第8章 | Factory Method | 作るオブジェクトが変わる | 生成と利用の混在 |

「変化の混在」が多いのは、現実の設計問題の多くが原則
（変わるものをカプセル化せよ）を見落としたことで生じるからです。

### 第二部：応用編 ―― 複数パターンの融合（第9章〜第11章）

現実のシステムでは、「変わる理由」が1つだけとは限りません。
1つのシステムに複数の「変わる理由」が混在していることの方がむしろ普通です。

第二部では、第一部で身につけた思考プロセスを、より複雑なコードに適用します。
同じ8ステップで分析を進めると、ステップ4（原因分析）で**複数の根本原因**が見つかり、
ステップ5（対策案）で**複数のパターンが自然に組み合わさる**体験をします。

| 章 | 題材 | 扱う問題 |
|:---|:---|:---|
| 第9章 | ECサイトの注文処理 | 割引ルールと注文ステータスが同時に変わる |
| 第10章 | 外部連携バッチシステム | 外部API＋通知先＋生成手段が絡む |
| 第11章 | レポート生成エンジン | 処理の骨格＋装飾＋操作履歴が絡む |

名前を覚えることが目的ではありません。
**「次の現場で同じ問題に出会ったとき、あの問いで考えてみよう」**
と思えることが、この本を読む意味です。

状況が変わればまた考え直せばいい、という軽さで向き合ってほしいと思います。
一つの参考として受け取っていただければと思います。*
