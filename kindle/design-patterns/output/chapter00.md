# 第0章　この本の読み方
―― デザインパターンは「考えた結果」に過ぎない

---

## なぜ「パターンを覚えても使えない」のか

ソフトウェア設計を学ぼうとすると、必ずと言っていいほど
「GoFのデザインパターン」に出会います。
本で学び、構造図を頭に入れ、いざ自分のコードに使おうとしたとき——

「どこに適用すればいいのか、わからない。」
「無理に使ってみたら、かえってコードが複雑になった。」

私自身、同じ壁に何度もぶつかりました。
パターンの名前と図は頭に入った。でも、
目の前の問題にどう当てはめればいいのか、
まるでわからなかったのです。

その感覚、うまく伝わっているでしょうか。

なぜ、実績のある優れた設計手法が、
時にはコードをより複雑にしてしまうのか。

理由はシンプルです。
**パターンを「最初から目指すべき答え」として扱っているから**です。

デザインパターンは、先人たちが泥臭い現場で問題に向き合い、
いくつかの選択肢を天秤にかけ、
「この状況ではこれが一番割に合う」と判断した
**決断の結果**として生まれたものです。

結果だけを真似ても、状況が違えばうまく機能しません。
大切なのは、その結果に至るまでの**思考のプロセス**を体験することです。

## すべてのパターンを貫く3つの哲学

GoFの23のデザインパターンは、一見バラバラに見えます。
でも、すべてのパターンは、たった3つの哲学を
それぞれの状況に具体化したものに過ぎない——と、私は整理しています。

これを先に知っておくと、パターンが「暗記する公式の集まり」から
「同じ哲学を別の形で表現したもの」に見え始めます。

### 哲学1：変わるものをカプセル化せよ

「変わりやすい部分」と「変わってほしくない部分」を、同じ場所に書かない。

GoF 23パターンのほぼすべては、この1つの哲学から導けます。
Strategyは「アルゴリズムの実装」を隔離し、
Observerは「通知先の種類」を隔離し、
Facadeは「外部サービスの詳細」を隔離します。
やっていることは全部同じで、「変わりやすいものを取り出して別の場所に閉じ込める」だけです。

**この哲学を使うための問い——この本を通じて使い回せる1つの問い：**

> 「このコードの中に、**『変わる理由』が異なる2つのものが、同じ場所に混在していないか？**」

パターンが違っても、問いはこれひとつです。
「何と何が混在しているか」の組み合わせが違うだけで、
本質的に問いかけていることは変わりません。

**コードで確かめる：**

```cpp
// NG：計算ロジックと出力形式が同じクラスに混在している
//      計算が変わっても、出力形式が変わっても、このクラスを変える羽目になる
class ReportService {
    void generate() {
        double value = calcAmount();      // 計算ロジックが変わる理由
        std::string text = format(value); // 出力形式が変わる理由
        writeToPdf(text);
    }
};

// OK：変わる理由ごとに分離する
//     計算が変わっても ReportFormatter は変わらない。出力が変わっても Calculator は変わらない
class Calculator    { virtual double calcAmount() = 0; };
class ReportFormatter { virtual std::string format(double v) = 0; };

class ReportService {
    Calculator*     calc_;
    ReportFormatter* fmt_;
public:
    void generate() { fmt_->format(calc_->calcAmount()); }
};
```

### 哲学2：実装ではなくインターフェースに対してプログラムせよ

「何をするか（契約）」と「どうやるか（実装）」を分ける。

呼び出す側が `EmailNotifier` という具体クラスを知っていたら、
Emailの送り方が変わるたびに呼び出す側も変わります。
でも `INotifier` というインターフェースだけを知っていれば、
Email→SMS→Pushに切り替わっても呼び出す側はまったく変わりません。

この哲学は「変わる理由の異なるものを分離した後、その境界を何でつなぐか」の答えです。
インターフェースが、安定した呼び出し側と不安定な実装側の間に立つ「緩衝材」になります。

**コードで確かめる（依存の方向）：**

```cpp
// NG：具体クラスに直接依存している
//     EmailSender が変わるたびに Notifier も変わる
class Notifier {
    EmailSender* sender_; // 具体クラスを知っている
public:
    void notify(std::string msg) { sender_->send(msg); }
};

// OK：インターフェースに依存する
//     IMessageSender の実装が Email→SMS→Push に変わっても Notifier は変わらない
class IMessageSender {
public:
    virtual void send(std::string msg) = 0;
    virtual ~IMessageSender() {}
};

class Notifier {
    IMessageSender* sender_; // 契約（インターフェース）だけを知っている
public:
    void notify(std::string msg) { sender_->send(msg); }
};
```

ただし、インターフェースが守れないものがひとつあります。**引数の型が変わるとき**です。

同じ引数の型（例：`int userId`）を複数のインターフェースで共有しているとき、
「`int` を `string` に変えてほしい」という要求が来ると、
そのすべてのインターフェースのシグネチャが変わります。
どんな設計構造があっても、この変更からは逃げられません。

こういう状況に直面したとき、立てるべき問いは「なぜこのパターンでは守れないのか」ではなく、
**「この型はどこまで安定していると言えるか？チームで合意できているか？」** です。
型を決める前に、システムの所有者や連携する部署の担当者に確認することが、最初の一手になります。

**コードで確かめる（型の安定性）：**

```cpp
// ① 型を合意・固定する
//    シンプルだが、型が変われば全インターフェースのシグネチャを変更しなければならない
class IUserService {
public:
    virtual void process(int userId) = 0;
};

// ② 独自型でくるむ
//    インターフェースのシグネチャは変わらない。型の変更は UserId の中だけに留まる
struct UserId {
    std::string value;  // int → string に変わってもここだけ直す
};
class IUserService {
public:
    virtual void process(UserId id) = 0;
};

// ③ void* で型情報をインターフェースに持たせない
//    インターフェースは絶対に変わらない。代わりに型安全を失う
class IUserService {
public:
    virtual void process(void* context) = 0;  // 呼び出し側でキャストが必要
};
```

| 選択肢 | インターフェース変更 | 型安全 | コードの複雑さ |
|---|---|---|---|
| ①合意・固定 | 型が変われば変わる | ✅ 高い | シンプル |
| ②独自型でくるむ | **変わらない** | ✅ 高い | 型定義が1つ増える |
| ③void\* | **絶対変わらない** | ❌ 低い | キャストが必要 |

どれが正解かは状況次第です。各章では、パターンが直面したこの問題と、関係者との確認を経て選んだ判断を示します。

### 哲学3：継承よりコンポジションを優先せよ

機能を「is-a（〜は〜である）」ではなく「has-a（〜を部品として持つ）」で組み合わせる。

継承は一見便利ですが、親クラスが変わると子クラスも変わる依存関係を生みます。
コンポジションは部品を差し替えるだけで振る舞いを変えられます。
DecoratorやStrategyが「インターフェースを実装したオブジェクトを内部に持つ」
という構造をしているのは、この哲学に従っているからです。

**コードで確かめる：**

```cpp
// NG：継承（is-a）で機能を拡張する
//     LoggingNotifier は Notifier に依存している
//     Notifier の実装が変わると LoggingNotifier も影響を受ける
class LoggingNotifier : public Notifier {
    void notify(std::string msg) override {
        log(msg);
        Notifier::notify(msg); // 親の実装に依存
    }
};

// OK：コンポジション（has-a）で振る舞いを組み合わせる
//     inner_ を差し替えるだけで、ログ付き通知の相手を自由に変えられる
class LoggingNotifier {
    IMessageSender* inner_; // インターフェースとして持つ
public:
    explicit LoggingNotifier(IMessageSender* inner) : inner_(inner) {}
    void notify(std::string msg) {
        log(msg);
        inner_->send(msg); // 差し替え可能
    }
};
```

## 先人たちが使っていた「8ステップの思考プロセス」

先人たちも、私たちと同じように混沌としたコードの前で頭を抱えていました。
彼らが問題を解決するとき、意識的かどうかはともかく、
以下の8つのステップを踏んでいたと解釈することができます。

### ステップ0：現状の共有と視点のチューニング

コードを読む前に、「何のシステムか」を共有してから、「設計のレンズ」をセットする。

**順序が重要です。**
「このコードの中に〜」という問いを立てるには、まず「このコード」を知っていなければなりません。
だから各章は必ず、システムの仕様・クラス構造・起点コードを先に示してから、問いをセットします。

> 「このコードの中に、『変わる理由』が異なる2つのものが、同じ場所に混在していないか？」

このレンズがセットされた状態で初めて、コードが「動いている処理の羅列」ではなく「責任の配置図」として見え始めます。

### ステップ1：現状把握 ―― 責任外の関心を探す

セットした問いを使って「各クラスの責任」を定義し、「責任範囲外の知識を持っていないか」を確認する。

コードを「読む」のではなく「観察する」感覚です。
「このクラスの責任は何か」を1文で定義したうえで、コードの各行が本当にその責任の中にあるかを問います。
依存関係を図（依存グラフ）で可視化し、問題の広がりをマクロで確認します。

### ステップ2：変動と不変の峻別

ステップ1で把握した現状のコードをもとに、「変わりそうなもの」と「変わらないもの」の仮説を立て、ヒアリングで確定する。

実際のコードを観察して仮説を立て、さらに関係者と照合した上で、「変わる」「変わらない」を表として明示します。
この表が、以降の設計判断の根拠になります。

### ステップ3：課題分析

「変更要求を受けて、何が難しいのか」を具体化する。

変更要求を頭の中で試しに実装してみます。
「あ、これをやろうとするとここが壊れる」
「この引数がまた増える……」という
**詰まりの感覚**を言葉にするステップです。

### ステップ4：原因分析

「なぜ難しいのか」の根本を突き止める。

詰まりの感覚には、必ず構造的な原因があります。
多くの場合、それは以下のいずれかです：

- 変わり続けるものと変わってほしくないものが同じ場所にいる
- 使う側が知らなくていいことまで知っている
- 状態と振る舞いの管理が1か所に集まっている
- 生成の責任と利用の責任が混在している

原因が特定できると、「ここを変えれば楽になる」という
仮説が立てられます。

### ステップ5：対策案の検討

「どう解決するか」を、「理想の契約」から逆算して構造を作る。

解決の順序は常に同じです。インターフェース（契約）を先に定義し、
実装は後から充てる。最初に思いつく素朴な案から始め、その限界に気づき、
発想を転換し、より良い構造へたどり着くプロセスが、このステップの核心です。

### ステップ6：天秤にかける

「この柔軟性は、将来発生する変更コストに見合うものか」を判断する。

評価軸を先に宣言してから比較します（後から基準を決めると「結論ありきの比較」になります）。
テストは「設計の質を測るセンサー」として使います。
テストが書きにくい案は、まだ原因が解消されていないサインです。

### ステップ7：決断と、手に入れた未来

「費用対効果」を考えて決断し、解決後の構造を確認する。

変化が今後も続くなら投資の価値があります。
変化が止まるなら、シンプルなままにしておく方が合うかもしれません。
**設計に絶対の正解はありません。**

決断したら、変更前に感じていた「設計のツッコミ」が消えたかを確認します。
消えていなければ、原因がまだ残っています。

## このプロセスの結果として生まれたものが「パターン」

先人たちは、このプロセスをさまざまな現場の問題に適用しました。
そこで繰り返し現れた「構造の解法」に、後の人たちが名前をつけたものが
GoFのデザインパターンです。

| 先人たちが向き合った問題 | 同じ場所にいた「2つのもの」 | 結果のラベル |
|:---|:---|:---|
| 処理の手段が増えるたびに制御処理が膨らんでいく | 「何をするかという制御」× 「どうやるかという手段」 | Strategy |
| 外部APIの仕様変更が、呼び出し元のコード全体に波及する | 「呼び出し元の要求」× 「外部サービスの詳細」 | Facade |
| 状態によって同じ操作が別の振る舞いをする | 「操作のインターフェース」× 「状態ごとの振る舞い」 | State |
| 処理の一部だけが変わるたびに、処理全体を複製している | 「処理の骨格（変わらない）」× 「ステップの実装（変わる）」 | Template Method |
| 実行する操作の種類が増えるたびに、実行する側のコードが膨らむ | 「処理を実行する側」× 「何の処理をするかという操作の種類」 | Command |
| 機能の組み合わせが増えるたびに、クラスが爆発的に増える | 「コアの機能」× 「後から追加したい機能の組み合わせ」 | Decorator |
| 通知先が増えるたびに通知元のコードを書き換えなければならない | 「自分の状態管理」× 「誰にどう通知するか」 | Observer |
| 使うオブジェクトの種類が変わるたびに、利用側のコードも変わる | 「オブジェクトを使う側」× 「どのクラスをどう生成するか」 | Factory Method |

8つすべてに共通するのは、「変わる理由の異なる2つのものが同じ場所にいた」という構造です。
各章では、それぞれの問題にこの7ステップのプロセスを適用していきます。

**パターンは最初から目指す形ではなく、このプロセスを踏んだ結果のラベルです。**

## この本の読み方

### どの章から読んでも大丈夫です

第1章以降、各章は完全に独立しています。
興味のあるパターンから読み始めてください。

各章では、毎回同じ7ステップの思考プロセスを体験します。
章の違いは「今回、何と何が混在しているか」だけです。

### 各章で出会う「変化の種類」と原因の切り口

| 章 | パターン | 今回の変化の種類 | 原因の切り口 |
|:---|:---|:---|:---|
| 第1章 | Strategy | 実行する振る舞いが変わる | 変化の混在 |
| 第2章 | Facade | 複雑な依存関係を隠したい | 依存の過多 |
| 第3章 | State | 状態によって振る舞いが変わる | 状態と振る舞いの混在 |
| 第4章 | Template Method | 処理の一部のステップが変わる | 変化の混在 |
| 第5章 | Command | 実行する操作が変わる | 変化の混在 |
| 第6章 | Decorator | 機能の組み合わせが変わる | 変化の混在 |
| 第7章 | Observer | 通知を受け取る相手が変わる | 変化の混在 |
| 第8章 | Factory Method | 作るオブジェクトが変わる | 生成と利用の混在 |

「変化の混在」が多いのは、現実の設計問題の多くが哲学1
（変わるものをカプセル化せよ）を見落としたことで生じるからです。

名前を覚えることが目的ではありません。
**「次の現場で同じ問題に出会ったとき、あの問いで考えてみよう」**
と思えることが、この本を読む意味です。

---

*間違えても大丈夫です。設計は一度決めたら終わりではなく、
状況が変わればまた考え直せばいい、という軽さで向き合ってほしいと思います。
一つの参考として受け取っていただければと思います。*
