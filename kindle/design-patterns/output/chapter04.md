# 第4章 Template Methodパターン ―― 処理の骨格を固定し、中身をサブクラスに委ねる

## ステップ0：システムを把握し、仮説を立てる

**この章で扱うシステム：レポート生成バッチ**

毎日・毎週・毎月の3種類の定期レポートを生成するシステムです。各レポートは「データ収集 → データ集計 → フォーマット適用 → 出力」という共通の流れを持ちますが、具体的な集計ロジックや出力先が異なります。

**変動と不変の仮説**

| 分類 | 仮説 | 根拠 |
|---|---|---|
| 🔴 **変動する** | 集計ロジック、出力先 | レポート種別ごとに要件が異なるため |
| 🟢 **不変** | 処理の全体的な順序（骨格） | 全てのレポートバッチが守るべき標準手順であるため |

---

## ステップ1：実装コードを読む ―― 責任の混在を見つける

### 4.1 現状のコード

```cpp
class DailyReportBatch {
public:
    void run() {
        std::cout << "[LOG] Started" << std::endl;
        std::cout << "Collecting data..." << std::endl;
        
        // 独自ロジック
        std::string data = "Daily Data";
        std::cout << "Output to CSV: " << data << std::endl;
        
        std::cout << "[LOG] Finished" << std::endl;
    }
};

class WeeklyReportBatch {
public:
    void run() {
        std::cout << "[LOG] Started" << std::endl;
        std::cout << "Collecting data..." << std::endl;
        
        // 独自ロジック
        std::string data = "Weekly Data";
        std::cout << "Sending Email: " << data << std::endl;
        
        std::cout << "[LOG] Finished" << std::endl;
    }
};
```

**問題の発見：**
各クラスの `run()` メソッドの冒頭と末尾に、ログ出力やデータ収集といった「共通の骨格」がベタ書きされています。

---

## ステップ2：仮説を確定する ―― 関係者ヒアリング

**関係者の声：**
- **経営企画：** 「レポートの基本手順は今後も変わりません。」
- **インフラ：** 「将来、全レポートの開始・終了時に監査ログを出力する制約を加えたいです。」

---

## ステップ3：課題分析 ―― 変更が来たとき、どこが辛いか

新しい「四半期レポート」を追加しようとすると、既存の「ログ出力」や「データ収集」の呼び出しコードを丸ごとコピペする必要があります。もし呼び出し順を間違えると、監査ログが出ないといったバグが発生します。

---

## ステップ4：原因分析 ―― 困難の根本にある問題を言語化する

**真因：** 「処理の流れ（骨格）」と「個別の中身」が同じ場所にあること。

---

## ステップ5：対策案の検討 ―― 方向性を決め、手段を順に試す

### 4.5.1 方向性の特定

原因が「処理の骨格と中身が同居している」なら、解消する方向性は一つです。

**→ 「変わらない骨格」を固定し、「変わる中身」だけを切り出して委ねる。**

---

### 4.5.2 手段①：共通処理のユーティリティ関数化（直感的な案）

重複する処理を `BatchUtils` クラスにまとめ、各バッチから呼び出す方法です。

```cpp
void DailyReportBatch::run() {
    BatchUtils::startLog();
    BatchUtils::collect();
    // 独自処理...
    BatchUtils::endLog();
}
```

**残る課題：**
「正しい順序で呼び出す責任」が依然として各バッチクラスに残っています。開発者が `startLog()` を呼び忘れるリスクを防げません。

---

### 4.5.3 手段②：骨格を親クラスに固定する（発想の転換）

第0章の手札：「共通の骨格に対する部分的なステップが変わる」→ **骨格・ステップ分離**。

**「親クラスに全体の流れを固定し、変わる部分だけを子クラスに埋めさせる。」**

このように、アルゴリズムの骨格を定義し、一部の手順をサブクラスに委ねる構造を **Template Method（テンプレートメソッド）パターン** と呼びます。

```cpp
class ReportBatchTemplate {
public:
    void run() { // 骨格を固定（不変）
        startLog();
        collect();
        process(); // ここだけ子クラスに委ねる
        endLog();
    }
protected:
    virtual void process() = 0; // 抽象メソッド
    // ...
};
```

---

## ステップ6：天秤にかける ―― 手段①と手段②を比べる

### 4.6.1 評価軸の宣言

1. **骨格の保護度**：呼び出し順のミスを防げるか
2. **実装の容易さ**：新しいレポートを作るときに迷わないか

### 4.6.2 手段①vs手段②の比較

| 評価軸 | 手段①（ユーティリティ化） | 手段②（Template Method） |
|---|---|---|
| 骨格の保護度 | ❌ 呼び出し忘れが発生しうる | ✅ 親クラスが強制するため安全 |
| 実装の容易さ | ❌ 呼び出す順序を知る必要がある | ✅ 抽象メソッドを埋めるだけでよい |

今回の状況では、監査要件という「守るべき骨格」があるため、**手段②（Template Methodパターン）を採用します。**

---

### 4.6.3 耐久テスト ―― 監査ログの追加

インフラチームから「全レポートに監査ログを！」という要求が来ても、親クラスの `run()` を1箇所直すだけで、全レポートに自動適用されます。

---

## ステップ7：決断と、手に入れた未来

### 4.7 解決後のコード

```cpp
class ReportBatchTemplate {
public:
    void run() {
        std::cout << "[AUDIT] Start" << std::endl;
        execute();
        std::cout << "[AUDIT] End" << std::endl;
    }
protected:
    virtual void execute() = 0;
};

class DailyReportBatch : public ReportBatchTemplate {
protected:
    void execute() override {
        std::cout << "Doing Daily Work" << std::endl;
    }
};
```

### 振り返り：3つの哲学

- **哲学1（変わるものをカプセル化）**：各レポートのロジックをサブクラスに閉じ込めた。
- **哲学3（コンポジション優先）**：今回は例外的に、順序の強力な制御のために「継承」を正しく活用した。
