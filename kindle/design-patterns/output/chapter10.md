## 第10章（応用）外部連携バッチシステムを設計する

―― 思考の型：依存の過多 ＋ 変化の混在 ＋ 生成と利用の混在

### この章の核心

外部システムという「コントロールできない変化」の集合体から、自社の業務コアを安全に守り抜くための多重防壁を築く。

> **【レゴブロックで考える：窓口への集約、別レーンでの生成、一斉通知】**
> 
> 規格がバラバラな他社のブロック群をひとまとめに扱うための「大きなカバー（窓口）」を作ります。さらに、その中に入れる専用の通信ブロックは「別の工場レーン」で作って運び込みます。そして、カバーの中で作業が終わったら、ヒモで繋がった複数のメガホン（通知先）へ一斉に合図を送る仕組みを作ります。これにより、外のブロックがどう変わろうと、中の仕組みは一切影響を受けません。
>

>

### この章を読むと得られること

- **得られること1：** 複数の外部システムに依存した複雑なコードから、真の業務ロジックだけを安全に抽出できるようになる
    
- **得られること2：** 通信手段、通知先、部品の生成といった「理由の異なる変化」を正確に見極め、それぞれ別々に切り離せるようになる
    
- **得られること3：** 仕様変更の際、何十箇所もgrepして影響範囲に怯えることなく、新しいコードを追加するだけで対応できるようになる

### ステップ0：システムを把握し、仮説を立てる ―― クラス構成を見てから「変わりそうな場所」を予測する

> **全パターンに共通する問い**
> 
> 「このコードの中に、『変わる理由』が異なる2つのものが、同じ場所に混在していないか？」
> 
> ※「変わる理由」とは「誰の判断で変わるか」のことです。

現場で巨大なシステムを触っていると、「外部APIの仕様が変わっただけなのに、なぜかバッチ処理のメインクラスまで修正することになり、おまけに無関係なはずのメール通知のテストまでやり直すハメになった」という泥臭いトラブルによく遭遇します。

どこか1つの修正が、予想もしない場所に飛び火してしまう。そんなとき、コードは確実に「変わりそうな場所」と「変わらない場所」の境界線を見失っています。

まずは、今回扱うシステムの全体像を把握し、どこに危険な「混在」が潜んでいそうか、当たりをつけていきましょう。

#### 10.0 この章のシステム構成と仮説

**この章で扱うシステム：**

ECサイトの毎日の売上実績データを集計し、社内の会計システム（ERP）へ連携し、完了結果を管理者に通知する「外部連携バッチシステム」です。異なる複数の外部システムと通信を行う、現場でよくある構成です。

※専門用語の簡単な補足：

・**ERP**：企業のお金やモノの流れを管理する統合システム（ここでは連携先の会計システム）。

・**SOAP / REST**：システム同士がデータをやり取りするための通信ルールのこと。

・**SMTP**：メールを送信するための仕組みのこと。

**仕様表（何ができるシステムか）**

|**機能名**|**担当クラス**|**入力**|**出力**|
|---|---|---|---|
|売上データの外部連携フロー制御|`SalesSyncBatch`|外部APIからの売上CSVデータ|ERPへのSOAP送信、各種手段での完了通知|

**クラス構成の概要**

まずは、現状のシステムがどのような依存関係を持っているか、マクロな視点で図解してみましょう。

コード スニペット

```

→ **【このグラフが示す問題】業務フローの進行を管理するはずの統括クラスが、連携先システムの具体的な通信プロトコル（API、SOAP）や通知手段（SMTPメール）の実装詳細まで、すべて直接知ってしまっている。**

**各クラスの責任一覧**

|**クラス名**|**対象責任（1文）**|**知るべきこと**|
|---|---|---|
|`SalesSyncBatch`|売上データを取得し、会計ERPに連携して結果を通知する業務の進行順序を管理する。|業務ステップの順番（取得→同期→通知）|
|`SalesApiClient`|外部APIにアクセスして売上のCSVデータを取得する。|接続先のURL、HTTP GETメソッドの仕様|
|`LegacyErpClient`|会計ERPに対してデータをSOAPプロトコルで送信する。|ERPのエンドポイント、SOAPのXMLデータ構造|
|`EmailNotifier`|システムの管理者に処理完了のメールを送信する。|SMTPサーバーの設定、管理者のメールアドレス|

この構成を踏まえた上で、実際にコードを読み解く前の「仮説」を立てます。エディタを開いてgrepを叩く前に、少しだけ立ち止まって「どこが変わるのか」を想像することが、設計の第一歩です。

**変動と不変の仮説（実装コードを読む前に立てる）**

|**分類**|**仮説**|**根拠（クラス構成から読み取れること）**|
|---|---|---|
|🔴 **変動する**|外部APIの通信手段やデータフォーマット（CSV等）|自社の裁量ではなく、提供元SaaSの仕様変更に引きずられるため|
|🔴 **変動する**|ERPとの連携プロトコル（SOAPからRESTへの移行等）|全社的なインフラ刷新やクラウド移行の影響を直接受けるため|
|🔴 **変動する**|通知の手段と宛先（SlackやTeamsへの移行、複数人への通知）|社内コミュニケーションツールの流行り廃りの影響を受けるため|
|🔴 **変動する**|どの通信部品を使うかの判断（部品の生成ルール）|テスト環境と本番環境の違いや、インフラの構成変更によって都度切り替わるため|
|🟢 **不変**|データを取得し、連携し、結果を通知するという「業務の進行順序」|会社のビジネスプロセスそのものであり、外部ツールが変わっても目的は変わらないため|

この表を見ると、外部と通信する部分はほとんど「赤い丸（変動する）」で埋め尽くされていることがわかります。外部システムへの依存とは、自分たちではコントロールできない他人の都合（変更理由）を、自社のシステム内に抱え込むことに他なりません。

この仮説が正しいかどうかを、次のステップ1で実際の実装コードに照らし合わせ、ステップ2で関係者へヒアリングして確定させていきましょう。

### ステップ1：実装コードを読む ―― 責任チェックで問題の行を見つける

前回のステップでは、システムの全体像とクラス構成から「外部システムとの通信部分が変わりそうだ」という仮説を立てました。ここからは、現場で実際に稼働している実装コードを開き、クラスが「自分の責任を超えた余計な知識」を持っていないか、1行ずつ入念にチェックしていきましょう。

巨大で複雑な処理を担うクラスを前にして、どこから手を付けていいか迷う経験は誰にでもあると思います。そんな時は、コードが期待通りに動くという事実を一旦脇に置いて、「誰が何を知っているか」という知識の境界線に目を向けるのが設計を読み解くコツです。

まずは、現状のシステムにおける依存の広がりをマクロな視点で確認してみましょう。エディタでコードを追いかけると、以下のような関係性が浮かび上がってきます。

**依存の広がり（実装前の全体像）**

```

この状態をレゴブロックで例えるなら、車を作るための「ボディ」「タイヤ」「エンジン」といった部品が、専用の接着剤で完全に固定されてひとつの巨大な塊になっている状態です。タイヤだけを別の種類に付け替えたいのに、ノコギリでボディごと無理やり切り離さなければならない。これがコードにおける「依存が過多で固着している状態（密結合）」の正体です。

実際のコードを見てみましょう。まずは外部連携を担う周辺の具象クラス群です。

C++

```
#include <iostream>
#include <string>

// ==========================================
// 外部システムとの通信を担当する具象クラス群
// （※解説のため、実際の通信処理は標準出力で簡略化しています）
// ==========================================

class SalesApiClient {
public:
    std::string fetchCsvData(const std::string& url) {
        std::cout << "GET " << url << " -> CSVデータ取得成功\n";
        return "itemA,1000\nitemB,2000";
    }
};

class LegacyErpClient {
public:
    void sendSoapRequest(const std::string& endpoint, const std::string& xmlData) {
        std::cout << "POST " << endpoint << " -> SOAP連携成功 (XML形式)\n";
    }
};

class EmailNotifier {
public:
    void sendEmail(const std::string& address, const std::string& title) {
        std::cout << "Email sent to " << address << " : " << title << "\n";
    }

```

次に、これらを呼び出して業務フローを制御する、問題の起点となるメインのバッチクラスです。今回は、ファイルが肥大化するのを避けるため、ここにすべての処理が書かれていると想像してください。

C++

```
// ==========================================
// 業務バッチの統括クラス
// ==========================================
class SalesSyncBatch {
private:
    SalesApiClient* apiClient;
    LegacyErpClient* erpClient;
    EmailNotifier* emailNotifier;

public:
    SalesSyncBatch() {
        // クラス内部で具体的な通信クラスを自ら生成してしまっている
        apiClient = new SalesApiClient();
        erpClient = new LegacyErpClient();
        emailNotifier = new EmailNotifier();
    }

    ~SalesSyncBatch() {
        delete apiClient;
        delete erpClient;
        delete emailNotifier;
    }

    void run() {
        std::cout << "--- 売上同期バッチ処理 開始 ---\n";

        // 1. データ取得（どのURLから、どのメソッドで取るかを知っている）
        std::string csvData = apiClient->fetchCsvData("https://api.sales.example.com/v1/daily"); // ← 知らなくていい

        // 2. 業務ロジックとデータ変換（CSVの解析方法や、ERPが求めるXMLの構造を知っている）
        std::cout << "CSVデータを解析し、ERP連携用のXMLに変換中...\n"; // ← 知らなくていい
        std::string xmlPayload = "<SalesData><Item>converted</Item></SalesData>";

        // 3. ERPへ連携（どのエンドポイントに、SOAPプロトコルで送るかを知っている）
        erpClient->sendSoapRequest("http://erp.internal:8080/soap", xmlPayload); // ← 知らなくていい

        // 4. 通知（管理者のアドレスや、メールで送ることを知っている）
        emailNotifier->sendEmail("admin@example.com", "バッチ連携完了"); // ← 知らなくていい

        std::cout << "--- 売上同期バッチ処理 終了 ---\n";
    }
};
```

最後に、このプログラムを起動するメイン関数です。

C++

```
// ==========================================

// ==========================================

    SalesSyncBatch batch;
    batch.run();
    
    return 0;
}
```

実行結果の通り、このプログラムはシステムとして完璧に正しく動いています。バッチ処理の要件は十分に満たしているように見えます。

しかし、「設計」という観点からコードを評価するとどうでしょうか。

ここで、`SalesSyncBatch` クラスが何をするクラスなのかを改めて1文で定義してみます。

**対象責任：** 「売上データを取得し、会計ERPに連携して結果を通知する」という業務の進行順序を管理する。

この定義に従って、先ほどのコードの各行が「本当にこのクラスが持つべき知識なのか？」を判定する責任チェックを行ってみましょう。

**責任チェック：SalesSyncBatch は自分の責任だけを持っているか**

|**コードの行**|**持っている知識**|**責任内か（誰の責任か）**|
|---|---|---|
|`apiClient = new SalesApiClient();` 等|利用する具体的な具象クラスとその生成手順|✗ 組み立てる側の責任（Factoryや外部インジェクション）|
|`fetchCsvData("https://api...");`|接続先のURLと、取得手段がCSVであること|✗ 設定管理とAPI通信クラスの責任|
|`xmlPayload = "<SalesData>...";`|連携先が要求する独自のXMLデータフォーマット|✗ データ変換（Adapter）の責任|
|`erpClient->sendSoapRequest(...);`|ERPへの連携手段がSOAPプロトコルであること|✗ ERP通信クラスの責任|
|`emailNotifier->sendEmail(...);`|通知手段がメールであることと、送信先のメアド|✗ 通知クラスの責任|

このように確認していくと、`SalesSyncBatch` は業務の進行順序を管理する立場でありながら、「どのURLにアクセスするか」「どんな形式のデータか」「どのプロトコルで通信するか」さらには「部品をどうやって作るか」といった、**外部システムの具体的な実装詳細という余計なことまで知りすぎている**ことが分かります。

これが、コードのあちこちに「知らなくていい」というラベルが付いてしまった理由です。現状の `SalesSyncBatch` クラスは、巨大な接着剤の塊なのです。

#### 10.2 届いた変更要求

コードの全体像と責任の偏りが見えてきたタイミングで、現場ではよくある光景が展開されます。開発チームのチャットに、インフラ担当者と業務部門のリーダーから立て続けにメンションが飛び込んできました。

- **差出人：** インフラ担当チーム ＆ 業務部門リーダー
    
- **要求内容：**
    
    「お疲れ様です。来月から全社の会計システムが新しいクラウドERPに移行します。これに伴い、売上バッチの連携先を従来のSOAP方式から新しいREST API方式（JSON形式）に変更してください。
    
    それから、バッチが完了した時の通知ですが、今後はメールだけでなく、運用チームのSlackチャンネルにも同時に送るようにしてください。
    
    あ、あと来期から『検証環境』と『本番環境』で、接続するAPIのクライアント（モックか本番か）を切り替えられるようにしておいてほしいです」
    
- **期日：** 今月末のリリースで
    

…思わずため息が出そうになる変更要求ですね。

通信方式の変更（RESTへ）、通知先の増加（Slack追加）、そして環境による部品の切り替え（生成の分岐）。これらはすべて「変わる理由」が異なる別の問題です。しかし、現在の `SalesSyncBatch` クラスのままこれらに対応しようとすると、どうなるでしょうか。

通信方式を変えるために `run()` の中の処理を書き換え、通知方法を足すためにメンバ変数を増やし、環境を切り替えるためにコンストラクタの中に `if (env == "test")` といった条件分岐を無限に足していく未来が容易に想像できます。

コードが機能として「動いている」ことと、変更要求に耐えられる「構造になっている」ことは、全く別の問題なのです。

次はこの変更要求の背景にある「変わる理由」を関係者にヒアリングし、私たちの仮説を確固たる設計の前提へと昇華させていきましょう。

### ステップ2：仮説を確定する ―― 関係者ヒアリングで「変わる理由」に根拠をつける

#### 10.3 仮説の検証と変動/不変の確定

ステップ0でシステムの全体像から仮説を立て、ステップ1の責任チェックで、現在の `SalesSyncBatch` クラスが「通信プロトコル」「宛先」「部品の作り方」という実装の細部に過剰に依存している事実を確認しました。

そして飛び込んできたのが、「REST APIへの移行」「Slack通知の追加」「環境による接続先の切り替え」という3つの変更要求です。

現場のプレッシャーに追われ、納期が迫っていると、ここでいきなりエディタを開いて「よし、まずはREST用のクラスを書いて、if文で切り替えて……」とコードを書き始めたくなります。しかし、少しだけ立ち止まってください。言われた通りの変更をそのまま if 文で既存の巨大クラスに追加していくと、半年後には「今度はTeams通知も追加して」「やっぱり別のSaaSに乗り換える」と言われ、同じように巨大な統括クラスを開いて、怯えながら書き換えることになります。

コードを読んだだけで「ここは変わる」「ここは変わらない」と断定するのは危険です。コードを触る前に、「なぜこの変更が起きるのか」「次に何が変わるのか」を、システムを利用する業務部門や、インフラを管理する担当者に直接聞いてみましょう。

現場の声を聞くことで、「なんとなく変わりそう」という開発者の勘が、確かな根拠のある「設計の前提（契約）」へと変わります。

**関係者ヒアリング**

- **開発者：** 「今回のクラウドERPへのREST移行や、Slack通知の追加、承知しました。少し先の見通しについて確認させてください。今後もこういった連携先のSaaS変更や、通知先の追加は発生しそうですか？」
    
- **インフラ担当リーダー：** 「はい、発生する前提でいてください。クラウドシフトは全社的な方針なので、数年単位で利用するインフラサービスは変わっていくはずです。また、検証環境と本番環境で接続先（モックと本番）を頻繁に切り替える運用も定着させたいです。」
    
- **業務部門リーダー：** 「通知先についても同じですね。今はSlackですが、全社ツールがTeamsに変わればそちらにも通知したいですし、監査部門からも『完了ログをメールで送ってほしい』と後から追加要望が来るかもしれません。」
    
- **開発者：** 「なるほど。インフラの都合や社内ツールの流行り廃りで、手段はどんどん変わっていくわけですね。では、バッチ処理の大きな流れ——『売上データを取得して、ERPに連携して、最後に関係者に結果を通知する』という業務の進行順序自体は変わりますか？」
    
- **業務部門リーダー：** 「そこは絶対に変わりません。うちの部署の法的要件を含んだ仕事の流れそのものですから。通信方式がRESTになろうが手紙になろうが、データを取得して相手に報告するという手順は不変です。」
    
- **開発者：** 「安心しました。ちなみに、連携するデータの性質についてはどうでしょう？ 将来的に、例えば売上金額などのデータの持ち方が根底から変わる可能性はありますか？」
    
- **業務部門リーダー：** 「あぁ、実は来年から海外展開を見据えていて、今は円（整数：`int`）で扱っている金額を、多通貨対応のために独自の通貨型や小数で扱うようにビジネスルールが変わるかもしれません。」
    

ヒアリングを通じて、システムのどこが変わりやすく、どこが安定しているかが見えてきました。そして何より重要なのは、「変わる理由（誰の判断で変わるか）」が明確になったことです。

この結果をテーブルにまとめましょう。

|**分類**|**具体的な内容**|**変わるタイミング（誰の判断か）**|**根拠**|
|---|---|---|---|
|🔴 **変動する**|外部システムとの通信手段（HTTP, SOAP, REST等）|利用するSaaSやインフラ方針が変わったとき|インフラ担当リーダー|
|🔴 **変動する**|通知先の種類と数（Email, Slack, Teams等）|社内コミュニケーションツールや監査要件が変わったとき|業務部門リーダー|
|🔴 **変動する**|どの通信部品を使うか（本番用かモックか）|実行する環境（検証/本番）を切り替えるとき|インフラ担当リーダー|
|🔴 **変動する**|引数のプリミティブ型（金額をintから多通貨対応型へ）|グローバル展開等でビジネスの数値定義が変わったとき|業務部門リーダー|
|🟢 **不変**|業務フローの進行順序（取得→連携→通知）|会社の業務プロセスそのものが変わらない限り|業務部門リーダーとの合意|

このテーブルを見ると、システムの中に「インフラ都合で変わるもの」「通知の都合で変わるもの」「環境都合で変わるもの」という、全く異なる複数の「変わる理由」が混在していることがはっきりと分かります。

> **設計の決断：**
> 
> 🟢 不変な部分である「業務の進行順序」を、バッチ統括クラスの「契約（インターフェース）」として固定します。
> 
> 🔴 インフラ都合で変わる「通信手段」はインターフェースの裏側に押し込み（**Facadeの準備**）、
> 
> 🔴 業務都合で増減する「通知先」は統括クラスから切り離し（**Observerの準備**）、
> 
> 🔴 環境都合で変わる「部品の生成」は外の工場に任せます（**Factory Methodの準備**）。

ここで特筆すべきは、業務担当者から引き出した「**金額を扱う型自体が変わる可能性（引数のプリミティブ型が変わるリスク）**」です。

インターフェースを使って部品を差し替え可能にすれば、通信方式や通知ツールの変更からは身を守ることができます。しかし、「メソッドに渡す引数の型」そのものが変わってしまった場合はどうなるのでしょうか？

この「型変更リスク」に対して、私たちがこれから構築する強固な設計パターンがどこまで立ち向かえるのか、あるいは限界があるのかについては、ステップ6の耐久テスト（10.10）でコードを通して正直に向き合うことにします。

さて、変動と不変の境界線、そして「誰の都合で変わるのか」が明確に引けました。次はこの変更要求を「今のコード構造のまま」実装しようとした場合、私たちが現場でどんな痛みを味わうことになるのか、課題の分析を進めていきましょう。

### ステップ3：課題分析 ―― 変更が来たとき、どこが辛いかを確認する

ステップ2のヒアリングを通じて、通信プロトコル（インフラ都合）や通知先（業務都合）、部品の切り替え（環境都合）は「それぞれ別の理由で頻繁に変わるもの」、業務の進行順序は「変わらないもの」という境界線が見えました。

では、この境界線が引かれておらず、すべてが接着剤で固着してしまっている「今のコード（ステップ1の `SalesSyncBatch`）」のまま、先ほどの変更要求（ERPのREST化、Slack通知の追加、モック切り替え）に応えようとすると、現場で何が起きるでしょうか。

仕様書とコードを照らし合わせながら、頭の中で実装作業をシミュレーションしてみましょう。

「まずは、REST API用の新しい通信クラス `RestErpClient` を作って…」

「通知用に `SlackNotifier` も新しく書く必要があるな…」

「それができたら、バッチの統括クラス `SalesSyncBatch` を開いて、`new` している部分を検証環境用と本番用で切り替えるために `if (env == "test")` の分岐を書いて…」

「あっ、通信のメソッド名も `sendSoapRequest` から `sendRestRequest` に変わるから、呼び出し箇所も全部書き換えないと…」

「Slack通知も追加だから、メール送信の直後に `slackNotifier->send()` を足して…」

この一連の作業を想像したとき、私たちは現場で2つの大きな「痛み」に直面します。

#### 痛み1：影響範囲が広すぎて「grep地獄」に陥る

新しい通信クラスや通知クラスを作ること自体は、それほど難しくありません。本当に辛いのは、その「新しいクラスを使うように既存のコードを書き換える」作業です。

現状の `SalesSyncBatch` は、具象クラスを直接 `new` して、その具体的なメソッドを直接呼び出しています。つまり、通信プロトコルやメソッド名が変われば、呼び出し元である `SalesSyncBatch` のコードも必ず書き換えなければなりません。

「どこで古いクラスを呼んでいる？」「どこで引数の形式（XMLからJSONへ）を組み立てている？」

影響範囲を特定するために、エディタの検索窓に古いクラス名やメソッド名を入れて、ひたすらプロジェクト全体を grep（文字列検索）して回ることになります。もしこの統括クラスが数百行、数千行に肥大化していたら、修正漏れがないかビクビクしながらコードを追いかけることになります。影響範囲が事前に予測できない状態での修正作業は、エンジニアの気力を確実に削っていきます。

#### 痛み2：変更の「飛び火」による無関係なテストのやり直し

さらに厄介なのが、Slack通知の追加や、テスト環境と本番環境の切り替えという要件です。

今の構造のままこれらに対応しようとすると、`SalesSyncBatch` の中に「環境がテストならモックを生成する」「完了したらメールを送り、さらにSlackにも送る」といった処理を直接書き足していくことになります。

インフラ側の都合や、監査の都合といった「外側の世界の変化」が起きるたびに、この `SalesSyncBatch` というクラスに手を入れなければなりません。業務の進行順序を管理している「システムのコア」となるクラスに変更が加わるということは、本来なら影響を受けないはずの「データ取得は今まで通り動くか」「メール送信は壊れていないか」まで、すべて回帰テストをやり直す必要が出てくるということです。

現場でコードを修正しようとエディタを開いたとき、思わずこんな独り言が漏れてしまいます。

「通知先が1つ増えただけなのに、なんで業務フロー全体を管理してるクラスまで開いて、if文を足さなきゃいけないんだ…？」

この呟きの原因を、図で可視化してみましょう。変更要求が、どのように無関係なクラスへと「飛び火」していくかを示すグラフです。

**依存の広がり（改善前の変更影響グラフ）**

この図を見ると、現状のコードに変更を加えようとしたとき、何クラス開けばいいのか（どこまで直せばいいのか）が一目で分からない状態になっていることが分かります。

ここから、私たちが目指すべきゴールが見えてきます。

設計の価値というのは、「新しいコードをきれいに書けること」だけではありません。既存のコードに変更加わるとき、影響を「この1箇所」に閉じ込められること。変更要求が来た瞬間に、「あぁ、あのファイルだけを開いて、あそこだけ直せば済むな」と一目で分かる状態を作ること。これこそが、設計の真の価値なのです。

なぜ、現状のコードはあらゆる変更が中央のクラスに飛び火してしまうのでしょうか。次のステップで、この困難の根本にある原因を言語化し、私たちが持っている設計の「手札」の中から、この複雑に絡み合った問題を解きほぐすための武器を探っていきましょう。

### ステップ4：原因分析 ―― 困難の根本にある設計の問題を言語化する

ステップ3で直面した、終わりの見えない「grep地獄」と、無関係な箇所への「変更の飛び火」。これらは現場のエンジニアの気力を確実に削っていきます。仕様変更が来るたびに残業してコードを書き換えていると、「自分が書いたコードが悪いのだろうか」「もっと注意深くif文を書けばよかったのか」と落ち込んでしまうこともあるかもしれません。

しかし、自分を責める必要はありません。これはエンジニア個人のスキルの問題ではなく、コードの「構造」が引き起こしている必然的な現象だからです。ここでは一旦エディタから手を離し、なぜあのような苦しい影響の連鎖が起きるのか、冷静に現象を整理して言葉にしてみましょう。

**観察と原因の方向**

コードを注意深く観察すると、次のような事実が浮かび上がってきます。

|**観察**|**原因の方向**|
|---|---|
|`SalesSyncBatch` の中に、URLやXMLの組み立て、HTTPやSOAPといった具体的なプロトコル操作が直接書かれている。|使う側が、相手の「詳細な通信方式やデータ形式」といった知らなくていいことまで知ってしまっている（依存の過多）。|
|`SalesSyncBatch` の中に、`emailNotifier->sendEmail(...)` という特定の手段による通知処理が直接書かれている。|業務の進行管理という「変わらないもの」の中に、通知先という「増減しやすい変化」が混ざってしまっている（変化の混在）。|
|`SalesSyncBatch` が自らのコンストラクタ内で `new SalesApiClient()` のように具体的な部品を生成している。|何を作るかの判断（環境切り替え等）と、どう使うか（業務フロー）が同じ場所にある（生成と利用の混在）。|

これらの観察から分かるのは、`SalesSyncBatch` というクラスが、本来の責任である「業務フローの進行管理」を遥かに越えて、周囲のあまりにも多くの事情を抱え込んでしまっているという事実です。

そして、ステップ2のヒアリングで得た結果と照らし合わせると、より決定的な問題の構図が見えてきます。

**変わるものと変わらないものが同じ場所にいる**

|**変わり続けるもの（🔴）**|**変わってほしくないもの（🟢）**|
|---|---|
|インフラ都合による通信手段（SOAP方式からREST方式へ）|業務フローの進行順序（データを取得して連携する）|
|業務都合による通知手段（EmailからSlackの追加へ）|処理が完了したことを「誰か」に知らせるという事実|
|環境都合による通信部品の選択（モックを使うか本番を使うか）|契約通りの通信部品を使って業務を進めるという事実|

インフラの都合、運用チームの都合、環境構築の都合という「全く異なる理由で変わるもの」が、会社の業務プロセスという「絶対に変わってほしくないもの」と、ひとつのファイルの中でドロドロに混ざり合っています。

レゴブロックで想像してみてください。本来なら「車のボディ（業務フロー）」「タイヤ（通信）」「サイレン（通知）」は別々の部品であるべきです。しかし今の状態は、これらが製造段階で接着剤を流し込まれ、一つの塊に固着してしまっている状態です。サイレンをもう一つ足そうと思ったら、ボディにドリルで穴を開けなければなりません。

この「複雑に絡み合った固着」を解きほぐすために、私たちが第0章で学んだ設計の「4つの手札」をテーブルに並べてみましょう。現状は3つの異なる病（依存の過多、変化の混在、生成の混在）を併発しているため、複数の手札を組み合わせて立ち向かう必要があります。

**根本原因に対する構造的対策案（手札の選定）**

|**次元**|**物理操作（手札）**|**本質的な原因（何が問題か）**|**使うべき構造的対策案（本質）**|
|---|---|---|---|
|関係|**④ 間接化する**（間に挟む）|バッチ統括クラスが、複数の複雑な外部APIと直接通信しており、仕様（URLやXML構造）を知りすぎている（依存の過多）。|業務と外部システムの間に、複雑さを隠蔽する「窓口（Facade）」となる中間層を配置する。|
|関係|**③ 規格化する**（形を揃える）|バッチ統括クラスが、「メール送信」「Slack送信」といった具体的な通知実装に直接依存している（変化の混在）。|通知先を「イベントの購読者（Observer）」としてインターフェースで統一し、抽象への依存へ逆転させる。|
|要素|**① 分割する**（切る）|使う部品（通信クライアント）の生成（new）と、それを使った業務処理が一つの塊に癒着している（生成と利用の混在）。|「どの環境の部品を作るか」という生成の責任を、専用の工場クラス（Factory）へ切り出す。|

今回の「外部連携バッチシステム」は非常に厄介です。なぜなら、単に振る舞いが変わるだけでなく、「知らなくていいことまで知っている問題（Facadeの領域）」「通知先が増減する問題（Observerの領域）」そして「環境によって作るものが変わる問題（Factory Methodの領域）」が、すべて1つのクラスに集中しているからです。

しかし、原因は明確に言語化できました。これらが別々の問題であると見極められた時点で、設計の勝負は半分勝っています。

次のステップ5では、この選定した手札を使って、実際にコードの構造を段階的に変革していく思考のプロセスを体験していきましょう。どうすれば既存のコードを壊さずに、3つの変化から同時に身を守る「多重防壁」を手に入れられるのか、手を動かして検証します。

### ステップ5：対策案の検討 ―― 原因から手札を選ぶ

**特定した真因：** バッチ統括クラスが、①通信手段の実装詳細、②通知先の宛先リスト、③環境に応じた部品の生成方法という、「変わる理由が全く異なる3つの問題」をすべて自分の中に抱え込んでいる。

この複雑に絡み合った病を解決するために、第0章で学んだ設計の手札を順番に切っていきましょう。いきなり完璧な答えを目指すのではなく、段階的に構造をほぐしていきます。

#### 1. 分離・隠蔽を試す（手段①：直接依存をそのままに、処理を別クラスに切り出す）

まずは手札①（分割）と手札②（隠蔽）を使って、巨大な `SalesSyncBatch` の中から「データ変換」や「具体的な通信」の処理を外に追い出してみます。ここでは、連携に必要な処理をひとまとめにする別クラス群を作って、処理を委譲してみましょう。

C++

```
#include <iostream>
#include <string>
#include <vector>

// ==========================================
// 処理を切り出した別クラス群（分割・隠蔽）
// ==========================================

// 通信とデータ変換をまとめたクラス（モック用と本番用）
class MockSyncAdapter {
public:
    void sync() {
        std::cout << "テスト環境：モックAPIを使用して同期成功\n";
    }
};

class RestSyncAdapter {
public:
    void sync() {
        std::cout << "本番環境：REST APIでデータ同期成功\n";
    }
};

// 通知専用のクラス群
class EmailNotifier {
public:
    void sendEmail() { std::cout << "Email: バッチ連携完了\n"; }
};

class SlackNotifier {
public:
    void sendSlack() { std::cout << "Slack: バッチ連携完了\n"; }
};
```

これにより、統括クラスである `SalesSyncBatch` のコードは次のようにすっきりします。

C++

```
// 業務バッチの統括クラス（手段①）
class SalesSyncBatch {
private:
    std::string currentEnv;
    MockSyncAdapter* mockAdapter;
    RestSyncAdapter* restAdapter;
    EmailNotifier* emailNotifier;
    SlackNotifier* slackNotifier;

public:
    // 環境情報を渡し、内部でどの部品を使うか判断して生成している
    SalesSyncBatch(const std::string& env) : currentEnv(env) {
        mockAdapter = new MockSyncAdapter();
        restAdapter = new RestSyncAdapter();
        emailNotifier = new EmailNotifier();
        slackNotifier = new SlackNotifier();
    }

    ~SalesSyncBatch() {
        delete mockAdapter;
        delete restAdapter;
        delete emailNotifier;
        delete slackNotifier;
    }

    void run() {
        std::cout << "--- バッチ処理開始 ---\n";

        // 1. 環境に応じた通信処理の実行（分岐が残っている）
        if (currentEnv == "test") {
            mockAdapter->sync(); // ← 呼び出し元が環境の違いを知っている
        } else {
            restAdapter->sync();
        }

        // 2. 通知処理（通知先が増えるたびにここに追記が必要）
        emailNotifier->sendEmail();  // ← メールの送り方を知っている
        slackNotifier->sendSlack();  // ← Slackの送り方を知っている

        std::cout << "--- バッチ処理終了 ---\n";
    }
};
```

コードの見通しはかなり良くなり、`run()` メソッドの中から細かいURLやXMLの組み立て処理は消えました。手続き型プログラミングとしては、かなり綺麗に整理された状態です。

しかし、この「手段①」の構造のまま、もう一度責任チェックを行ってみましょう。

**手段①コードの責任チェック：**

|**コードの行**|**持っている知識**|**責任内か**|
|---|---|---|
|`mockAdapter = new ...` 等|どの具象クラスを使うかという「生成」の知識|✗ 組み立てる側の責任|
|`if (currentEnv == "test")`|実行環境の違いと、それに紐づく部品の選択ルール|✗ 工場（Factory）の責任|
|`slackNotifier->sendSlack()`|通知先としてSlackが存在し、専用のメソッドがあること|✗ 購読者（Observer）の管理責任|

この構造の最大の弱点は、「呼び出し元が、依存先の変更理由を引き継いでしまう」という点にあります。

例えば、全社ツールがTeamsに移行してSlack通知が廃止されたとします。すると `SlackNotifier` が消えるだけでなく、それを使う `SalesSyncBatch` クラスを開いて `sendSlack()` の呼び出し行を削除し、代わりに `teamsNotifier->sendTeams()` を追加しなければなりません。さらに本番環境のインフラが変われば、また `if` 文の中身を書き換えます。

つまり、手札①（分割）で処理を別クラスに分けても、クラス同士のつなぎ目が「具象クラスへの直接依存」のままであるため、外部の変更要求が中央のクラスに簡単に飛び火してしまうのです。

#### 2. さらに規格化・間接化を重ねる（手段②：複合パターンの導入）

手段①の限界を突破するために、手札③（規格化）と手札④（間接化）を追加適用し、設計パターンを組み合わせます。

**解決策のアプローチ：**

1. **Facade（間接化）:** `SalesSyncBatch` が知るべき「ビジネス上の責任」だけで通信用のインターフェースを定義し、複雑な処理をその裏に隠します。
    
2. **Observer（規格化）:** 通知先を「バッチ完了を購読する者（`ISyncObserver`）」として規格化します。「メール」や「Slack」といった実装手段の名前は使わず、ビジネス上の役割で命名します。
    
3. **Factory Method（生成の分離）:** 統括クラスの中での `new` と `if` 文をやめ、「どの環境でどの通信部品を作るか」という判断を専用の工場インターフェース（`IBatchFactory`）に任せます。
    

実装を見てみましょう。まずは「ビジネスの言葉で定義された契約（インターフェース）」です。

C++

```
#include <iostream>
#include <string>
#include <vector>

// ==========================================
// 1. ビジネスの言葉で定義されたインターフェース群（契約）
// ==========================================

// 通信とデータ同期の複雑さを隠蔽する窓口（Facade的役割）
class ISystemSyncer {
public:
    virtual ~ISystemSyncer() {}
    virtual void executeSync() = 0; // ← 相手がRESTかモックかを隠蔽
};

// バッチ完了の通知を受け取る購読者（Observer）
class ISyncObserver {
public:
    virtual ~ISyncObserver() {}
    virtual void onSyncCompleted() = 0; // ← 宛先がメールかSlackかを隠蔽
};

// 適切な通信部品（Syncer）を生成する工場（Factory Method）
class IBatchFactory {
public:
    virtual ~IBatchFactory() {}
    virtual ISystemSyncer* createSyncer() = 0; // ← 環境の分岐判断を隠蔽
};
```

次に、これらを実装する具体的なクラス群（アダプターやコンクリートな部品）を作ります。ここで初めて、実装手段の名前（RestやSlack）が登場します。

C++

```
// ==========================================
// 2. インターフェースを実装する具象クラス群
// ==========================================

// --- 通信部品 ---
class RestErpSyncer : public ISystemSyncer {
public:
    void executeSync() override {
        std::cout << "POST /api/v2/sync -> REST APIで連携成功 (JSON形式)\n";
    }
};

class MockErpSyncer : public ISystemSyncer {
public:
    void executeSync() override {
        std::cout << "テスト環境：モックAPIを使用して同期成功\n";
    }
};

// --- 通知部品 ---
class EmailNotifier : public ISyncObserver {
public:
    void onSyncCompleted() override {
        std::cout << "Email sent to admin: バッチ連携完了\n";
    }
};

class SlackNotifier : public ISyncObserver {
public:
    void onSyncCompleted() override {
        std::cout << "Slack channel #alerts: バッチ連携完了\n";
    }
};

// --- 工場部品 ---
class ProductionBatchFactory : public IBatchFactory {
public:
    ISystemSyncer* createSyncer() override {
        return new RestErpSyncer(); // 本番用の部品を作る
    }
};

class TestBatchFactory : public IBatchFactory {
public:
    ISystemSyncer* createSyncer() override {
        return new MockErpSyncer(); // テスト用の部品を作る
    }
};
```

そして最後に、業務のコアとなる統括クラスを書き換えます。

C++

```
// ==========================================
// 3. 業務統括クラス（Facadeであり、Subjectでもある）
// ==========================================
class SalesSyncBatch {
private:
    // インターフェース型のポインタのみを保持している。
    // 具体的な通信クラスや通知クラスを一切知らない。
    IBatchFactory* factory;
    std::vector<ISyncObserver*> observers;

public:
    // コンストラクタで工場を受け取る（依存性の注入）
    SalesSyncBatch(IBatchFactory* f) : factory(f) {}

    // 通知先（Observer）を後から追加できる仕組み
    void addObserver(ISyncObserver* observer) {
        observers.push_back(observer);
    }

    void run() {
        std::cout << "--- バッチ処理開始 ---\n";

        // 1. 工場に部品の生成を依頼する（自分で new しない、if文で分岐しない）
        ISystemSyncer* syncer = factory->createSyncer();

        // 2. 外部システムへ連携する（RESTかモックかは知らない）
        syncer->executeSync();

        // 3. 登録されている全ての人に一斉通知する（宛先がどこかは知らない）
        for (ISyncObserver* observer : observers) {
            observer->onSyncCompleted();
        }

        delete syncer; // 工場で作った部品は使い終わったら破棄
        std::cout << "--- バッチ処理終了 ---\n";
    }
};
```

見てください。`SalesSyncBatch` の中から、具体的なクラスの `new` も、環境を判定する `if` 文も、具体的な技術名（REST、Slack）も完全に消え去りました。メンバ変数として持っているのは、インターフェース型のポインタだけです。このクラスは「自分がどの具体的なクラスを使っているか」を一切知らないことが、コードの型情報から証明されています。

これにより、呼び出し元である統括クラスの「変わる理由」が、「業務の進行順序自体が変わったとき」という1つだけに絞り込まれました。

レゴブロックで例えるなら、外側に規格化されたジョイントを持つ「大きなカバー」を用意し、通信ブロックはそのジョイントに合うものを専用のレーン（工場）から運んできてカチッと嵌め込む。そして、カバーの配線から外に向かって、形の違う複数のメガホン（通知先）へ一斉に合図を送る仕組みが完成した状態です。

では、この手札を組み合わせた構造（手段②）の導入コストは、変更耐性という未来のメリットに本当に見合うのでしょうか？ そして、ヒアリングで挙がった「型自体の変更リスク」に対してはどう立ち向かうべきなのでしょうか。次のステップ6で、天秤による評価と「耐久テスト」を行っていきましょう。

### ステップ6：天秤にかける ―― 手段を評価する

ステップ5で、私たちは2つのアプローチを試しました。

手段①は、処理を別のクラスに切り出す「分離・隠蔽」の手法です。統括クラスから細かいURLの指定やデータ組み立てのロジックは消え、コードは格段に読みやすくなりました。しかし、統括クラスは依然として「具象クラスを自分で生成し、環境に応じてif文で分岐する」状態であり、連携先や通知先が増えれば、そのたびに統括クラスも書き換えが必要になります。つまり、「拡張」という評価軸では不合格です。

手段②は、そこへさらに「規格化」と「間接化」を重ね、Facade・Observer・Factory Methodの複合構造を導入した手法です。統括クラスは具体的な通信プロトコルや通知先を一切知らず、ただ「契約（インターフェース）」通りに処理を進めるだけになりました。これにより、既存コードを一切触らずに部品を差し替える「置換」と、新機能を足す「拡張」の能力を手に入れています。

とはいえ、手段②はインターフェースや工場クラスをいくつも定義しなければならず、ファイル数も増えます。「クラスが増えて複雑になる。そのコストは変更耐性に見合うのか？」という疑問は、現場のエンジニアとして当然の感覚です。

ここで、両者を冷静に天秤にかけ、ヒアリング結果を踏まえてどちらを採用するか決断しましょう。

#### 10.4 手段①vs手段②の比較

設計の善し悪しを測るため、4つの評価軸を宣言します。

|**評価軸**|**手段①（分離・隠蔽のみ）**|**手段②（複合パターンの導入）**|**評価の観点**|
|---|---|---|---|
|**現状の設計コスト**|**良い**：切り出すだけなので実装工数は低い。|**悪い**：インターフェース定義と工場（Factory）の構築に初期コストがかかる。|将来の変更コストを「今の実装コスト」で買う投資判断として見合うか。|
|**現状の評価コスト**|**普通**：各クラス単体のテストは可能だが、環境ごとの分岐は結合テストが必要。|**良い**：工場でモック部品を注入できるため、外部システムが未完成でもテストが容易。|テストの独立性と自動化のしやすさ。|
|**未来の設計コスト**|**悪い**：変更のたびに統括クラスのif文やメソッド呼び出しを書き換える必要がある。|**良い（置換・拡張）**：統括クラスを一切触らず、新しい部品を追加するだけで対応できる。|**「変更容易性」**。別の仕様変更が来た際、どれだけ少ない修正で対応できるか。|
|**未来の評価コスト**|**悪い**：変更要求が来たとき、統括クラスのどこを修正すればいいかgrepで探す必要がある。|**良い**：影響範囲が追加した具象クラスと工場の局所化されるため、影響の追跡が不要。|**「保守・追跡性」**。不具合調査時に影響範囲をすぐ特定できるか。|

> **結論：** 今回の状況では、**手段②（Facade × Observer × Factory Method）**を採用します。
> 
> ステップ2のヒアリングで「クラウドERPへの移行」「Teams通知などの追加」「本番とモックの切り替え運用」という確実な変動リスクが確認されているためです。将来起こる終わりのない「grep地獄」と「無関係なテストのやり直し」を防ぐための投資として、今の実装コストは十分に回収できると判断できます。

#### 10.5 耐久テスト ―― ヒアリングで挙がった変化が来た

では、私たちが採用した手段②の複合構造が、ステップ2で予告されていた変更要求の「痛み」から本当に守ってくれるのか、耐久テストとして実演してみましょう。

**シナリオ：**

1. ERP連携を「REST API方式」へ移行する。
    
2. 通知先に「Slack」を追加する。
    
3. 検証環境用に「モック部品」に切り替える。
    

手段②の構造であれば、既存の `SalesSyncBatch` や各インターフェースのコードは**1行も開きません**。ただ、新しい規格に合わせた部品クラスを「追加」し、工場の組み立てラインを変更するだけで済みます。

C++

```
#include <iostream>
#include <vector>

// 既存のインターフェース群（変更なし）
class ISystemSyncer { public: virtual ~ISystemSyncer() {} virtual void executeSync() = 0; };
class ISyncObserver { public: virtual ~ISyncObserver() {} virtual void onSyncCompleted() = 0; };
class IBatchFactory { public: virtual ~IBatchFactory() {} virtual ISystemSyncer* createSyncer() = 0; };

// ==========================================
// 耐久テスト：新しい部品を「追加」するだけ
// ==========================================

// 1. 新しいREST API用の通信部品（Facadeの裏側に追加）
class RestErpSyncer : public ISystemSyncer {
public:
    void executeSync() override {
        std::cout << "POST /api/v2/sync -> REST APIで連携成功 (JSON形式)\n"; 
    }
};

// 2. 新しいSlack用の通知部品（Observerに追加）
class SlackNotifier : public ISyncObserver {
public:
    void onSyncCompleted() override {
        std::cout << "Slack channel #alerts: バッチ連携完了\n";
    }
};

// ==========================================
// 変更を吸収する場所（工場の切り替え）
// ==========================================

// 3. 環境に応じた工場を用意する（今回は本番用のREST工場）
class ProductionBatchFactory : public IBatchFactory {
public:
    ISystemSyncer* createSyncer() override {
        return new RestErpSyncer(); // ← ここで新しいREST部品を返すようにするだけ
    }
};
```

**結果の解説：** 既存の業務ロジックを一切壊すことなく、新しい通信方式と通知先を実装できました。統括クラス（`SalesSyncBatch`）にはif文を1つも足していません。実装クラス名に `SlackNotifier` のように具体的な技術手段を含んでいますが、差し替え後のクラス名がそのまま「どう変更したか」という変更内容の説明になるため、問題ありません。

あの恐ろしいgrep地獄は発生しません。変更要求が来たら、「新しいファイルを作り、工場で部品を入れ替え、統括クラスに購読者（Observer）として登録する」。それだけでタスクは完了します。

##### 守れない変更への正直な告白と、カプセル化への応用

しかし、設計に絶対の魔法はありません。 ヒアリングで挙がっていた「来年、金額のデータ型が `int` から多通貨対応の独自型に変わるかもしれない」というリスクについて正直にお話しします。

もし引数のプリミティブ型が根底から変わってしまった場合、例えば `executeSync(int totalSales)` だったメソッドが `executeSync(Currency totalSales)` に変わるような事態が起きれば、いかなる設計パターンであっても防ぐことはできません。すべてのインターフェースのシグネチャを書き換えることになり、変更の飛び火が発生します。

「こういう状況ではパターンの限界ではなく、型をどう扱うかを先に決める問題」になります。

インターフェースは「呼び出し元が知らなくていい詳細を隠す契約書」です。メソッドシグネチャ（引数の型）もその詳細の一部であると考え、基本の型を直接引き回すのではなく、独自の型（例えば `SalesData` クラス）で型情報自体を呼び出し元から隠蔽する（カプセル化する）ことが有効です。

C++

```
// 型変更の影響を防ぐための、カプセル化の応用例
class SalesData {
private:
    // 内部にどんな型でデータを持っているか（intか、多通貨型か）を隠蔽する
    int amount; 
public:
    // ...
};

class ISystemSyncer {
public:
    virtual ~ISystemSyncer() {}
    // 引数をプリミティブ型ではなく、カプセル化された独自型にする
    virtual void executeSync(const SalesData& data) = 0; 
};
```

どれが正解かは、どこまで型が安定しているかをチームで話し合って決めるべきでしょう。

#### 10.6 使う場面・使わない場面

手段②の複合構造は極めて変更に強いですが、常にこのパターンを使えばいいというわけではありません。 パターンを使うとクラスが増えて複雑になります。本当に毎回使わなければならないのでしょうか？

結論から言えば、「シンプルに保つこともエンジニアリングの重要な選択」です。パターンを使わずに既存コードへの最小限の変更で対処できる場合（手段①のような単純な関数分割など）は、それを選ぶべきです。変更頻度・チーム規模・将来の見通しの3軸で判断しましょう。

**【過剰コード：変化の予定がないものまでパターン化した例】**

「自社内の閉じたネットワークにある、絶対に仕様変更されないレガシーな社内DB」にデータを書き込み、エラーがあれば標準出力にログを出すだけの使い捨てバッチ処理に、わざわざ `ISystemSyncer` や `ISyncObserver`、さらには Factory まで構築するのは、明らかな過剰設計（オーバーエンジニアリング）です。その場合は、全ての手続きを1つのクラスにベタ書きする方が、よほど素早く価値を提供できます。

|**状況**|**適切な選択**|**理由**|
|---|---|---|
|**連携先がSaaSで、通知要件や環境の切り替えが頻繁に発生する**|手段②（Facade × Observer × Factory の複合）を採用|将来の「grep地獄」とテストの手戻りを防ぐ投資として見合うため。|
|**連携先が単一で固定されており、通知先も増える予定がない**|手段①（処理を別クラスや関数に分割するだけ）|複雑なインターフェースを導入するコストが回収できないため。|
|**一時的なデータ移行用の、使い捨てバッチツール**|起点コードのまま（すべて直書き）で良い|将来の保守や拡張を考える必要がないため。|

設計に絶対の正解はありません。しかし、私たちは武器の使い所を判断する基準を手に入れました。次はいよいよステップ7です。すべての部品を組み合わせ、変更に強い「外部連携システム」の全貌を完成させましょう。
