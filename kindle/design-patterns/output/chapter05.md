# 第5章 Commandパターン ―― 「命令」をオブジェクトとしてカプセル化する

## ステップ0：システムを把握し、仮説を立てる

**この章で扱うシステム：スマートホーム・コントローラー**

1つのリモコン（コントローラー）で、照明、エアコン、テレビなど様々な家電を操作するシステムです。家電の種類は今後も増える予定です。

**変動と不変の仮説**

| 分類 | 仮説 | 根拠 |
|---|---|---|
| 🔴 **変動する** | 操作対象のデバイス、操作の内容（ON/OFFなど） | 家電製品は次々と新製品が登場するため |
| 🟢 **不変** | 「ボタンを押したら何かを実行する」というコントローラーの骨格 | リモコンのボタン配置や基本的な操作フローは変わらないため |

---

## ステップ1：実装コードを読む ―― 責任の混在を見つける

### 5.1 現状のコード

```cpp
class RemoteController {
public:
    void pressButton(int slot, const std::string& action) {
        if (slot == 0) {
            light.power(action == "ON"); // Lightingクラスに依存
        } else if (slot == 1) {
            aircon.setTemp(25); // Airconクラスに依存
        }
    }
private:
    Lighting light;
    Aircon aircon;
};
```

**問題の発見：**
`RemoteController` が、個別の家電クラス（`Lighting`, `Aircon`）を直接知ってしまっています。新しい家電を追加するたびに、この `pressButton` メソッドを修正し、メンバ変数を増やす必要があります。

---

## ステップ2：仮説を確定する ―― 関係者ヒアリング

**関係者の声：**
- **製品企画：** 「来月にはロボット掃除機とスマートロックにも対応させたいです。」
- **ユーザー：** 「どのボタンにどの家電を割り当てるか、自由に設定したいです。」

---

## ステップ3：課題分析 ―― 変更が来たとき、どこが辛いか

新しい家電を追加するたびに、コントローラーの核心部分である `if-else` 文を書き換える必要があります。これは「ボタンを押す」という本来不変であるべき仕組みが、個別の家電という変動要因に汚染されている状態です。

---

## ステップ4：原因分析 ―― 困難の根本にある問題を言語化する

**真因：** 「命令を出す側（リモコン）」と「命令を受ける側（家電）」が、具象クラスレベルで強固に結合していること。

---

## ステップ5：対策案の検討 ―― 方向性を決め、手段を順に試す

### 5.5.1 方向性の特定

原因が「出す側と受ける側が直接繋がりすぎている」なら、解消する方向性は一つです。

**→ 「命令そのもの」を独立したデータ（オブジェクト）として切り出し、間接的に伝える。**

---

### 5.5.2 手段①：個別のメソッドに切り出す（直感的な案）

`if-else` の中身をメソッド化する方法です。

```cpp
void pressButton(int slot) {
    if (slot == 0) executeLight();
    else if (slot == 1) executeAircon();
}
```

**残る課題：**
メソッドに分けても見通しが良くなるだけで、**コントローラーが全ての家電を知っているという依存関係の本質**は変わりません。家電が増えればメソッドが増え、コントローラーの肥大化は止まりません。

---

### 5.5.3 手段②：命令をオブジェクトとしてカプセル化する（発想の転換）

第0章の手札：「振る舞いの内容と実行のタイミングを切り離す」→ **振る舞いのデータ化**。

**「『何をしろ』という命令を、一つのインターフェースを持つオブジェクトにする。」**

このように、要求をオブジェクトとしてカプセル化することで、出す側と受ける側を完全に分離する構造を **Command（コマンド）パターン** と呼びます。

```cpp
class ICommand {
public:
    virtual void execute() = 0;
};

class LightOnCommand : public ICommand {
public:
    void execute() override { light.power(true); }
private:
    Lighting light;
};
```

---

## ステップ6：天秤にかける ―― 手段①と手段②を比べる

### 5.6.1 評価軸の宣言

1. **拡張性**：新しい家電を追加するときに既存コード（コントローラー）を触らずに済むか
2. **柔軟性**：ボタンへの機能割り当てを動的に変更できるか

### 5.6.2 手段①vs手段②の比較

| 評価軸 | 手段①（メソッド化） | 手段②（Commandパターン） |
|---|---|---|
| 拡張性 | ❌ 家電追加のたびにコントローラー修正が必要 | ✅ 新しいCommandクラスを追加するだけでよい |
| 柔軟性 | ❌ コンパイル時に固定される | ✅ 実行時にCommandオブジェクトを差し替え可能 |

今回の状況では、家電の種類が今後も増え続け、ボタンのカスタマイズ要求もあるため、**手段②（Commandパターン）を採用します。**

---

### 5.6.3 耐久テスト ―― 「マクロボタン（一括操作）」の追加

「寝る前にすべての家電をOFFにする」というマクロボタンを追加したくなっても、既存のCommandをリストとして持つ `MacroCommand` を作るだけで、コントローラー側は一切変更せずに対応できます。

---

## ステップ7：決断と、手に入れた未来

### 5.7 解決後のコード（抜粋）

```cpp
class RemoteController {
public:
    void setCommand(int slot, ICommand* command) {
        commands[slot] = command;
    }
    void pressButton(int slot) {
        if (commands[slot]) commands[slot]->execute();
    }
private:
    std::map<int, ICommand*> commands;
};
```

### 振り返り：3つの哲学

- **哲学1（変わるものをカプセル化）**：各家電の操作をCommandクラスに閉じ込めた。
- **哲学2（インターフェースに対してプログラム）**：コントローラーは `ICommand` だけを知るようにした。
- **哲学3（コンポジション優先）**：コントローラーがCommandを「持つ」構造にし、柔軟性を得た。
