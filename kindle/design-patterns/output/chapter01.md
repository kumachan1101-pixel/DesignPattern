# 第1章 Strategyパターン ―― 変わるアルゴリズムをカプセル化して交換可能にする

## ステップ0：システムを把握し、仮説を立てる

**この章で扱うシステム：税計算モジュール**

商品の価格に対して消費税を計算し、請求書を発行するシステムです。国や地域、商品の種類によって税率や計算規則が異なります。

**変動と不変の仮説**

| 分類 | 仮説 | 根拠 |
|---|---|---|
| 🔴 **変動する** | 税率の計算ロジック（標準税率、軽減税率など） | 法改正や地域によって頻繁に変わるため |
| 🟢 **不変** | 請求金額の合計を算出する手順 | 合計 ＝ 小計 ＋ 税額 という基本構造は変わらないため |

---

## ステップ1：実装コードを読む ―― 責任の混在を見つける

### 1.1 現状のコード

```cpp
class Invoice {
public:
    double calculateTotal(double subtotal, std::string type) {
        double tax = 0;
        if (type == "Standard") {
            tax = subtotal * 0.10;
        } else if (type == "Reduced") {
            tax = subtotal * 0.08;
        }
        return subtotal + tax;
    }
};
```

**問題の発見：**
`Invoice`（請求書）クラスが、具体的な税率の「数値」や「計算式」を直接知ってしまっています。

---

## ステップ2：仮説を確定する ―― 関係者ヒアリング

**関係者の声：**
- **経理担当：** 「来年から食品の税率が変わる可能性があります。」
- **海外拠点：** 「輸出先によって独自の税計算ルールを適用したいです。」

---

## ステップ3：課題分析 ―― 変更が来たとき、どこが辛いか

新しい税率を追加するたびに、`Invoice` クラスの `calculateTotal` メソッドを修正し、`if-else` 文を増やす必要があります。

---

## ステップ4：原因分析 ―― 困難の根本にある問題を言語化する

**真因：** 「請求書の発行手順」と「税率の計算ロジック」という、変わる理由の異なる2つの知識が1つの場所に同居していること。

---

## ステップ5：対策案の検討 ―― 方向性を決め、手段を順に試す

### 1.5.1 方向性の特定

原因が「ロジックの混在」なら、解消する方向性は一つです。

**→ 変わる部分（税計算）を、不変の部分（請求書手順）から「切り出す」。**

---

### 1.5.2 手段①：関数として切り出す（直感的な案）

計算ロジックを別メソッドに分ける方法です。

```cpp
double Invoice::calculateTax(double subtotal, std::string type) {
    if (type == "Standard") return subtotal * 0.10;
    // ...
}
```

**残る課題：**
メソッドに分けても見通しが良くなるだけで、`Invoice` クラスが依然として「全ての税率」を知っているという依存関係は解消されません。

---

### 1.5.3 手段②：インターフェースとして切り出す（発想の転換）

第0章の手札：「変わるアルゴリズムをカプセル化する」→ **インターフェース抽出**。

**「税計算そのものを一つの『契約（インターフェース）』にして、外から差し込めるようにする。」**

このように、アルゴリズムをクラスとして独立させ、交換可能にする構造を **Strategy（ストラテジー）パターン** と呼びます。

---

## ステップ6：天秤にかける ―― 手段①と手段②を比べる

### 1.6.1 評価軸の宣言

1. **変更の局所性**：新しい税率を追加したときに、請求書クラスを修正せずに済むか
2. **テストの独立性**：税計算ロジックだけを単独でテストできるか

### 1.6.2 手段①vs手段②の比較

| 評価軸 | 手段①（メソッド化） | 手段②（Strategyパターン） |
|---|---|---|
| 変更の局所性 | ❌ Invoiceクラスの修正が必要 | ✅ 新しい税計算クラスを追加するだけ |
| テストの独立性 | ❌ 請求書全体を動かさないとテスト困難 | ✅ 税計算クラス単体で検証可能 |

**結論：将来の変化を見据え、手段②を採用します。**

---

## ステップ7：決断と、手に入れた未来

### 1.7 解決後のコード

```cpp
class ITaxStrategy {
public:
    virtual double calculate(double amount) = 0;
};

class Invoice {
    ITaxStrategy* strategy;
public:
    Invoice(ITaxStrategy* s) : strategy(s) {}
    double getTotal(double subtotal) {
        return subtotal + strategy->calculate(subtotal);
    }
};
```
