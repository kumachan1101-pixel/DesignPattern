# AIコンテキスト - 設計思考を鍛える本

## このファイルの位置づけ

このファイルは、章を生成・修正するAIが最初に読む入口である。
詳細ルールを重複して収録せず、次の正本を順に読む。

1. `CLAUDE.md`
2. `design-philosophy.md`
3. `shared/skills/author-voice.md`
4. `rules/writing-rules.md`
5. `rules/connection-point-review.md`

7. `templates/chapter-template.md`
8. `agents/chapter-agent.md`

## 本の目的

デザインパターン名を暗記することではなく、変更要求を受けたときに次を説明できるようにする。

1. 何が変わるのか。
2. 何を変えたくないのか。
3. 変更を試すとどこが痛むのか。
4. 変わる理由が異なる判断・条件・処理順序・生成判断がどこに混在しているのか。
5. 接続点で何を受け渡しているのか。
6. 呼び出し元へどの判断や前提が漏れているのか。
7. どの判断・条件・処理・生成責任をどの責任へ移すのか。
8. 導入コストに見合うか。
9. 変更要求を再度当てたとき、修正・再テスト範囲がどう変わるか。

## 7つのフェーズ

| フェーズ | 目的 |
|---|---|
| 1：現状把握 | 仕様、動作例、登場クラス、現状コード、変更要求を事実として読む |
| 2：仮説立案 | 変わりそうな部分、今回確実に変わること、将来リスクを分ける |
| 3：問題特定 | 変更要求を試し、修正・確認・再テスト範囲として痛みを確認する |
| 4：原因分析 | 混在する判断・条件・処理・生成判断と、接続点へ漏れている前提を言語化する |
| 5：課題定義 | 接続点で受け渡すもの、決定者、安定する約束、制約を定める |
| 6：対策検討 | 必要な分離案だけを比較し、移す判断・残る変更・導入コストを示す |
| 7：対策実施 | 実装・実行し、変更要求で効果と限界を確認する |

各フェーズは、前フェーズの出力を次フェーズの入力として使う。
入力、加工、出力の流れが見えにくい箇所は、文章だけで説明せず図または表で示す。

## 接続点レビュー

接続点を固定的な種類へ分類しない。次の問いを使う。

- 境界を越える値・型・操作・イベントは何か。
- その仕様を変更するのは誰か。
- 呼び出し元が相手のクラス名を知る必要はあるか。
- 呼び出し元が生成方法や処理順序を知る必要はあるか。
- 条件分岐や生成判断はどちらの責任か。
- 変更要求を当てると、どのファイルを修正・再テストするか。
- 呼び出し元へ残す最小限の約束は何か。

## 対策検討

必要なものだけを段階的に比較する。

1. 関数へ切り出す。
2. 別クラスへ責任を移す。
3. 複数実装を交換する場合、共通の契約を定める。
4. 複雑な手順を隠す場合、窓口や仲介役へ移す。
5. 動的に組み合わせる場合、コンポジションを使う。
6. 生成方法が変わる場合、生成責任を移す。

各案で、移す判断・処理・生成責任、接続点、減る変更、残る変更、導入コストを示す。

## パターン名の扱い

パターン名から設計を選ばない。
問題を解いた後、その構造をチームで共有する語彙として紹介する。

Strategy、Facade、Stateなどを一つの分類体系へ無理に当てはめない。
各パターンが解く問題と意図を、その章の変更要求から説明する。

## 執筆上の注意

- 既存コードを否定しない。
- 一つの正解を押しつけない。
- 架空の著者体験を書かない。
- 「変更がゼロ」「絶対に壊れない」と断定しない。
- 組み立て、登録、設定、テストに残る変更を隠さない。
- コードは実行可能にし、掲載した出力と照合する。
- AIへの指示文や引用マーカーを本文へ混入させない。
- `templates/chapter-template.md` の目的・達成基準・図解基準を章本文の起点にする。
