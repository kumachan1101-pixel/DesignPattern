# テストコード生成スキル

---

## このスキルの役割

章の各コード例に対して対応するテストコードを生成する。
テストは「動作確認」と「設計品質のセンサー」の両方として機能する。

---

## テストの種類と書き方

### 【試行コード】のテスト

目的：試行コードの「書きにくさ・重さ」を示す。
書き方：
- テストが書けないわけではないが、セットアップが重い様子を示す
- 「この1つの条件を確認するだけでも、これだけの準備が必要」という現実を見せる
- テスト後に「このテストの何が辛いか」を箇条書きで添える

```cpp
// 【試行コード】のテスト例
TEST(BillingProcessorTest, SomeCondition) {
    // セットアップが重い様子を示す
    // （DBモック、複数の依存関係のセットアップなど）
}
// → このテストの何が辛いか：
// ・〇〇を確認するだけでも△△の準備が必要
// ・□□が無関係なのに巻き込まれる
```

---

### 【解決コード】のテスト

目的：境界線が引けたことで、テストが軽くなったことを証明する。
必ず2種類に分けて書く：

#### ① コンテキスト側のテスト

```cpp
// ① コンテキスト側のテスト
// 「委譲が正しく行われるか」だけをモックで確認する
// 試行コードのテストで必要だった「〇〇の準備」が不要になった
TEST(ContextTest, DelegatesToStrategy) {
    // モックを用意して委譲されることを確認するだけ
    // DBも外部サービスも不要
}
```

#### ② ストラテジー側のテスト

```cpp
// ② ストラテジー側のテスト
// 「計算ロジック自体が正しいか」を純粋にテストする
// コンテキストの存在を知らなくてよい
TEST(StrategyTest, CalculatesCorrectly) {
    // 四則演算レベルのシンプルなテスト
}
```

テスト後に「テストの軽さが、境界を正しく引けた証拠」という一文を添える。

---

### 【深化コード】のテスト

目的：解決コードのテストと同じ構造になることを示す。
（同じ思考が転用できた証拠）

解決コードのテストと同じ2分割構造で書く。
書き方の説明に「解決コードのテストと構造が同じ」であることを一言添える。

---

## テストフレームワーク

- C++ の場合：GoogleTest（`#include <gtest/gtest.h>`）
- モックが必要な場合：GoogleMock（`#include <gmock/gmock.h>`）

## テストコードの品質ルール

- テスト名は「何を確認するか」が明確な英語にする
- 1テストで1つの事実を確認する（複数の EXPECT を乱用しない）
- テスト後のコメントで「なぜこのテストが軽い/重いか」を必ず一言添える
