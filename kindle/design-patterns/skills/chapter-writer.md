# 章を執筆する手順

## 正本

章構成は `templates/chapter-template.md` を正本とする。
第1章や過去章の見出しがテンプレートと食い違う場合は、テンプレートを優先する。
別のステップ番号や独自のフェーズ構成を作らない。

執筆前に次を読む。

1. `CLAUDE.md`
2. `design-philosophy.md`
3. `rules/writing-rules.md`
4. `rules/connection-point-review.md`
5. `rules/design-decision-guide.md`
6. `rules/phase-consistency-check.md`
7. `templates/chapter-template.md`

## 章作成の基本

- 各標準見出しの直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` を置く。
- 目的には「なぜこの項目を置くのか」を書く。
- 達成基準には「この項目を読み終えた読者が何を説明・予測・判断できる状態になっているか」を書く。
- 本文、図、表、コード、実行結果は、その目的と達成基準を満たすために置く。
- 図だけで新しい仕様や判断を初出ししない。図に出る名前・値・状態は、本文または表で説明してから使う。
- 各フェーズ単体の達成基準だけで合格にしない。仕様、動作例、クラス、コード、実行結果、変更要求、変更影響、最終コードが章全体で同じ対象を追っているか確認する。
- フェーズ間整合は `rules/phase-consistency-check.md` を正本とする。
- 仕様は、コードに出る値・状態・判定条件・出力名へ紐づく粒度まで定義する。
- 掲載コードで `print`、固定データ、スタブ、簡略化した `main()` を使う場合、実システムでは何に相当するかを図または表で示す。
- 論点から外す処理は、実際はどう動くか、掲載コードでは何で代替するか、なぜ割愛するか、設計論点への影響がないかを補足する。

## 7つのフェーズ

### フェーズ1：現状把握

標準見出し：

1. `1-1：このシステムの仕様`
2. `1-2：動作例テーブル`
3. `1-3：登場クラスとクラス構成図`
4. `1-4：実装コード（現状）`
5. `1-5：変更要求`

事実だけを確認する。原因分析、設計上の痛み、解決策を先取りしない。
仕様は、入力、加工、出力の構造図または表で示す。
実システムの入出力を簡略化する場合は、省略範囲と代替表現を示す。
クラス構成図の前に、登場クラスの責任説明を置く。

### フェーズ2：仮説立案

フェーズ1の事実から、変わりそうな部分を仮説として挙げる。
今回確実に変わること、将来リスク、当面安定していることを混ぜない。
関係者ヒアリングは、仮説を確認するために置く。

### フェーズ3：問題特定

変更要求を現状コードへ当てる。
痛みを感想で語らず、修正箇所、確認範囲、再テスト範囲として示す。
変更影響はグラフで可視化し、直接変更と巻き込まれた変更を区別する。

### フェーズ4：原因分析

変更が広がった理由を、依存、分岐、責任の混在として説明する。
「知識」は曖昧に使わず、判定条件、計算式、処理順序、生成判断、クラス名、型、操作名などに分解して書く。
呼び出し元が相手の何を知りすぎているかを明示する。

### フェーズ5：課題定義

原因分析から、解くべき接続点を定義する。
境界を越える値・型・操作・イベント、変化の種類、安定する約束、非機能制約、クライアント影響を整理する。
「何をどこから切り離し、接続点には何を残すか」を課題文として書ける状態にする。

### フェーズ6：対策検討

見出しはテンプレートに合わせる。
冒頭に、フェーズ1の現行コード全体を振り返りとして一度だけ示し、その直後にフェーズ3で変更した関連部分を痛みの差分として置く。この一組を対策の起点にする。
フェーズ5の課題を課題IDごとに分け、根拠・着目コード・移すもの・残す約束・制約・完了条件を対応づける。
各課題では、関連部分の変更前コードと変更後コードを使い、「なぜこの変更か」「何が減ったか」「何が残ったか」「なぜ次へ進むか」を段階ごとに説明する。採用後の全体コードはフェーズ7へ置き、フェーズ6で完成形を先出ししない。
クラス分離案では、必要になった段階で生成場所・所有者・依存注入・利用側の呼び出しを関連コードに含める。
関数化、責任移動、契約導入、窓口・仲介、コンポジション、生成分離から、必要な案だけを比較する。
各案で、移す判断・処理・生成責任、接続点、減る変更、残る変更、導入コストを示す。
採用する形は、今回の文脈で受け入れるコストと合わせて決める。

### フェーズ7：対策実施

最終コードでは、内部数値の変更を対象ID・変更前→変更後・上限/単位付きでログへ出す。システムが続けるべき昇格・通知・補償・履歴記録は、起点ユースケースから自動で呼び、`main()` の手動操作で成立させない。状態不変のエラーは状態遷移図へ描かず、エラー条件表と実行結果で扱う。

標準見出し：

1. `7-1：解決後のコード（全体）`
2. `7-2：動作シーケンス図`
3. `7-3：変更影響グラフ（改善後）`
4. `7-4：変更シナリオ表`

掲載コードを実行し、動作例と照合する。
組み立て、登録、設定に残る変更も記載する。
改善前後の変更影響を図または表で比較する。

## 後段

順序は次のとおり。

1. 整理
2. 振り返り
3. あなたのコードで考える
4. パターン解説

パターン名は、問題を解いた構造の共有語彙として説明する。
