# skills/chapter-writer.md
# 章を執筆する手順スキル

---

## このスキルの役割

`patterns/*.yaml` の定義と `templates/chapter-template.md` を組み合わせて
1章分のMarkdownを生成する手順を定義する。
対象は第1〜8章。第0章には `templates/chapter0-template.md` を使うこと。

> **注意：章の構造（何をどの節に書くか）の唯一の正は `templates/chapter-template.md`。**
> このファイルはその補完説明（品質基準・ニュアンス・生成手順）を提供する。
> 両者に差異がある場合は `chapter-template.md` を優先すること。

このスキルを使う前に以下を読んでいること：
- `../../shared/skills/author-voice.md`
- `../../shared/personas/reader-profiles.md`
- `CLAUDE.md`
- `templates/chapter-template.md`（v3）

## 分量と詳しさの基準（重要）

他章との品質・ボリュームの統一を図るため、各章は **全体で約1000〜1500行程度** のしっかりとした分量になるように執筆すること。
単に説明を長くするのではなく、以下の点に注力して「詳しさ」を担保すること：
1. **実装コードの充実**：変更前・変更後のコードは省略せず、実用的な文脈（DIの組み立て、`main`関数、テストコード等）を必ず含める。
   - **【コードブロックの分割ルール】**：1つのコードブロックに複数の話題（NG例とOK例、選択肢①と②など）を詰め込んではならない。必ず状態や話題ごとにコードブロックを分割し、直前に説明の散文を置くこと。
2. **関係者ヒアリングの具体性**：ステップ2での会話はリアルに、なぜその決断に至ったのかの背景を厚く描く。
3. **課題分析と原因分析の深掘り**：表面的な事実だけでなく「なぜその構造が辛いのか」を読者が共感できるレベルで詳述する。
4. **代替案との比較**：「一見よさそうな代替案」も具体的なコードを示し、なぜそれが破綻するのかを丁寧に解説する。

## 生成手順

### ステップ1：pattern_file から素材を取り出す

以下の情報を確認する：
- `scenario.domain` : システムのドメイン
- `scenario.system_overview` : システムの概要
- `scenario.current_spec` : 現在の仕様
- `scenario.situation` : 変更要求（誰から・何の要求・いつまでに）
- `observations` : 観察リスト（ステップ3 課題分析の素材）
- `advanced_scenario` : より難しい変化（ステップ6 耐久テストの素材）
- `threshold` : 使う・使わない状況（ステップ6 使い分けの素材）

---

### ステップ2：各節を順番に生成する

#### 冒頭：この章を読むと得られること

**ステップ0より前に必ず書く（`> **この章の核心**` ブロックの直後）。**

`## この章を読むと得られること` というH2セクションとして配置する。

書き方の原則：
- 「〜できるようになる」または「〜がわかるようになる」形式で3〜4項目
- パターン名を前面に出さない（「〇〇パターンが書けるようになる」はNG）
- 必ず含める視点：
  1. このパターンが解決する問題を自分のコードで発見できる目
  2. 使うべき場面・使わない場面を判断できる目
  3. 変更要求が来たとき影響範囲を事前に読む目

また、書き始める前に CLAUDE.md のドメイン割り当て表を確認し、この章のドメインが割り当て済みのものと一致していることを確認する。

#### X.0 この章のシステム構成と仮説

**入力：** システムのシナリオ説明 ＋ クラス構成の概要。実装コードはまだ読まない。
**産物：** 変動と不変の「仮説テーブル」

4層で書く：
- **第1層**：システムの概要（2〜3文。業務用語を定義）
- **第2層**：仕様表（機能・担当・入力・出力）
- **第3層**：クラス構成の概要（mermaid classDiagram。変更前の問題の構造を可視化する）
- **第4層**：各クラスの責任一覧 ＋ 仮説テーブル（変動/不変の予測と根拠）

当時の担当者への敬意を忘れない。
「このコードが今日まで現場を支えてきた事実」に触れる。

#### X.1 実装コードと責任チェック

**入力：** ステップ0で把握したクラス責任 ＋ 実際の実装コード
**産物：** 責任チェック表。「このクラスが持つべきでない知識」が混在している行の発見。

- `scenario.situation` の内容から起点コードを示す（main()含む）
- コメントで「なぜこう書いたかの背景」を示す
- 実行結果を示す（「このコードは正しく動く。問題は構造にある」を明示する）
- 責任チェック表：コードの行 ｜ 持っている知識 ｜ 責任内か（✅または✗）

#### X.2 届いた変更要求

- `scenario.situation` の内容を人間的な文体で書く
- 「誰から・何の要求が・いつまでに」の形式
- 著者の口癖を1箇所入れる
- 読者が「自分がこの担当者だったら」と感じられる具体性

#### X.3 仮説の検証と変動/不変の確定

**入力：** ステップ0の仮説 × ステップ1の責任チェック結果。関係者に直接確認する。
**産物：** 確定した変動/不変テーブル（「誰の判断で変わるか」明記）

- 「コードを読んだだけで断定するのは危険」という前置きを必ず入れる
- 関係者ヒアリングを問答形式（開発者の問いかけ → 関係者の回答）で示す
- 確定テーブル：分類 ｜ 具体的な内容 ｜ 変わるタイミング ｜ 根拠
- 「🟢不変を契約（インターフェース）として固定し、🔴変動をその裏側に押し込む」という設計の決断を宣言して締める

#### ステップ3：課題分析

- 変更要求を「素朴に実装しようとすると何が起きるか」を描く
- 「痛み」を2点、散文で言語化する
- 変更が飛び火する様子をmermaidで図示する

#### ステップ4：原因分析

- 観察 → 原因の方向 の表を示す
- 「変わり続けるもの ｜ 変わってほしくないもの」の2列表で仕分ける
- 哲学1（変わるものをカプセル化せよ）との直接的な繋がりを散文で説明する
- 「立ち止まって考える」コールアウトを置く

#### ステップ5：対策案の検討

この節の核心：ステップ4で特定した**真因から「方向性」を導き、段階的に解決へと近づく思考プロセス**を示す。「いきなり正解の手札を出す」のではなく、自然な試行錯誤（手段①→課題発見→手段②）として描く。

**方向性の特定**
- 原因の種類（変わるもの/変わらないものの混在、依存の分散等）から、「分ける」「集める」「隠す」などの大まかな解消方向を導く。

**手段①：最初に思いつくシンプルな案**
- 直感的な案（関数の切り出し等）を提示し、コードを示す。
- **残る課題**：その手段では解消しきれない根本的な問題を2点ほど明示する。これが手段②への強力な動機付けとなる。

**手段②：本質的な解決案（発想の転換）**
- 手段①の課題を受けて、第0章の手札選択表を引用し、本質的な手札（Strategy, State等）を提示する。
- 提示する際は「第0章の手札選択表を引くと…」の定型句を必ず使う。
- コード（インターフェース + 実装クラス）とクラス図を示す。

#### ステップ6：天秤にかける

この節の核心：手段②が本当に優れているかを、**事前に定義した評価軸**で検証し、将来の変化への耐久性を実証する。

**評価軸の宣言**
- 比較より前に「この状況で何を重視するか」を表形式で宣言する。

**手段①vs手段②の比較**
- 宣言した評価軸に基づき、両方の案を比較した表を作成する。
- 比較結果から、手段②（今回のパターン）の採用を正式に決定する。

**耐久テスト**
- ステップ2のヒアリングで挙がった「将来の変化」を実際にコードで試す。
- 採用した構造がいかに変更を局所化できるかを読者に納得させる。

**使う場面・使わない場面**
- 【過剰コード】：変化の予定がないところに適用して複雑になった例を示す。
- 状況ごとの選択指針を表で示す（状況 ｜ 適切な選択 ｜ 理由）。
- **最小コスト代替案との比較**：パターンを使わずに済む場合も示し、判断基準を説明する。

#### ステップ7：決断と、手に入れた未来

- BatchApplication / main() を含む最終コード全体を示す
- 変更シナリオ表（シナリオ ｜ 変わるクラス ｜ 変わらないクラス）
- 最終責任テーブル（クラス ｜ 責任（1文） ｜ 変わる理由）

#### 整理

- 8ステップとこの章でやったことの対応表（ステップ0〜7 ｜ この章でやったこと）
- 各クラスの最終的な責任（クラス ｜ 責任 ｜ 変わる理由）
- 「このプロセスを回した結果にたどり着いた構造こそが【パターン名】パターンです」で締める

#### 振り返り：第0章の3つの哲学はどう適用されたか

- **哲学1「変わるものをカプセル化せよ」の現れ**：どのクラス・どの構造に現れているかを具体化した場所を示してから散文で説明
- **哲学2「実装ではなくインターフェースに対してプログラムせよ」の現れ**：同様
- **哲学3「継承よりコンポジションを優先せよ」の現れ**：同様

各哲学の説明は箇条書きにせず、散文で書く。

Markdownの構文崩れ（階層構造やリストの不整合）を修正し、執筆テンプレートとしてそのまま使える形に整理しました。

## パターン解説：〇〇パターン

### パターンの骨格

このパターンが解決する構造上の問題を、ここに1〜2文で記述します。

mermaid
```
classDiagram
    class Context {
    }
    class ITarget {
        <<interface>>
        +request()*
    }
    class ConcreteImplA {
        +request()
    }
    class ConcreteImplB {
        +request()
    }
    Context --> ITarget
    ITarget <|.. ConcreteImplA
    ITarget <|.. ConcreteImplB
```

- **Context**: パターンの利用者であり、抽象的なインターフェースを通じて処理を呼び出します。
    
- **ITarget**: 具象クラスが実装すべき共通の振る舞いを定義するインターフェースです。
    
- **ConcreteImplA/B**: インターフェースを具体的に実装し、それぞれの固有の振る舞いを提供します。

### この章の実装との対応

本章で作成したクラス（またはモジュール）と、GoFの抽象ロールを対応させた図です。

mermaid

```
classDiagram
    class [章のContextクラス] {
    }
    class [章のInterface] {
        <<interface>>
    }
    class [章の具象クラスA] {
    }
    [章のContextクラス] ..> [章のInterface]
    [章のInterface] <|.. [章の具象クラスA]
```

抽象ロールとしてのインターフェースと、実際に動く具象クラスがどのように対応しているかを、ここで1〜2文で説明します。

---

### どんな構造問題を解くか

「処理の骨格」と「処理の中身」が同じ場所に混在している状態こそが、このパターンの出番です。全体の手続きの流れを変えずに、詳細なステップだけを差し替えたい場合に、この分離が威力を発揮します。

---

### 使いどころと限界

- **使いどころ**: 「いつ、どの順序で実行するか」という制御ロジックは不変だが、「具体的に何をするか」が環境や条件によって変わる場合に使用します。「誰の判断で振る舞いが変わるか」という観点で設計を検討するのがよいでしょう。
    
- **限界**: 共通化した「骨格」側に頻繁な変更が予想される場合は注意が必要です。抽象化のコストと、将来の変更頻度のバランスを考慮し、あえてパターンを適用しないのも一つの重要な判断です。

### ステップ3：生成後の確認（セルフチェックリスト）

※出力時にはこのセクションの内容を元に、原稿の整合性を確認してください。

#### 1. 構造チェック（節の存在・順序）

- [ ] **author-voice.md** の口癖（「設計に絶対の正解はない」「チームで決める話」等）が3〜5箇所、自然に含まれているか。
    
- [ ] パターン名が「X.1 実装コード」より前に出現していないか（試み②の後に初登場しているか）。
    
- [ ] 評価軸が「X.5 のテスト比較」より前に宣言されているか。
    
- [ ] 比較まとめ表（縦：基準、横：試み①/②）が正しく配置されているか。
    
- [ ] 整理セクションに「8ステップ対応表」と「最終責任テーブル」の両方があるか。
    
- [ ] パターン解説に「GoF抽象クラス図」と「章固有マッピング図」の両方があるか。
    

#### 2. 内容整合チェック

- [ ] ヒアリング（X.3）の変更リスクが、試みのインターフェース設計（X.5）に反映されているか。
    
    - 例：「型が変わりうる」リスクに対し、値オブジェクトを採用したか。
        
- [ ] 将来の変化として挙げたものが、耐久テスト（X.6）で実際に検証されているか。
    
- [ ] 仮説テーブル（X.0）が、ヒアリング確定テーブル（X.3）で根拠付きで更新されているか。
    
- [ ] 試み①の限界が、試み②でどう解消されたか明示されているか。
    

#### 3. 設計判断の明示

- [ ] 型変更リスクがある引数の「値オブジェクト化」の判断理由を明示しているか。
    
- [ ] 試み①を「失敗」ではなく「限界のある有効な選択肢」として扱っているか。
    
- [ ] 「正解はない」という前置きの後に、「今回の状況ではこう判断した」という結論が続いているか。
