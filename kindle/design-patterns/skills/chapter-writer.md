# 章を執筆する手順

## 正本

章構成は第1章と`templates/chapter-template.md`を正本とする。
別のステップ番号や独自のフェーズ構成を作らない。

執筆前に次を読む。

1. `CLAUDE.md`
2. `design-philosophy.md`
3. `rules/writing-rules.md`
4. `rules/connection-point-review.md`
5. `rules/design-decision-guide.md`
6. `templates/chapter-template.md`

## 7つのフェーズ

### フェーズ1：現状把握

標準見出し：

1. このシステムの仕様
2. 動作例テーブル
3. 実装コード（現状）
4. クラス構成図
5. 変更要求

事実だけを確認する。補足の依存グラフや実行結果は、標準見出しの配下に置く。

### フェーズ2：仮説立案

標準見出し：

1. 責任チェック表
2. 変わる理由の分析
3. 今回の変更で確実に変わること
4. 関係者ヒアリング
5. ヒアリングで判明した将来リスク

仮説、確定変更、将来リスクを混ぜない。

### フェーズ3：問題特定

標準見出し：

1. 変更を試みる
2. 変更影響グラフ
3. 痛みの言語化

痛みを先に断定せず、変更した結果として修正・再テスト範囲を示す。

### フェーズ4：原因分析

標準見出し：

1. 痛みの根源を探る（観察と原因）
2. 変わるもの/変わってほしくないもの
3. 接続点に漏れている知識を確認する

クラス名、生成方法、処理順序、条件のうち、呼び出し元が知りすぎているものを示す。
固定的な接続分類へ当てはめない。

### フェーズ5：課題定義

見出しは「接続点で何が流れているかを見る」とする。

- 値・型・操作・イベント
- 変更の決定者
- 安定する約束
- 移すべき知識
- 非機能制約

を整理し、課題を確定する。

### フェーズ6：対策検討

見出しは「段階的な改善と決断」とする。

関数化、責任移動、契約導入、窓口・仲介、コンポジション、生成分離から必要な案だけを比較する。
各案で「移す知識」「減る変更」「残る変更」「導入コスト」を示す。

最後に「どこまで設計を進めるのが良いか（採用ステップの決断）」を置く。

### フェーズ7：対策実施

標準見出し：

1. 解決後のコード（全体）
2. 動作シーケンス図
3. 変更影響グラフ（改善後）
4. 変更シナリオ表

掲載コードを実行し、動作例と照合する。組み立て・登録・設定に残る変更も記載する。

## 後段

順序は次のとおり。

1. 整理
2. 振り返り
3. パターン解説

パターン名は問題を解いた構造の共有語彙として説明する。
