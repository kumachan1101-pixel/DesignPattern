# skills/chapter-writer.md
# 章を執筆する手順スキル

---

## このスキルの役割

`patterns/*.yaml` の定義と `templates/chapter-template.md` を組み合わせて
1章分のMarkdownを生成する手順を定義する。

---

## 前提：必ず読んでいること

このスキルを使う前に以下を読んでいること：
- `../../shared/skills/author-voice.md`
- `../../shared/personas/reader-profiles.md`
- `CLAUDE.md`

---

## 5ステップの構造

この章テンプレートは以下の5ステップで構成される。
各ステップがH2見出しとして章の大項目になる。

| ステップ | H2見出し | 含まれる節 |
|:---|:---|:---|
| 1 | 現状把握 | 〇.1 仕様・コード、〇.2 変更要求 |
| 2 | 課題の発見 | 〇.3 変更しようとした時の困難 |
| 3 | 原因特定 | 〇.4 困難の根本原因 |
| 4 | 対策案の検討 | 〇.5 原因から論理的に導く、〇.6 各アプローチの実装 |
| 5 | 天秤・決断 | 〇.7 基準の宣言、〇.8 テスト比較、〇.9 深化、〇.10 使い分け |
| — | 整理 | 〇.11 振り返りと次章予告 |

---

## 生成手順

### ステップ1：pattern_file から素材を取り出す

以下の情報を確認する：
- `scenario.domain` : システムのドメイン
- `scenario.system_overview` : システムの概要
- `scenario.current_spec` : 現在の仕様
- `scenario.situation` : 変更要求（誰から・何の要求・いつまでに）
- `observations` : 観察リスト（〇.3 課題の発見の素材）
- `advanced_scenario` : より難しい変化（〇.9 深化の素材）
- `threshold` : 使う・使わない状況（〇.10 の素材）

---

### ステップ2：各節を順番に生成する

#### 〇.1 今のシステムの仕様とコードの構造

3層で書く：
- **第1層**：システムの概要（2〜3文。業務用語を定義）
- **第2層**：現在の仕様（箇条書き。テストの期待値の出どころ）
- **第3層**：【起点コード】（コメント＝「なぜこう書いたかの背景」）

当時の担当者への敬意を忘れない。
「このコードが今日まで現場を支えてきた事実」に触れる。

#### 〇.2 届いた変更要求

- `scenario.situation` の内容を人間的な文体で書く
- 「誰から・何の要求が・いつまでに」の形式
- 著者の口癖を1箇所入れる
- 読者が「自分がこの担当者だったら」と感じられる具体性

#### 〇.3 変更しようとしたときに現れる困難

- 変更要求を「素朴に実装しようとすると何が起きるか」を描く
- `observations` リストを素材として使う
- 困難を2点、箇条書きで言語化する
- 「なぜこんなに難しいのか？」という問いを引用ブロックで置く
- 答えは出さない。「難しい事実を確認する」だけ

#### 〇.4 困難の根本にあるもの

以下の切り口から1つを選んで展開する。
**選んだ切り口以外のコメントブロックは削除すること。**

**切り口A（変化の混在）**：Strategy / Observer / Template Method / Command / Decorator
- 「変わり続けるもの」と「変わってほしくないもの」が同じ場所にある
- 表で仕分けを示す（2列：変わるもの ｜ 変わらないもの）

**切り口B（依存の過多）**：Facade
- 使う側が「知らなくていいこと」まで知っている
- 変化が1か所で起きても、複数の場所に波及する

**切り口C（状態と振る舞いの混在）**：State
- 「状態が何か」と「そのときに何をするか」が同じ場所にある
- 状態が増えるたびに振る舞いの記述も増殖する

**切り口D（生成と利用の混在）**：Factory Method
- 「何を作るか」の判断と「どう使うか」の処理が同じ場所にある

どの切り口でも、最後に「立ち止まって考える」コールアウトを置く。
「もし～を解消できたら、何が変わるか」を具体的に書く。

#### 〇.5 原因から対策の方向性を論理的に導く

**この節の核心：アプローチは「原因への正当な対処」として複数導出する。**

- 〇.4で特定した原因を起点として書く
- 「この原因を解消するには、こういうアプローチが考えられる」という論理的な導出
- 大きくN個のアプローチを示す（通常2個）
- 著者の口癖を1箇所入れる

**各アプローチの書き方**：

```
アプローチA：[名前]
[原因 → なぜこれが対処になるか、の論理を1〜2文で。]
コスト：[実装の手軽さ・学習コスト]
適する状況：[どんな条件のときにこれが合うか]

アプローチB：[名前]
[原因 → なぜこれが対処になるか、の論理を1〜2文で。]
コスト：[実装の手軽さ・学習コスト]
適する状況：[どんな条件のときにこれが合うか]
```

最後に「いずれのアプローチも論理的に正しい対処です。
最善かはステップ5の天秤で判断します。」という一文で締める。

**注意**：アプローチAを「失敗する試み」として扱ってはいけない。
どちらも有効で、適する状況が異なるという提示にする。

#### 〇.6 各アプローチの実装

- 各アプローチのコードを順番に示す
- アプローチAのコード → アプローチAで解消されること・残る課題
- アプローチBのコード → アプローチBで解消されること・残る課題
- アプローチBにはMermaidクラス図＋テキスト補足を必ず添える
- 著者の口癖を1箇所入れる
- **★アプローチBの後、ここで初めてパターン名を出してよい★**

コードラベルは各アプローチを区別できる形にする。
複数になる場合は章に応じた説明的な名前を使う。

#### 〇.7 比較の基準を先に宣言する

- 比較を始める前に「何を重視するか」を表で明示する
- 表形式（2列：判断の基準 ｜ なぜこの状況で重要か）
- 〇.2の変更要求・〇.3の困難と結びついた理由を書く
- 基準の例：テストの独立性 / 変更の局所性 / チームの分担 / 変化の継続性
- パターンに応じて最も意味のある2〜4個を選ぶ（全部列挙しない）

#### 〇.8 各アプローチをテストで比較する

- 〇.7で宣言した基準を使って、アプローチAとBをテストで比較する
- アプローチAのテスト：「〇〇の基準に照らすと」という見出し
- アプローチBのテスト：同じ基準で何が変わったかが伝わるように書く
- 比較表（縦：基準、横：アプローチA/B、セルに○△×）で締める
- 「この比較はあくまで今回の状況と基準に対するもの」という注記を入れる

テストコードのルール：
- テストは1コードセットが完結してから書く（後回し禁止）
- 各テストに「何を確認しているか」がわかるコメントを入れる

#### 〇.9 より難しい変化への耐久テスト

- `advanced_scenario` の内容を使う
- 同じドメイン・同じシステムへの、より難しい変更要求
- 「ステップ1〜4の思考は使えるか？」という問いかけ
- アプローチAとBがそれぞれこの変化にどう対応するかを示す
  - 「Aは〇〇すれば対応できる」「Bは拡張するだけでよい」という形で
  - AがBに対して明確に劣ることで、Bを選ぶ根拠が確かになる
- 【深化コード】：アプローチBのどこをどう拡張したかをコメントで明示
- 【深化コード】のテストをその場で示す
- 「この耐久テストを経て、この状況ではアプローチBが合っていると判断できる」で締める

#### 〇.10 使う場面・使わない場面

- 「では、アプローチBを常に選べばいいのか？」という問いでつなぐ
- 著者の口癖を1箇所入れる
- 【過剰コード】：アプローチBを変化しない箇所に使いすぎた例
- 「このケースではアプローチAの方が合っていた」という評価を入れる
- 状況ごとの選択指針（3行の表：状況 ｜ 適切な選択 ｜ 理由）
- この現場での選択肢（4列の表：状況 ｜ 選ぶ形 ｜ 省略するもの ｜ 次の判断タイミング）
- アプローチA（最小採用）のコードスニペット
- 「どの形を選んでも守る一線」コールアウト

#### 〇.11 この章で踏んだ思考の整理

- 5ステップとこの章でやったことの対応表
  - ステップ名は「1.現状把握 / 2.課題の発見 / 3.原因特定 / 4.対策案検討 / 5.天秤・決断」
- パターン名の位置づけ（辿り着いたラベル）の一文
- 次章予告（「次章では〇〇という変化を扱います」の一文のみ）

---

### ステップ3：生成後の確認

以下を確認してから出力する。

- author-voice.md の口癖が3〜5箇所、不自然でなく入っているか
- パターン名が〇.6より前に出ていないか
- 〇.5のアプローチが「原因から導いた論理的な対処」として書かれているか
- 〇.5のアプローチAが「失敗する試み」になっていないか（両方が有効な提示か）
- 〇.7の基準が〇.8の比較より前に宣言されているか
- 〇.8に比較まとめ表があるか
- 執筆指示コメント（`<!-- -->`）が出力に残っていないか
- 他章への言及がないか（「前章で」「次章と同様に」禁止）
- 〇.4の切り口選択肢コメントが削除されているか
