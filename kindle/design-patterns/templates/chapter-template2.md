# 【第二部 応用編】執筆テンプレート（第一部からの差分）

第二部（第9〜11章）は、「1つのシステムに複数の『変わる理由』が混在している現実の複雑さ」に向き合います。第一部のテンプレート「chapter-template.md」の構造（ステップ0〜7）は1文字も崩さず、以下の差分を適用して執筆します。

## 1. タイトル〜導入部の差分

### タイトルと核心

- **タイトル形式:** `第X章　【パターンA】 × 【パターンB】 × 【パターンC】`
    
- **この章の核心:** 1つの問題ではなく、「なぜAとBは別の理由で変わるのか。それを1箇所に押し込むとどうなるか」という複数要因の同居による痛みを1〜2行で示します。
    

### レゴブロックで考える

- レゴブロックのメタファーも2つの操作の組み合わせになります。
    
- （例）「今回は、赤いブロック（割引ルール）を『分離』する操作と、青いブロック（状態）を『モノ化』する操作を組み合わせます」
    

## 2. ステップ0〜4の差分（複数要因の発見と分析）

### ステップ0：仮説を立てる

- **仮説テーブル:** 🔴（変動する）項目が必ず2系統（2つの異なる理由）存在するように記載します。
    

### ステップ1：責任チェック

- **責任チェック表:** 起点コードの中に、異なる2つの知識（管轄違い）が混在していることを示します。「△ / ✗ 営業部門の管轄」「△ / ✗ 業務部門の管轄」のように、誰の責任かが2つに割れている事実を明記します。
    

### ステップ2：関係者ヒアリング

- **ヒアリングの会話:** 営業担当者と業務担当者など、**立場の違う2人の関係者**との対話シーンを描き、それぞれから別々の変更要求（将来のリスク）が来ることを示します。
    

### ステップ3：課題分析（変更影響グラフ）

- **変更影響グラフ:** 異なる2系統の変更要求（T1, T2）が、同じ1つのクラス（A）に飛び火する様子を描きます。
    
    コード スニペット
    
    ```
    graph LR
        T1["割引ルールが変わった"]
        T2["状態遷移のルールが変わった"]
        A["OrderService.cpp\n（同一ファイル）"]
    
        T1 -->|"影響が飛び火"| A
        T2 -->|"影響が飛び火"| A
    ```
    

### ステップ4：原因分析

- **原因の定義:** 「原因が1つ」ではなく、「原因A（要因1の混在）」と「原因B（要因2の散在）」の**2つの根本原因**を言語化します。
    

## 3. ステップ5の差分（手札の段階的適用）

第一部では「手段①（手続き的）→手段②（オブジェクト指向的）」という進化の過程を描きましたが、応用編では「原因Aの解消 → 原因Bの解消」という原因ごとの手札の適用を描きます。

### X.5.1 手札の適用①：【要因A】を切り出す

- 原因Aに対する「手札（分離・隠蔽・規格化など）」を適用したコード（一部）を示します。
    
- **中間の責任チェック:** これにより要因Aの問題は消えますが、「要因B（例：状態ごとのif文）は依然として残っている」ことを散文と表で正直に示します。
    

### X.5.2 手札の適用②：【要因B】を切り出す

- 残った原因Bに対する次の「手札」を適用します。
    
- ここで初めて、2つのパターンが融合した最終的な構造のコードが姿を現します。
    
- **改善後のクラス図:** 主要クラスが、2つの異なるインターフェース（または部品）をコンポジションとして持っている様子を描画します。
    

## 4. ステップ6〜7の差分（複数軸の検証）

### ステップ6：耐久テストと評価

- **耐久テスト:** ステップ2で2人の関係者から予告された2つのリスク（変更要求）を両方ともシミュレートします。
    
- 新しいルールを追加するコード、新しい状態を追加するコードを提示し、**どちらの軸の変更が来ても既存コード（主要クラス）に一切触れずに済む**ことを実証します。
    

### ステップ7：変更シナリオ表と影響グラフ

- **変更シナリオ表:** 「要因Aが変わったとき」「要因Bが変わったとき」のシナリオを併記し、それぞれ独立したクラスだけが変わる（互いに干渉しない）ことを示します。

## 5. 整理・パターン解説の差分

### 第一部との違い（明記必須）

- 「2つのパターンを組み合わせようと最初から決めていたわけではない。2つの根本原因を8ステップで分析し、一つずつ手札を切った結果として、自然と2つのパターンが組み合わさった」という**思考のプロセス**を読者に強調して伝えます。
    

### パターン解説

- 【パターンA】【パターンB】【パターンC】の骨格を解説します。
    
- **パターンの対比:** 単なる解説にとどまらず、「UMLの構造は似ているが、決定的な違い（差し替えの主体など）はここにある」といった、組み合わせたからこそ分かる実践的な比較論を必ず含めます。
