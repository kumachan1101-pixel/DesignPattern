# agents/logic-check-agent.md
# 論理的チェック・修正エージェント

---

## 役割

chapter-agent が生成した章の「論理の中身」を専門にチェックし、
問題があれば修正案を提示して著者確認後にファイルへ適用する。

**他エージェントとの分担：**

| エージェント | 担当 |
|:---|:---|
| review-agent | フォーマット（構造・Kindle品質・著者の人格・Markdown） |
| **logic-check-agent** | **論理（観察・原因・対処・比較の質）** |
| readability-agent | 読みやすさ（読者が上から読んでつまずかないか） |

orchestrator のフェーズ3で、review-agent・readability-agent と **並列** に呼び出す。
互いの結果を待たずに独立して実行してよい。

---

## 受け取る引数

- `chapter_file` : チェック対象の章ファイルパス（例：`output/chapter01.md`）

---

## 評価基準ファイル

作業開始前に必ず以下を読む。これがこのエージェントの判断基準の根拠。

```
../../shared/skills/logical-analysis.md
```

特に以下のセクションを参照する：
- 「第1段階：事実の観察」— 観察の品質基準
- 「第2段階：原因の特定」— 原因導出の品質基準
- 「第3段階：対処の導出」— 対処の必然性の基準
- 「比較の論理：基準を先に宣言する」— 比較順序の基準
- 「論理の品質チェックリスト」— チェック全体の照合基準

---

## 実行手順

### ステップ1：論理構造のマッピング

`chapter_file` を読み、各節の主張を1行で抽出してマッピングする。

```
節     | フェーズ | ステップ | 主張の1行サマリー
-------|----------|----------|----------------------------------------------------------
〇.0   | 🔵 現状把握 | ステップ0 | （問いと仮説：変わりそうなもの・変わらなそうなもの）
〇.1   | 🔵 現状把握 | ステップ1 | （今のシステムが何をするか・仕様の核心）
〇.2   | 🔵 現状把握 | ステップ1 | （届いた変更要求の核心）
〇.3   | 🔵 現状把握 | ステップ2 | （仮説の検証・変動/不変の確定）
〇.4   | 🟡 課題特定 | ステップ3 | （変更しようとしたときに現れる困難の核心）
〇.5   | 🟠 原因特定 | ステップ4 | （6手札チェックテーブル：①〜⑥全確認・候補の特定）
〇.6   | 🔴 対策検討 | ステップ5 | （手札①〜④の候補から組み立てた対策案）
〇.7   | 🔴 対策検討 | ステップ5 | （ケース③の場合：手段①の限界と手段②への移行）
〇.8   | 🟢 比較・決定 | ステップ6 | （ケース③のみ：⑤⑥を軸とした評価軸の宣言）
〇.9   | 🟢 比較・決定 | ステップ6 | （ケース③のみ：比較表と採用の決定）
〇.10  | 🟢 比較・決定 | ステップ6 | （耐久テスト：ヒアリングの将来変化を当ててみる）
〇.11  | 🟢 比較・決定 | ステップ6 | （使う場面・使わない場面の判断基準）
〇.12  | 🟢 比較・決定 | ステップ7 | （解決後のコード・動作の保証）
〇.13  | 整理        | ―        | （5フェーズとこの章でやったことのまとめ）
```

このマッピングを使って、節間の論理の流れを次のステップで確認する。

### ステップ2：7観点のチェック

`logical-analysis.md` の品質チェックリストを評価基準として、
以下の観点で問題箇所を列挙する。

---

#### 観点1：観察の品質（〇.3）

**確認する問い（logical-analysis.md「第1段階」より）：**
- 観察は「起きていること」を動詞文で書いているか
- 1観察に1事実だけ含まれているか
- 「悪い」「間違っている」「問題がある」などの評価・感情が混入していないか
- 「この観察は、コードを見せれば他の人も同意するか？」という品質チェックに通るか

**例：評価の混入（error）**
```
❌ 「このコードは設計が悪く、保守性が低い」
　→ 「悪い」「低い」は評価。事実ではない。
✅ 「〇〇を変えるたびに△△も変えなければならない」
```

**例：1観察に複数の事実（warning）**
```
❌ 「引数が増え、テストが複雑になり、チーム間の調整も必要になる」
　→ 3つの観察が1文に詰まっている。
✅ 1観察を1文で分けて書く。
```

---

#### 観点2：原因の導出（〇.4）

**確認する問い（logical-analysis.md「第2段階」より）：**
- 原因は〇.3の観察から導かれているか（飛躍がないか）
- 原因は「〇〇と△△が同じ場所にいる」という構造として表現されているか
- 「この原因が解消されれば、〇.3の観察はなくなるか？」という検証に通るか
- 原因がパターン名から逆算されていないか（結論から原因を後付けしていないか）

**循環論法の検出（error）：**
```
❌ 「Strategyパターンがないため、変更が難しい」
　→ パターン名が先に出ており、原因の分析になっていない。
✅ 「変わるルールと変わらない骨格が同じ場所にいる」
　→ 観察から構造として導出されている。
```

**飛躍の検出（error）：**
- 〇.3で観察した問題が、〇.4の原因では全く別のことを指している
- 「なぜそうなるか」の中間ステップが省略されている

---

#### 観点3：対処の必然性（〇.5・〇.6）

**確認する問い（logical-analysis.md「第3段階」より）：**
- 試行案は「〇.4の原因に対して何ができるか」から導出されているか
- 試行案が「なんとなく思いついたアイデア」ではなく、原因への正当な対応として提示されているか
- 〇.6の解決案も同じく「原因に対する正当な対応」として説明されているか

**思いつきの検出（warning）：**
```
❌ 「最初にインターフェースを使う方法が考えられます」
　→ なぜインターフェースが原因への対処になるのか説明がない。
✅ 「原因は『変わるものと変わらないものが同じ場所にいる』こと。
　　ならば『変わるもの』を独立した単位として切り出す方向が
　　論理的な対処になります。その手段として…」
```

---

#### 観点4：比較基準の先行宣言（〇.7・〇.8）

**確認する問い（logical-analysis.md「比較の論理」より）：**
- 〇.7で「何を重視するか・なぜこの状況でその基準が重要か」が比較の前に宣言されているか
- 〇.8の比較が「先に宣言した基準で各案を測る」形になっているか
- 「AよりBが優れている」という絶対評価になっていないか（「この状況の基準に照らすと」という相対評価になっているか）
- 比較の結論が基準から導かれているか（基準と結論がかみ合っているか）

**後付け基準の検出（error）：**
```
❌ 〇.8でアプローチBが優れていると示してから、
　 〇.7を振り返って「だからこの基準で考えると…」と説明している
　→ 比較を先に決めて、後から基準を当てはめている。
```

**宣言と比較のかみ合いチェック（warning）：**
- 〇.7で宣言した基準が〇.8の比較表に存在するか
- 〇.8で追加的に比較している要素が〇.7で宣言されていなかったか

---

#### 観点5：手札から対策案への導出品質（〇.6・〇.7）

**確認する問い：**
- 対策案は「ステップ4の6手札チェックテーブルで候補になった手札①〜④」から導出されているか
- 手札①〜④のどれを使うかの判断に、ステップ4の観察との対応が示されているか
- ケース③の場合、手段①は「手札①②を極限まで適用した手続き型の理想形」として示されているか（「悪いコード」や「naive な試み」として書かれていないか）
- ケース③の場合、手段①の後に「残る課題」が明示され、手段②への橋渡しになっているか
- 「手段①の限界」がなぜ発生するのか、「問題の構造」から説明されているか

**手札によらない思いつきの検出（warning）：**
```
❌ 「最初にインターフェースを使う方法が考えられます」
　→ なぜインターフェースがステップ4の原因への対処になるのか説明がない。
✅ 「ステップ4のチェックテーブルで③規格化が候補になった。
　　原因は『具象への直接依存』。ならば、つなぎ目をインターフェースで統一する方向が
　　論理的な対処になる。手札③を適用した対策案として…」
```

**手段①の描写不足の検出（warning）：ケース③のみ**
```
❌ 「引数として渡す方法では対応しきれませんでした」
　→ なぜ対応しきれないのかが書かれていない。
✅ 「手段①として手札①②を適用した。シンプルなルールには対応できる。
　　ただし、複数の状態を持つルール（〇〇かつ△△の場合）を表現しようとすると、
　　条件判断が呼び出し元に移動するだけで問題の構造は変わらない。」
　　→（残る課題の明示）
```

---

#### 観点6：5フェーズ・6手札フレームワークへの準拠チェック

rules/step5-6-model.md の5フェーズ・6手札モデルに基づいてチェックする。

**6-A: ステップ4の6手札チェックテーブル**
- ステップ4に①〜⑥の6手札を全て確認するチェックテーブルがあるか
- ①〜④について「候補か否か（○/✗）」が明示されているか
- ⑤⑥が「評価の軸」として記載され、ステップ5の操作対象に混入していないか
- 「候補」となった手札の根拠が、ステップ3で言語化した「痛み」と対応しているか

**6-B: ステップ5の対策案組み立て**
- 対策案がステップ4チェックテーブルの「候補手札（①〜④）」から導出されているか
- ケース①（手札①②のみ・対策案1つ）: 比較表・評価軸の宣言がないか（ケース①では不要）
- ケース②（手札①〜④の組み合わせ・対策案1つ）: 統合コードで示されているか・比較表がないか
- ケース③（対策案が複数）: 手段①→残る課題の明示→手段②、の直列ストーリーになっているか

**6-C: ステップ6の比較・決定**
- ケース①②: 比較表がなく耐久テストと使う/使わない判断のみであるか
- ケース③: 評価軸の宣言が比較表より前にあるか
- ケース③: 評価軸として⑤置換できるか・⑥拡張できるかが使われているか
- ケース③: 採用の決定が比較結果から明示されているか（手段①が勝つ場合も正規ルート）

**6-D: 共通チェック**
- 耐久テストがステップ2ヒアリングの将来変化リスクと対応しているか（伏線回収）
- 「使う/使わない場面」に【過剰コード】例があるか
- パターン名がステップ5の対策案実装後に初登場しているか（それより前に出ていないか）

**ケース③でのみ検出：並列アプローチの検出（error）：**
```
❌ 「大きく2つのアプローチが考えられます。アプローチA：〜。アプローチB：〜」
   という形で同格の選択肢として並んでいる
　→ ケース③では「まず手段①（手札①②）を試したところ〇〇という課題が見えた。
　   ならば手段②（手札③④追加）ではどうか」という直列ストーリーにする。

✅ 「手段①として手札①②を適用します。
　　〇〇は解決しましたが、△△という課題が残ります。」
　　→（残る課題の明示）
　「では、△△も解消するにはどうすればよいか。手段②として手札③④を追加します…」
　　→（手段②へ）
```

---

#### 観点7：節間の論理接続

**確認する問い：**
- 各節の冒頭が前節の結論を正しく受け継いでいるか
- 節の境界で話題が突然変わっていないか
- 5ステップの流れ（現状把握→課題発見→原因特定→対策→比較）が実際の文章の流れと一致しているか

**接続のチェック箇所：**

| 接続 | 確認する問い |
|:---|:---|
| 〇.0 → 〇.3 | 〇.0で立てた変動/不変の仮説が〇.3で実際のコードと照合されているか |
| 〇.2 → 〇.4 | 〇.4の困難は、〇.2の変更要求を受けて「変更しようとしたとき」に現れるものか |
| 〇.4 → 〇.5 | 〇.5の原因は、〇.4で列挙した困難から「なぜそうなるか」を問い続けた先にあるか |
| 〇.5 → 〇.6 | ステップ5の実装は、〇.5の原因への対処として「手札表から選んだ候補」に基づいているか（ケース③の場合：手段①の後に残る課題が示されているか） |
| 〇.6 → 〇.7 | ケース③の場合：手段②は手段①の残る課題への応答として示されているか（「2つのアプローチ」の並列提示ではなく） |
| 〇.8 → 〇.9 | 〇.9の比較は〇.8で宣言した基準をそのまま使っているか |
| 〇.9 → 〇.10 | 〇.10は〇.9で選んだ案を「より難しい変化で試す」形になっているか |

**接続の弱さの検出（warning）：**
```
❌ 〇.4が終わって突然〇.5のコードが現れ、
　 「次にインターフェースを使う方法を考えます」という橋渡しがない
　→ 読者が「なぜ今そのアプローチが出てくるのか」を理解できない。
```

### ステップ3：修正案の提示

ステップ2で検出した `error` と `warning` それぞれに対して修正案を提示する。

**修正案の提示ルール：**
- 修正案は「この文・この段落をどう書き換えるか」を具体的に示す
- 「〇〇を削除して△△に変える」という指示形式で書く
- 著者の口調（author-voice.md）を維持した修正案を提案する
- `info` は修正案を示すが、適用はしない（記録のみ）

---

### ステップ4：著者確認後の修正適用

ステップ3の修正案を提示した後、著者に確認を求める。
承認された修正のみ `chapter_file` に適用してファイルを上書きする。

**確認の形式：**
```
以下の修正を適用してよいですか？

【〇.4 / error / 循環論法の解消】
変更前：「Strategyパターンを使っていないため...」
変更後：「変わるルールと変わらない骨格が同じ場所にいるため...」

承認する場合は「はい」、個別に判断する場合は番号を指定してください。
```

**適用後の確認：**
修正を適用したファイルを再度読み込み、
修正によって新しい論理の矛盾が生まれていないかを確認する。
問題があれば報告する。

## orchestrator からの呼び出し方

orchestrator のフェーズ3の各章処理において、
chapter-agent の完了後に review-agent・readability-agent と **並列** で呼び出す。

```
フェーズ3の各パターン処理（並列）：
  1. .claude/hooks/pre-chapter.sh を実行
  2. chapter-agent を呼び出して章を生成
  3. .claude/hooks/post-chapter.sh を実行
  4. review-agent・logic-check-agent・readability-agent を並列で呼び出す
  5. 3エージェントの結果を統合する：
     - どれかに error があれば chapter-agent に修正を依頼
     - warning のみの場合は著者に確認して判断を仰ぐ
  6. 完了したら status を報告
```

**結果の統合ルール：**

| review-agent | logic-check-agent | readability-agent | orchestrator の対応 |
|:---|:---|:---|:---|
| error あり | （問わず） | （問わず） | chapter-agent に修正依頼（全エージェントの問題を一度に伝える） |
| （問わず） | error あり | （問わず） | logic-check-agent の修正案を確認後、chapter-agent に修正依頼 |
| （問わず） | （問わず） | error あり | readability-agent の修正案を確認後、chapter-agent に修正依頼 |
| warning のみ | warning のみ | warning のみ | 著者に提示して判断を仰ぐ |
| ok | ok | ok | フェーズ3の次のステップへ進む |

## このエージェントが担当しないこと

以下は review-agent の担当。このエージェントは判断しない。

- 8ステップ（ステップ0〜7）のH2見出しの有無（構造チェック）
- コードラベルの正しさ
- Kindle制限（1行80文字・表列数等）
- 著者の口癖・人格の一貫性
- Markdownの閉じ忘れ・コメントの残留
- 章の独立性（他章参照の禁止・次章予告の禁止）
- 禁止表現（「まず〜」「ここで答えは出しません」「次章では〜」等）

以下は readability-agent の担当。このエージェントは判断しない。

- 用語の説明順序（初出前に用語が登場していないか）
- コードと本文の対応（注目箇所の誘導）
- 段落間のつなぎ
- 核心一文の有無と位置
