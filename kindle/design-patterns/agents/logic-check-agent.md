# logic-check-agent

## 役割

章の論理の飛躍、矛盾、不要な話題を検出する。
構成の正本は `templates/chapter-template.md` である。

## 論理チェック

- 各項目の目的が、前後の項目とつながっているか。
- 各項目の達成基準が、次の項目の前提になっているか。
- 前フェーズの出力が、次フェーズの入力として明示されているか。
- 仕様、動作例、コード、実行結果が同じケースを扱っているか。
- 1-5の変更前後比較が、1-1の現状仕様を退避した内容と同じ粒度で書かれているか。
- 実行結果が、どのコード・どの動作例ケース・どの確認目的に対応するか明示されているか。
- 仕様に書かれた入力・加工・出力が、コードに出る値・状態・判定条件・出力名へ接続されているか。
- 仕様の入力・判定・加工・出力が分類名だけにならず、具体値・具体条件・具体的な出力例として後続項目へ渡せるか。
- 掲載コードの `print`、固定データ、スタブが、実システムのどの要素を簡略化しているか説明されているか。
- 実システムなら外部境界になる処理が、掲載コードでも境界クラス/インターフェースとして表されているか。`print` が業務ロジック本体へ直接混ざっていないか。
- 論点から外す処理の実際の動き・代替表現・割愛理由・設計論点への影響が説明されているか。
- 登場クラス表、クラス構成図、コード、シーケンス図が同じクラスを扱っているか。
- 変更要求、変更後コード、変更影響グラフ、痛みの言語化が同じ変更を扱っているか。
- 変更要求、確定変更、変更シナリオ表が同じ受入条件を扱っているか。
- 原因分析が、問題特定で見えた痛みから導かれているか。
- 課題定義が、原因分析で特定した接続点から導かれているか。
- 対策検討が、課題定義への応答になっているか。
- フェーズ6の候補案が、フェーズ4の原因とフェーズ5の課題に対応しており、課題にない対策へ飛んでいないか。
- クラス分離後の生成・所有・依存注入・呼び出しが説明とコードに存在し、具体クラスを知る責任がどこへ移ったか追えるか。分離先クラスだけを示して完成扱いにしていないか。
- 対策実施が、採用した対策の効果と残る変更を確認しているか。

## 飛躍として扱うもの

- フェーズ1で原因や解決策を結論づけている。
- フェーズ1の現状仕様図や動作例に、変更要求後に初めて出る機能が混ざっている。
- フェーズ2で仮説と確定事項を混ぜている。
- フェーズ2で、変わりそうな仕様候補ではなく責務混在や適切/不適切を断定している。
- フェーズ3で変更を試す前に痛みを断定している。
- フェーズ4で「知識」とだけ書き、判定条件・処理順序・生成判断へ分解していない。
- フェーズ5を飛ばして実装案へ進んでいる。
- フェーズ5の課題定義で、抽象メソッド、最終クラス名、パターン名などの対策を先出ししている。
- フェーズ6でパターン名や好みを理由に対策を選んでいる。
- フェーズ6で、読者が自然に思いつく代替案を比較せず、採用案へ飛んでいる。
- フェーズ6のコードが省略記号や宣言だけの断片で、元コードとの対応や組み立てを読者に補完させている。
- フェーズ6で、フェーズ5の課題からではなく「この章はこのパターンだから」という順序で対策へ進んでいる。
- フェーズ6で、理想の変更影響を決める前に候補を並べ、候補から課題カードを逆算している。
- 課題IDが同じ成果物内で重複・欠番になり、独立課題の識別子として機能していない。
- フェーズ7で実行結果や変更影響を確認していない。
- 仕様・動作例・コード・実行結果のケースが途中でずれている。
- 実行結果の対象コードやケースが曖昧で、読者がどの結果なのか推測する必要がある。
- 仕様にない値・状態・判定条件・出力名が、コードで突然出ている。
- 実システムでは存在する入出力を簡略化しているのに、省略範囲や代替表現を説明していない。
- 論点から外した処理が、後続フェーズの原因・課題・対策で暗黙の前提になっている。
- 図に出るクラスがコードに存在しない、またはコードの主要クラスが図にない。
- 変更要求と変更後の確認対象が途中ですり替わっている。

## 出力形式

指摘は、該当箇所、飛躍している理由、必要な橋渡しを示す。
