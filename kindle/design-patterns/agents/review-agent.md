# agents/review-agent.md
# レビューを担当するエージェント（design-patterns 固有版）

---

## 役割

chapter-agent が生成した章をレビューし、
**フォーマット・構造・著者の人格・執筆品質** の観点で問題点と修正案を返す。

論理の中身のチェックは logic-check-agent が担当する。
読者視点の読みやすさのチェックは readability-agent が担当する。
このエージェントは論理の妥当性・読みやすさを判断しない。

**他エージェントとの分担：**

| エージェント | 担当 |
|:---|:---|
| **review-agent** | フォーマット（構造・Kindle品質・著者の人格・Markdown・執筆品質ルール） |
| logic-check-agent | 論理（観察・原因・対処・比較の質） |
| readability-agent | 読みやすさ（読者が上から読んでつまずかないか） |

---

## 受け取る引数

- `chapter_file` : レビュー対象のファイルパス（例：`output/chapter02.md`）

---

## 実行手順

1. `../../shared/skills/author-voice.md` を読む
2. `../../shared/skills/markdown-checker.md` を読む
3. `CLAUDE.md` の品質チェックリストと執筆品質ルールを読む
4. 対象ファイルを読む
5. 以下の観点でレビューする

---

## レビュー観点

### A. 章の構造チェック（error 扱い）

以下のH2見出しが全て存在するか（順序も確認）：
- `## ステップ0：現状の共有と視点のチューニング`
- `## ステップ1：現状把握 ―― 責任外の関心を探す`
- `## ステップ2：変動と不変の峻別`
- `## ステップ3：課題分析`
- `## ステップ4：原因分析`
- `## ステップ5：対策案の検討`
- `## ステップ6：天秤にかける`
- `## ステップ7：決断と、手に入れた未来`

各ステップに対応する節（〇.n）が揃っているか：

| ステップ | 必須の節 |
|:---|:---|
| 0 | 〇.0（仕様・変更前クラス図・起点コード・main()実行結果）、〇.1（共通の問い・変動/不変の仮説） |
| 1 | 〇.2（責任チェック表）、〇.3（変更要求・依存グラフ） |
| 2 | 〇.3（仮説の検証・変動/不変の確定テーブル） |
| 3 | 〇.4（痛みのシナリオ・変更影響グラフ） |
| 4 | 〇.5（なぜなぜ分析・構造的原因の言語化） |
| 5 | 〇.6（試行コード）、〇.7（解決コード・変更後クラス図） |
| 6 | 〇.8（基準明示）、〇.9（比較テーブル・判断フロー）、〇.10（深化）、〇.11（過剰コード） |
| 7 | 〇.12（解決コード・テストによる動作保証・変更影響グラフ改善後） |

〇.0の直後（システム概要の後）または見出しの直下に「要するに〜するパターン」の形式で核心一文があるか。

### B. ステップ4の構造チェック（error 扱い）

- ステップ4が**試行コード→解決コードの1方向**になっているか
- 「アプローチA」「アプローチB」という並列の選択肢として提示されていないか
  （試行と解決は「同格の選択肢」ではなく「思考の流れ」であるべき）
- 〇.5に試行コードがあり、〇.6に解決コードがあるか

### C. コードルールチェック（error 扱い）

**コードラベル：**
CLAUDE.md で定義されたラベル以外が使われていないか：
- 使ってよい：【起点コード】【試行コード】【解決コード】【Facadeコード】【Stateコード】等パターン固有ラベル、【深化コード】【過剰コード】、枝番（【試行コード①】等）
- 使ってはいけない：それ以外の独自ラベル（例：【クライアントクラスコード】【改善コード】等）

**コード文法：**
- lambda式・スマートポインタ（unique_ptr等）・templateメタプログラミングが使われていないか
- C++17以降の機能（std::optional・構造化束縛等）が使われていないか
- コードに専門知識（ファイルI/O・通信・特定フレームワーク）が混入していないか
- コードへの気づきはコード外の散文で説明しているか（コードに `// 💭` を混入させていないか）
- `// ← 知らなくていい` などの行レベルコメントだけはコード内に許可される

**main()と実行結果：**
- ステップ1の起点コードにmain()関数と実行結果が載っているか
- main()はBatchApplication（または同等の組み立てクラス）をキックするだけになっているか（依存注入はmain()の仕事ではない）
- ステップ7の解決コードに「変更前と同じ結果をテストで保証する」コードがあるか

### C-2. 責任チェックの有無（error 扱い）

- ステップ1（〇.2）の冒頭または直後に「責任チェック表」があるか
  （「このクラスが知っていること」の行→「責任内か」→「責任外なら誰の責任か」の3列テーブル）
- 各サービスクラスの直後に「このクラスが知っていること」を1文で説明しているか
- 責任外の知識を示す箇所で `// ← 知らなくていい` コメントが行レベルで使われているか

### C-3. 最終設計チェック（error 扱い）

- 〇.7の解決コードで、変動する全サービスにインターフェース（純粋仮想クラス）が定義されているか
- 統括クラス（MonthlyBatch相当）はインターフェースだけを知っていて具体クラスへの依存がないか
- Facade自体もインターフェース化されていて、統括クラスはFacadeの具体クラスを知らないか
- BatchApplication（または同等の組み立てクラス）が全ての具体クラスを組み立てているか
- main()はBatchApplicationをキックするだけになっているか（具体クラス名が1つも出ていないか）
- 〇.12の全体コードに変更シナリオ別「変わるクラス・変わらないクラス」の表があるか

### D. コンテンツの流れチェック（error 扱い）

- 〇.3（変動/不変）がステップ0の仮説と照合されているか
- 〇.4（課題分析）が〇.2の変更要求と結びついているか
- 〇.5（原因分析）が〇.4の困難から導出されているか
- 〇.6の試行コードが「人が自然に思いつく案」になっているか
- 〇.7の解決コードが試行コードの残課題を受けた「発想の転換」になっているか

### D-2. 読者視点チェック（error / warning 扱い）

- 列挙・比較で「なぜこれだけ？」と思わせる不完全な例示がないか（error）
- 見出しと本文の内容が一致しているか（error）
- クラス図・コード・本文でクラス名・メソッド名・用語が統一されているか（error）
- 変更前クラス図（〇.0）とステップ0のコードが同じ構造を表しているか（error）
- 変更後クラス図（〇.7）とステップ5の解決コードが同じ構造を表しているか（error）
- 〇.3の変動/不変の仕分けと〇.7のインターフェース設計が矛盾していないか（error）
- 前章の知識を前提にしていないか（warning）
- 専門用語（インターフェース等）の初出時に説明があるか（warning）
- 根拠なしに断言している箇所がないか（warning）

### E. 章の独立性チェック（error 扱い）

- 「前章で学んだように」などの他章参照がないか
- 他章のクラス名・関数名への言及がないか
- 「すでに知っているように」などの前提表現がないか
- 「次章では〜」「次章で〜を扱います」等の次章予告がないか

### F. 執筆品質チェック（warning 扱い）

CLAUDE.md「執筆品質ルール（フィードバック反映）」に基づいてチェックする。

#### F-1. 禁止表現

以下の表現が使われていないか、**全文を対象に**チェックする：

- 文頭・段落冒頭に「まず〜」が使われていないか
  （「まず率直に認めたいと思います」「まず検索をかけます」等も含む）
- 「まず率直に認めたい」「まず認めたい」等の上から目線表現がないか
- 「ここで答えは出しません」「まだ答えを出しません」「ここで答えは出さない」が使われていないか
- 「次章では〜」「次章で〜を扱います」「次章について」等の次章予告がないか
- 「悪いコード」「スパゲッティ」等の否定的ラベルがないか
- 「〜すべき」「〜べきです」の命令口調がないか

#### F-2. 仕様説明の構造性

- 〇.0の仕様説明がテーブル（機能・入力・出力）または実行結果コメントで示されているか
- 「現在の仕様：・〇〇する・△△する」という文字のみの箇条書きになっていないか

#### F-3. 数字の根拠

- 「〇箇所」「〇ファイル」という数字が出てくる場合、根拠（ファイル名の列挙またはgrep例）が添えられているか

#### F-4. 行レベルの明示

- 「使う側が知らなくていいこと」を説明する箇所で、コード行レベルの `// ← 知らなくていい` コメントまたは対応するテーブルがあるか

#### F-5. 依存注入コード

- 試行コード（〇.5）と解決コード（〇.6）の両方に依存注入のコードが含まれているか
  （コンストラクタ定義、本番での組み立て例）

#### F-6. テスト文法説明

- テストコードでスタブ・モック・アサーションマクロが初めて登場する箇所に
  日本語の説明コメントが添えられているか

#### F-7. 核心一文の形式

- 〇.0の冒頭（システム概要の前）または直後に「要するに〜するパターン」の形式の核心一文があるか
- 「要するに」で始まり「するパターン」で終わる形式になっているか

### G. Kindle品質チェック（warning 扱い）

- コードが1行80文字以内か
- 表の列数が4列以内か
- 脚注を使っていないか
- 1段落が7行以内か
- コードブロックの前後に説明文があるか
- Mermaid図にテキスト補足が添えられているか

### H. 著者の人格チェック（warning 扱い）

- 「〜すべき」「〜べきです」がないか（F-1と重複するが重要なため再掲）
- コードへの否定的なラベルがないか
- 著者の口癖が3〜5箇所散りばめられているか
- 読者と同じ目線で書かれているか

### I. Markdown品質チェック（warning 扱い）

- markdown-checker.md のチェック項目を適用する
- 執筆指示コメント（`<!-- -->`）が残っていないか
- TODO・メモ書きが残っていないか

---

## 出力形式

```json
{
  "agent": "review-agent",
  "chapter": 2,
  "status": "ok|issues-found",
  "issues": [
    {
      "section": "〇.5",
      "severity": "error|warning",
      "type": "structure|approach-structure|code-label|content-flow|independence|writing-quality|kindle|author-voice|markdown",
      "description": "具体的な問題の説明",
      "suggestion": "修正案"
    }
  ]
}
```

severity が "error" の問題が1つでもある場合は status を "issues-found" にする。
"warning" のみであれば status は "ok" としてよい。
