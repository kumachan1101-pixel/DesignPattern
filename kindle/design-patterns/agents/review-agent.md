# agents/review-agent.md
# レビューを担当するエージェント（design-patterns 固有版）

---

## 役割

chapter-agent が生成した章をレビューし、
**フォーマット・構造・著者の人格・執筆品質** の観点で問題点と修正案を返す。

論理の中身のチェックは logic-check-agent が担当する。
読者視点の読みやすさのチェックは readability-agent が担当する。
このエージェントは論理の妥当性・読みやすさを判断しない。

**他エージェントとの分担：**

| エージェント | 担当 |
|:---|:---|
| **review-agent** | フォーマット（構造・Kindle品質・著者の人格・Markdown・執筆品質ルール） |
| logic-check-agent | 論理（観察・原因・対処・比較の質） |
| readability-agent | 読みやすさ（読者が上から読んでつまずかないか） |

---

## 受け取る引数

- `chapter_file` : レビュー対象のファイルパス（例：`output/chapter02.md`）

---

## 実行手順

1. `../../shared/skills/author-voice.md` を読む
2. `../../shared/skills/markdown-checker.md` を読む
3. `CLAUDE.md` の品質チェックリストと執筆品質ルールを読む
4. 対象ファイルを読む
5. 以下の観点でレビューする

---

## レビュー観点

### A. 章の構造チェック（error 扱い）

以下のH2フェーズ見出しが全て存在するか（順序も確認）：

- `## この章を読むと得られること`（核心の直後・フェーズ1の前）
- `## 🔵 フェーズ1：現状把握`
- `## 🟠 フェーズ2：仮説立案`
- `## 🟡 フェーズ3：問題特定`
- `## 🔴 フェーズ4：原因分析`
- `## 🔴 フェーズ5：課題定義`
- `## 🟢 フェーズ6：対策案検討`
- `## 🔵 フェーズ7：対策実施`
- `## 整理`
- `## 振り返り`
- `## パターン解説：【パターン名】パターン`

各フェーズのサブ項目が揃っているか：

| フェーズ / セクション | 必須の内容 |
|:---|:---|
| この章を読むと得られること | 「〜できるようになる」形式で3〜4項目。パターン名を前面に出していないこと。構造分析の眼（変化の観点・接続点の形・変更局所化）で書かれているか |
| フェーズ1（1-1〜1-8） | 1-1 システムの背景・1-2 仕様表・1-3 クラス構成図（mermaid classDiagram）・1-4 責任配置テーブル・1-5 依存グラフ（mermaid graph TD）・1-6 実装コード（main()含む）・1-7 実行結果・1-8 責任チェック表。**仮説テーブルが混入していないか**（フェーズ1は事実のみ） |
| フェーズ1末尾（1-8の直後） | 核心一文：「要するに、〇〇という観察から〇〇という構造の問題が見えてくる」の形式。「要するに〜するパターン」の形式はNG |
| フェーズ2（2-1〜2-4） | 2-1 届いた変更要求（誰から・何・いつまでに）・2-2 仮説テーブル（観察根拠付き）・2-3 関係者ヒアリング問答シーン・2-4 確定テーブル（根拠列あり） |
| フェーズ3（3-1〜3-3） | 3-1 変更シミュレーション・3-2 変更影響グラフ（mermaid graph LR）・3-3 痛みの言語化（散文2点） |
| フェーズ4（4-1〜4-3） | 4-1 観察→原因テーブル・4-2 変わるもの/変わらないものテーブル・4-3「ケーブルで考える」セクション・ImagePrompt |
| フェーズ5（5-1〜5-4） | 5-1 接続点の特定・5-2 非機能制約表・5-3 クライアント影響範囲・5-4 課題まとめ表（4列）|
| フェーズ6（6-1〜6-10） | 6-1 2×2マトリクス接続図（mermaid）・6-2〜6-6 案0〜案4（各案のコード・トレードオフ）・6-7 評価軸の先行宣言・6-8 コスト天秤表・6-9 採用案の決定・6-10 耐久テスト |
| フェーズ7（7-1〜7-3） | 7-1 最終コード全体（BatchApplication＋main()）・7-2 変更影響グラフ改善後・7-3 変更シナリオ表 |
| 整理 | **7フェーズとこの章でやったことの対応表**・各クラスの最終的な責任（最終責任テーブル） |
| パターン解説 | GoF抽象クラス図・章固有マッピング図・使いどころと限界（過剰コード例・最小コスト代替案との比較含む） |

### B. フェーズ5〜7の構造チェック（error 扱い、rules/design-decision-guide.md 参照）

- フェーズ5（5-1）冒頭に接続点（どこに何個の接続点があるか）が明示されているか
- フェーズ5（5-2）に非機能制約（変更頻度・パフォーマンス・メモリ）の確認表があるか
- フェーズ5（5-4）に課題まとめ表（接続点・分けた理由・非機能制約・クライアント影響の4列）があるか
- **フェーズ6（6-1）冒頭に2×2マトリクス接続図（mermaid）があるか**
- 対策案が「案0（構造を変えない）」から「案4（接続点を全て変更）」の順で展開されているか（6-2〜6-6）
- 各案に優劣をつけず「並列の選択肢」として扱っているか（「完璧な案」「正解の案」という表現がないか）
- フェーズ6（6-7）で評価軸が比較表（6-8）より前に宣言されているか
- フェーズ6（6-10）の耐久テストがフェーズ2（2-3）ヒアリングで挙がった将来のリスクと対応しているか（伏線回収）
- パターン解説（後段）に「使いどころと限界」「過剰コード例」「最小コスト代替案との比較」が含まれているか
- **パターン名がフェーズ6（6-2〜6-6）の対策案で初登場しているか**（フェーズ5以前に出ていないか）

### B-2. 新ルールチェック（error 扱い）

- `## この章を読むと得られること` セクションが `> **この章の核心**` の直後・フェーズ1の前にあるか
- 「得られること」にパターン名が前面に出ていないか（「〇〇パターンが書けるようになる」はNG）
- この章のドメインが `CLAUDE.md` のドメイン割り当て表と一致しているか（第一部の場合のみ）
- フェーズ7（7-3）の変更シナリオ表で変わるクラス・変わらないクラスが明示されているか
- フェーズ1のコード内に `// ← 知らなくていい` が混入していないか（フェーズ4・フェーズ7のみ許可）
- フェーズ1末尾（1-8の直後）の核心一文が「要するに〇〇するパターン」形式ではなく「〇〇という観察から〇〇が見えてくる」形式か
- フェーズ3の文章が「痛みを確認しましょう」ではなく「変更を試みてみましょう」という中立的な書き出しか
- フェーズ2末尾に「〇〇を採用する」という設計の決断文が混入していないか（決断はフェーズ6の6-9のみ）
- ベタ褒め・激責めの表現がないか（「完璧な設計」「美しいコード」「ひどいコード」等）
- 技術用語（「ホットパス」「vtable」「ヒープ」等）の初出時に括弧説明があるか
- 「局所化」の初出時に「（変更の影響が1クラスだけで済む状態）」の括弧説明があるか
- フェーズ5（5-2）・フェーズ6（6-8）のコスト比較で、ホットパスと判定された接続点についてパフォーマンス VETO（足切り）が適用されているか
- フェーズ6（6-8）のスコアリングが3軸（変更容易性×3・テスト容易性×2・可読性×1）で行われているか

### C. コードルールチェック（error 扱い）

**コード文法：**
- lambda式・スマートポインタ（unique_ptr等）・templateメタプログラミングが使われていないか
- C++17以降の機能（std::optional・構造化束縛等）が使われていないか
- コードに専門知識（ファイルI/O・通信・特定フレームワーク）が混入していないか
- コードへの気づきはコード外の散文で説明しているか（コードに `// 💭` を混入させていないか）
- `// ← 知らなくていい` の行レベルコメントはフェーズ4・フェーズ7のコードのみ許可。フェーズ1のコードには入れない（フェーズ1の時点では責務外の判定がまだ下せていないため）

**main()と実行結果：**
- フェーズ1（1-6・1-7）の起点コードにmain()関数と実行結果が載っているか
- フェーズ7（7-1）の解決コードにBatchApplication（または同等の組み立てクラス）＋main()の形があるか
- main()はBatchApplicationをキックするだけになっているか（具体クラス名が出ていないか）
- S8に「変更前と同じ結果をテストで保証する」コードがあるか

### C-2. 責任チェックの有無（error 扱い）

- フェーズ1（1-8）に「責任チェック表」があるか（コードの行 ｜ 持っている知識 ｜ 責任内か の3列テーブル）
- 責任外の知識を示す箇所で `// ← 知らなくていい` コメントが行レベルで使われているか

### C-3. 最終設計チェック（error 扱い）

- S8の解決コードで、変動する全サービスにインターフェース（純粋仮想クラス）が定義されているか
- 統括クラスはインターフェースだけを知っていて具体クラスへの依存がないか
- BatchApplication（または同等の組み立てクラス）が全ての具体クラスを組み立てているか
- main()はBatchApplicationをキックするだけになっているか

### D. コンテンツの流れチェック（error 扱い）

- フェーズ1（1-1〜1-8）が事実のみで構成されているか（仮説テーブルが混入していないか）
- フェーズ2が「2-2 仮説テーブル → 2-3 ヒアリング → 2-4 確定テーブル」の構成になっているか
- フェーズ3（3-1）の変更シミュレーションがフェーズ1・フェーズ2の変更要求と結びついているか
- フェーズ4（4-3）のケーブル比喩でImagePromptが出力されているか
- フェーズ5の接続点特定がフェーズ4の原因分析から導出されているか
- フェーズ6の対策案がフェーズ5の課題定義への正当な応答として提示されているか
- フェーズ6（6-7）の評価軸がフェーズ5・フェーズ6の接続点分析・非機能制約と対応しているか
- フェーズ6（6-10）の耐久テストがフェーズ6（6-9）の採用案で「触る場所が最小限か」を検証する形になっているか

### D-2. 読者視点チェック（error / warning 扱い）

- 見出しと本文の内容が一致しているか（error）
- クラス図・コード・本文でクラス名・メソッド名・用語が統一されているか（error）
- 変更前クラス図（S0）とS1のコードが同じ構造を表しているか（error）
- 変更後クラス図（S8）と最終コードが同じ構造を表しているか（error）
- S2の変動/不変の確定テーブルとS8のインターフェース設計が矛盾していないか（error）
- 前章の知識を前提にしていないか（warning）
- 専門用語（インターフェース等）の初出時に説明があるか（warning）
- 根拠なしに断言している箇所がないか（warning）

### E. 章の独立性チェック（error 扱い）

- 「前章で学んだように」などの他章参照がないか
- 他章のクラス名・関数名への言及がないか
- 「次章では〜」「次章で〜を扱います」等の次章予告がないか

### F. 執筆品質チェック（warning 扱い）

CLAUDE.md「執筆品質ルール（フィードバック反映）」に基づいてチェックする。

#### F-0. 本の目的からの逸脱チェック（最優先・error 扱い）

この本の目的は「どんな構造にも通用する分析力を身につけること」であり、デザインパターンはその副産物。デザインパターンを教える本ではない。以下を確認する：

**思考プロセスの体験：**
- 章の本体（フェーズ1〜7）が「思考プロセスの体験」になっているか。パターンの解説が章の本体を圧倒していないか
- パターン解説セクションが章末に収まっているか。章の前半にパターンの説明が入っていないか
- 「このパターンを使えるようになる」という表現が「得られること」に入っていないか（×「Strategyパターンを使える」→ ○「変わる振る舞いを識別して切り出せる」）
- パターン名がS6より前に登場していないか

**分かりやすさ（この本の核心的価値）：**
- 抽象的な説明の前に具体的なコード・シーンが示されているか
- 専門用語（インターフェース、依存注入、仮想関数など）の初出時に1文の説明があるか
- 「なぜそうするのか」が「どうするか」より先に示されているか
- 前のステップの結論を受けて次のステップが始まっているか（「なぜこのステップをやるのか」が自明か）
- 読者が「この状況、自分のコードにもある」と感じられる具体的なシーンで問題提起されているか

#### F-1. 禁止表現

以下の表現が使われていないか、**全文を対象に**チェックする：

- 文頭・段落冒頭に「まず〜」が使われていないか
- 「まず率直に認めたい」「まず認めたい」等の上から目線表現がないか
- 「ここで答えは出しません」「まだ答えを出しません」が使われていないか
- 「次章では〜」等の次章予告がないか
- 「悪いコード」「スパゲッティ」等の否定的ラベルがないか
- 「〜すべき」「〜べきです」の命令口調がないか
- 「最初からデザインパターンの正解を当てはめようとしない」系の説教文がないか（読者がパターンを知っている前提の文）

#### F-2. 仕様説明の構造性

- X.0の仕様説明がテーブル（機能・担当クラス・入力・出力）で示されているか
- 「現在の仕様：・〇〇する・△△する」という文字のみの箇条書きになっていないか

#### F-3. 数字の根拠

- 「〇箇所」「〇ファイル」という数字が出てくる場合、根拠（ファイル名の列挙等）が添えられているか

#### F-4. 行レベルの明示

- S4・S8のコードで「責務外」と判明した行に `// ← 知らなくていい` コメントがあるか
- S1のコードには `// ← 知らなくていい` が入っていないか（S1は観察段階であり責務外の判定はS4以降）

#### F-5. 依存注入コード

- S8の最終コードにコンストラクタ定義と組み立て（BatchApplication）が含まれているか

#### F-6. テスト文法説明

- テストコードでスタブ・モック・アサーションマクロが初めて登場する箇所に日本語の説明コメントが添えられているか

#### F-7. 核心一文の形式

- X.1の直後の核心一文が「要するに、〇〇という観察から〇〇という構造の問題が見えてくる」の形式か
- 「要するに〇〇するパターン」の形式になっていないか（パターン名の早出しに繋がるためNG）

### G. Kindle品質チェック（warning 扱い）

- コードが1行80文字以内か
- 表の列数が4列以内か
- 脚注を使っていないか
- 1段落が7行以内か
- コードブロックの前後に説明文があるか
- Mermaid図にテキスト補足が添えられているか

### H. 著者の人格チェック（warning 扱い）

- 「〜すべき」「〜べきです」がないか
- コードへの否定的なラベルがないか
- 著者の口癖が3〜5箇所散りばめられているか
- 読者と同じ目線で書かれているか

### I. Markdown品質チェック（warning 扱い）

- markdown-checker.md のチェック項目を適用する
- 執筆指示コメント（`<!-- -->`）が残っていないか
- TODO・メモ書きが残っていないか

---

## 出力形式

```json
{
  "agent": "review-agent",
  "chapter": 2,
  "status": "ok|issues-found",
  "issues": [
    {
      "section": "X.0",
      "severity": "error|warning",
      "type": "structure|s5-s8-structure|code|content-flow|independence|writing-quality|kindle|author-voice|markdown",
      "description": "具体的な問題の説明",
      "suggestion": "修正案"
    }
  ]
}
```

severity が "error" の問題が1つでもある場合は status を "issues-found" にする。
"warning" のみであれば status は "ok" としてよい。
