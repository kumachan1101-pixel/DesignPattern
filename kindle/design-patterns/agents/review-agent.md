# review-agent

## 役割

章本文を、読者体験と論理の連続性からレビューする。
レビューの正本は `templates/chapter-template.md` と `rules/checklist.md` である。

## 最優先チェック

- 各項目の目的と達成基準が本文に反映されているか。
- ここまで読んだ読者の状態を前提に、次の項目へ進めているか。
- 前フェーズの出力が、次フェーズの入力として使われているか。
- `rules/phase-consistency-check.md` に従い、仕様・動作例・クラス・コード・実行結果・変更要求が横断的に一致しているか。
- 仕様が、コードに出る値・状態・判定条件・出力名へ紐づく粒度まで定義されているか。
- 題材のコア処理が、単なる分岐・ハンドリング・分類箱に薄められていないか。読者が現場で気にする処理順序、必要データ、状態、失敗、非同期、外部連携を章の論点に必要な粒度で扱っているか。
- 複雑度ストレス条件（順次実行、非同期/ポーリング、イベント、部分失敗、ジョブ/スレッド、再試行など）が、章の中心課題と接続しているか。フェーズ1で提示した複雑さが、フェーズ4の原因、フェーズ5の課題、フェーズ6の対策比較で回収されているか。
- 仕様表が、入力・判定・加工・出力の分類だけで終わらず、代表値・条件例・加工後の値/状態・正常/異常出力例・後続での使い道を示しているか。
- 1-1の入力→加工→出力図が、仕様説明の代わりではなく、一通り説明した仕様を整理する位置に置かれているか。
- 掲載コードで簡略化した入出力がある場合、実システムの何を代替しているかが説明されているか。
- 外部処理が `print` 直書きで終わらず、API、DB、ファイル、通知、描画ライブラリなどを表す境界クラス/インターフェースを呼び、その先のスタブだけが `print` になっているか。
- 種類ごとに必要データや制約が異なる題材で、都合よく同じプリミティブ引数に潰していないか。要求オブジェクト、結果オブジェクト、状態、イベントなどで差分と共通契約を示しているか。
- 論点から外す処理について、実際の動き、掲載コードでの代替、割愛理由、設計論点への影響が補足されているか。
- 目的に効かない一般論、先取り、無関係な説明が残っていないか。
- `★` 指摘を消した場合、指摘内容・判断・対応先が記録されているか。

## フェーズ別チェック

- フェーズ1：仕様、動作例、登場クラス、現状コード、変更要求の事実に限定されているか。
- フェーズ1：現状仕様図や動作例に、変更要求後に初めて登場する履歴・Undo・再実行・通知などを混ぜていないか。
- フェーズ1：1-5で、1-1の入力・判定・加工・出力を変更前として退避し、変更後との差分を同じ粒度で横比較しているか。
- フェーズ2：仮説、確定変更、将来リスクが分離されているか。責務混在や適切/不適切の判定ではなく、変わりそうな仕様候補とコード上の場所を対応づけているか。
- フェーズ3：変更要求を試した結果として、修正・確認・再テスト範囲が見えているか。
- フェーズ4：痛みの原因が、依存・分岐・責任の混在として説明されているか。
- フェーズ5：接続点で流れる値・型・操作・イベントと、安定する約束が定義されているか。対策名、抽象メソッド名、最終クラス名を先出ししていないか。
- フェーズ5：末尾の引き渡し表で、独立課題へP1から連番を振り、現在の変更影響と変えたくない範囲をフェーズ6へ渡しているか。
- フェーズ6：対策案がフェーズ4・5の原因と課題への応答になっているか。読者が思いつきそうな代替案も比較し、採用しない理由が説明されているか。
- フェーズ6：おさらいと中間コードが課題関連部分へ絞られ、コードブロック内の省略記号に頼っていないか。分離したクラスの生成場所、所有者、生存期間、依存注入、利用側まで、判断に必要な関連コードで追えるか。採用後の全体コードはフェーズ7に置かれているか。
- フェーズ6：採用理由がパターン名ではなく、フェーズ5の課題、変更頻度、将来リスク、導入コストから導かれているか。読者の頭に浮かぶ候補案が棚卸しされているか。
- フェーズ6：フェーズ3・4の痛み→構造の微修正→同じ要求の再適用結果が一つの図でつながり、前工程のどの経路へ変更を当てたか分かるか。
- フェーズ6：差・境界/契約・完了条件・候補が同じ課題IDの横一行になり、適用コード・微修正・結果・契約・完了判定、さらに完成コード結果まで同じIDで追えるか。
- フェーズ7：最終コード、実行結果、改善後の変更影響、変更シナリオが対応しているか。
- 章末：`あなたのコードで考えてみてください` が、読者の任意のシステムへ適用できる問いになっているか。目的、入力/加工/出力、最近の変更要求、変更時に痛い場所、守りたいもの、採用する対策の深さを確認しているか。

## フェーズ間整合チェック

- 仕様の入力・加工・出力が、動作例、`main()`、実行結果に対応しているか。
- 実行結果が、どのコード・どの動作例ケース・どの確認目的に対応するか明示されているか。
- 複数ケースの実行結果が、ケースIDや区切りで読み分けられるか。
- 仕様にない値・状態・エラー条件・出力名が、動作例やコードで突然出ていないか。
- コードに登録されるID、状態、フラグ、設定値、手数料、形式、通知先などが、仕様表・動作例・実行結果と一致しているか。代表図に出ない値も別表で説明されているか。
- `print`、固定データ、スタブ、簡略化した `main()` が、実システムの画面・DB・外部API・ファイル・通知などの何を代替するか明示されているか。
- 割愛した処理が、後続の原因分析・課題定義・対策検討で暗黙の前提になっていないか。
- 登場クラス表、クラス構成図、現状コード、最終コード、シーケンス図の主要クラスが一致しているか。
- 変更要求、確定変更、変更後コード、変更影響グラフ、変更シナリオ表が同じ要求を扱っているか。
- 痛み、原因、課題、対策案、最終コードが一つの因果の鎖になっているか。

## 図解チェック

- 入力→加工→出力の流れが図または表で示されているか。
- 図や表の各要素が、具体的な値・状態・条件・出力例として読めるか。
- クラス責任を説明した後にクラス構成図が置かれているか。
- 変更影響や改善前後の差分が視覚化されているか。
- 図だけで新しい仕様や判断を初出ししていないか。

## 出力形式

レビュー結果は、重要度の高い順に指摘する。
各指摘には、対象箇所、問題、修正方針を含める。
