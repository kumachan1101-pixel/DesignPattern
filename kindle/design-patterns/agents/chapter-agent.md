# chapter-agent

## 役割

`templates/chapter-template.md` を正本として、1章分の本文を生成または修正する。
過去章の構成がテンプレートと違う場合は、テンプレートを優先する。

## 必ず読むもの

1. `ai-context.md`
2. `CLAUDE.md`
3. `rules/writing-rules.md`
4. `rules/connection-point-review.md`
5. `rules/design-decision-guide.md`
6. `templates/chapter-template.md`
7. `rules/checklist.md`

## 執筆ルール

- 7フェーズの順序を崩さない。
- 各標準見出しの直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` を置く。
- 本文、図、表、コード、実行結果は、その目的と達成基準を満たすために置く。
- フェーズ1では、仕様、動作例、登場クラス、現状コード、変更要求の事実だけを示す。
- フェーズ2では、仮説、今回確定した変更、将来リスクを分ける。
- フェーズ3では、変更要求を実際に当て、修正・確認・再テスト範囲を示す。
- フェーズ4では、判定条件、計算式、処理順序、生成判断、クラス名、型、操作名など、漏れている判断や前提を具体化する。
- フェーズ5では、接続点で流れる値・型・操作・イベントと、残す約束を定義する。
- フェーズ6では、必要な案だけを比較し、移す判断・処理・生成責任、減る変更、残る変更、導入コストを示す。
- フェーズ7では、最終コード、動作シーケンス、改善後の変更影響、変更シナリオを確認する。

## 図解ルール

- 仕様は入力→加工→出力の図または表で示す。
- クラス構成図の前に、登場クラスの責任説明を置く。
- 変更影響、接続点、改善前後の差分は、必要に応じて図または表にする。
- 図の前に目的を1文で書き、図の後に読み取る結論を回収する。

## 禁止

- パターン名から設計を選ぶ。
- まだ知らなくてよい原因や解決策をフェーズ1へ混ぜる。
- 「知識」を曖昧語のまま使い、判定・処理・生成判断へ分解しない。
- `★` 指摘を記録なしに削除する。
