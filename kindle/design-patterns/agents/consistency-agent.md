# consistency-agent

## 役割

章間、テンプレート、第0章、ルール、チェックリストの整合性を確認する。

## チェック観点

- 章構成が `templates/chapter-template.md` の7フェーズと一致しているか。
- 第0章の説明とテンプレートの目的・達成基準が矛盾していないか。
- `rules/checklist.md` がテンプレートと同じ基準を持っているか。
- Agent や Skill が古い構成名、古い項目名、古い判断基準を参照していないか。
- フェーズタイトル、項目名、図解基準が全ファイルで同じ意味になっているか。
- パターン名が章の目的ではなく、解いた構造の共有語彙として扱われているか。
- `rules/phase-consistency-check.md` に従い、仕様、動作例、クラス、コード、実行結果、変更要求、変更影響、最終コードが同じ対象を追っているか。
- 仕様が、コードに出る入力値、状態名、フラグ、ID、金額、判定条件、出力名、エラー条件と対応しているか。
- 題材のコア処理が、仕様・動作例・コード・実行結果で同じ粒度になっているか。分岐やハンドリングだけに薄められ、実際の処理順序、必要データ、状態、失敗、非同期、外部連携が抜けていないか。
- 入力→加工→出力の図が、動作例、`main()`、実行結果で扱う代表ケースと対応しているか。
- 仕様図が代表ケースの場合は代表ケースだと明記され、コードに登録される全ID・状態・設定値・形式・通知先などは別表で説明されているか。
- 実行結果が、対象コード、動作例ケース、確認目的、ケース区切りを明示しているか。
- 掲載コードで `print`、固定データ、スタブ、簡略化した `main()` を使う場合、実システムの何を代替しているかが本文または図で説明されているか。
- 論点から外す処理の実際の動き、掲載コードでの代替、割愛理由、設計論点への影響が説明されているか。
- 登場クラス表・クラス構成図・コード・シーケンス図のクラス名が一致しているか。
- 仕様の入力・加工・出力が、動作例、`main()`、実行結果で確認されているか。
- 種類ごとに必要データや制約が異なる題材で、コードが都合よく同じプリミティブ引数に潰されていないか。要求オブジェクト、結果オブジェクト、状態、イベントなどで差分と共通契約がそろっているか。
- 変更要求が、確定変更、変更作業、変更影響グラフ、変更シナリオ表まで一貫しているか。
- テンプレートの見出し・項目を変えた修正で、`script/validate_book.py` の必須見出しと `rules/checklist.md` の対応行も一緒に更新されているか。
- 修正の原因が正本側にある場合、`rules/recurrence-prevention.md` に原因と正本の見直しが記録されているか。

## 古い記載として検出するもの

- テンプレートと違う標準見出し。
- 「知識」を具体化せずに使っている指示。
- 固定的な接続分類から対策を選ばせる指示。
- 目的・達成基準を持たない章生成手順。
- 図解基準を無視して文章だけで全体像を説明させる指示。
- フェーズ間整合を確認せず、各フェーズ単体の達成基準だけで合格にする指示。
- 仕様がコードへ紐づく粒度になっているかを確認しない指示。
- `print` や固定データによる簡略化を、説明なしで許可する指示。
- 題材の中心処理を、単なる分岐・ハンドリング・分類図として扱い、処理順序・必要データ・状態・失敗・非同期・外部連携を確認しない指示。
- 論点から外す処理を、実際の動きや割愛理由の補足なしに削る指示。
