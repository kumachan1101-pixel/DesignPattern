# CLAUDE.md - 設計思考を鍛える本

**design-patterns プロジェクト共有の設定ファイル**

## この本の目的

この本は、デザインパターン名を暗記する本ではない。
変更要求を受けたときに、読者が目の前のコードで次を説明できるようになるための本である。

1. 何が変わるのか。
2. 何を変えたくないのか。
3. 変更を試すとどこが痛むのか。
4. 変わる理由が異なる判断・条件・処理順序・生成判断がどこに混在しているのか。
5. 接続点で何を受け渡しているのか。
6. 呼び出し元へどの判断や前提が漏れているのか。
7. どの判断・条件・処理・生成責任をどの責任へ移すのか。
8. 導入コストに見合うか。
9. 変更要求を再度当てたとき、修正・再テスト範囲がどう変わるか。

パターン名は、問題を解いた後で構造を共有するための語彙として扱う。
パターン名から設計を選ばない。

## 正本

別のAIが作業する場合は、まず `AI_HANDOFF.md` を読む。
章構成の正本は `templates/chapter-template.md` である。
第0章 `output/chapter00_2.md` は、その思想と読み方を読者へ説明する章である。
両者が食い違う場合は、テンプレートと第0章を一緒に見直す。

章生成・レビュー・修正では、次を順に読む。

1. `AI_HANDOFF.md`
2. `ai-context.md`
3. `design-philosophy.md`
4. `rules/writing-rules.md`
5. `rules/connection-point-review.md`
6. `rules/design-decision-guide.md`
7. `rules/phase-consistency-check.md`
8. `templates/chapter-template.md`
9. `rules/checklist.md`

## 7フェーズ

| フェーズ | 目的 | 主な出力 |
|---|---|---|
| 1：現状把握 | 仕様、動作、登場クラス、現状コード、変更要求を事実として読む | 仕様構造、動作例、責任一覧、現状コード、変更差分 |
| 2：仮説立案 | 変わりそうな部分、今回確実に変わること、将来リスクを分ける | 仮説、確認結果、確定変更、将来リスク |
| 3：問題特定 | 変更要求を試し、修正・確認・再テスト範囲として痛みを確認する | 変更後コード、変更影響グラフ、痛みの言語化 |
| 4：原因分析 | 混在する判断・処理・生成判断と、接続点へ漏れている前提を言語化する | 原因表、分離すべき知識、漏れている前提 |
| 5：課題定義 | 接続点で受け渡すもの、決定者、安定する約束、制約を定める | 接続点定義、制約、課題文 |
| 6：対策検討 | 必要な分離案だけを比較し、移す判断・残る変更・導入コストを示す | 対策案比較、採用判断 |
| 7：対策実施 | 実装・実行し、変更要求で効果と限界を確認する | 最終コード、シーケンス図、改善後影響グラフ、変更シナリオ |

各フェーズは、前フェーズの出力を次フェーズの入力として使う。
論理の飛躍がある場合は、フェーズの入力または加工が足りない。

## 目的と達成基準

各標準見出しの直下に、本文へ出さない執筆用コメントとして `目的` と `達成基準` を置く。

- 目的：なぜこの項目を置くのか。
- 達成基準：この項目を読み終えた読者が何を説明・予測・判断できる状態になるべきか。

目的を達成しない説明、図、表、コード、実行結果は入れない。
達成基準に関係しない一般論や先取りは削る。

## 図のルール

文章だけで全体像を説明しきろうとしない。
次は、できるだけ図または表で先に見せる。

- 入力→加工→出力の流れ。
- クラス間の責任と依存。
- 変更影響の広がり。
- 接続点で流れる値・型・操作・イベント。
- 改善前後の差分。
- 責任がどこからどこへ移ったか。

図の前には目的を1文で書く。
図の後には、図から読み取るべき結論を1〜3文で回収する。
図だけで新しい仕様や判断を初出ししない。

## フェーズ間整合

各フェーズ単体の達成基準を満たしていても、フェーズ間で対象がずれていれば不合格とする。
仕様、動作例、コード、実行結果、クラス構成図、変更要求、変更影響グラフ、最終コードは同じ対象を追う。

詳細な照合基準は `rules/phase-consistency-check.md` を参照する。

## 仕様粒度と簡略化

仕様は、コードを読むための地図である。
入力→加工→出力の図は分かりやすいだけでなく、後続コードに出る値、状態名、フラグ、判定条件、出力名、エラー条件へ紐づいていなければならない。

掲載コードで `print`、固定データ、スタブ、簡略化した `main()` を使う場合は、実システムの画面、DB、外部API、ファイル出力、通知などの何を代替しているかを本文または図で示す。
省略した要素は、存在しないものとして扱わず、この章の設計論点では扱わない範囲として明示する。
論点から外す処理は、実際の動き、掲載コードでの代替表現、割愛する理由、設計論点への影響がないことを補足する。

## 執筆ルール

- 既存コードを否定しない。
- 一つの正解を押しつけない。
- 架空の著者体験を書かない。
- 「変更がゼロ」「絶対に壊れない」と断定しない。
- 組み立て、登録、設定、テストに残る変更を隠さない。
- コードは実行可能にし、掲載した出力と照合する。
- AIへの指示文や引用マーカーを本文へ混入させない。
- `★` 指摘を消す場合は、指摘内容・判断・対応先を `review-tasks.md` 等へ残す。

## Agent の扱い

`agents/` 配下の各 Agent は、テンプレートを正本として使う。
過去章や古いステップ番号を正本にしてはならない。
Agent が章をレビューするときは、`rules/checklist.md` の読者体験、図解、フェーズ別の基準を必ず確認する。
Agent の記載が `AI_HANDOFF.md`、`templates/chapter-template.md`、`rules/checklist.md` と食い違う場合は、Agent 側を疑い、正本へ合わせる。

## 完了条件

章またはルールを修正したら、次を確認する。

1. `templates/chapter-template.md` と第0章が矛盾していない。
2. `rules/checklist.md` が同じ基準を持っている。
3. `rules/phase-consistency-check.md` の照合ラインを満たしている。
4. 仕様がコードに出る値・状態・判定条件・出力名へ紐づく粒度になっている。
5. 掲載コードの簡略化範囲が本文または図で説明されている。
6. 論点から外す処理の実際の動き、代替表現、割愛理由、設計論点への影響が補足されている。
7. `agents/` と `skills/` に古い構成名や古い判断基準が残っていない。
8. `script/audit_book.py --write-baseline` が0件で通る。
9. `git diff --check` が通る。
