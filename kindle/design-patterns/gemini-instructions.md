# Gemini への章執筆指示書

---

## 概要：この指示書の使い方

Gemini に各章を生成させるための手順書です。
第1〜8章（基本編）と第9〜12章（応用編）で添付ファイルが若干異なります。
各章は**独立したセッション**で実施してください（前章のファイルは不要です）。

---

## 共通：毎回必ず添付するファイル（6点固定）

| # | ファイル | 役割 |
|---|---|---|
| 1 | `agents/chapter-agent.md` | 執筆手順・生成ルール・自己採点チェック |
| 2 | `CLAUDE.md` | 品質ルール・コードルール・読者像・スタイル原則 |
| 3 | `rules/design-decision-guide.md` | S5〜S8の思考モデル（案0〜案4・コスト天秤） |
| 4 | `templates/chapter-template.md` | 章の構造テンプレート（第一部用） |
| 5 | `design-philosophy.md` | 設計思想カタログ（全設計問いへの回答定義） |
| 6 | `shared/skills/author-voice.md` | 著者の人格・文体・禁止表現 |

> **第9〜12章（応用編）を生成する場合は `templates/chapter-template2.md` も追加で添付してください。**

## 章の執筆手順（1セッション内でこの順に送る）

### ① セッション開始（最初に1回だけ送る）

**第一部（第1〜8章）の場合：**

```
添付ファイルをすべて読んでください。
これから第X章を、S0〜S8のステップ別に作成します。
テンプレートは chapter-template.md を使ってください。

章タイトル：第X章　【パターン名】
ドメイン：【ドメイン名（CLAUDE.mdのドメイン割り当て表を確認）】

執筆時は以下を必ず守ってください：
- 各ステップの冒頭に「なぜこのステップをやるか」を1文入れる
- 各ステップの末尾に「次のステップへの橋渡し文」を入れる
- コードブロックの直後に「このコードで分かること」を1文入れる
- パターン名（〇〇パターン）はS6以前に絶対に出さない

準備ができたら「了解しました」とだけ返してください。
```

**第二部（第9〜12章）の場合：**

```
添付ファイルをすべて読んでください。
これから第X章を、S0〜S8のステップ別に作成します。
テンプレートは chapter-template2.md（応用編）を chapter-template.md の S0〜S8 構造に差分適用して使ってください。

章タイトル：第X章　【〇〇パターン】×【△△パターン】
ドメイン：【ドメイン名】

執筆時は以下を必ず守ってください：
- 各ステップの冒頭に「なぜこのステップをやるか」を1文入れる
- 各ステップの末尾に「次のステップへの橋渡し文」を入れる
- コードブロックの直後に「このコードで分かること」を1文入れる
- パターン名はS6以前に絶対に出さない

準備ができたら「了解しました」とだけ返してください。
```

### ② ステップ別指示（順番に送る）

```
第X章のS0（クラス構成と責任を読む）を作成してください。
事実のみを書いてください。仮説・パターン名・評価はまだ不要です。
```

```
第X章のS1（実装コードを読む）を作成してください。
責任チェック表はS1段階では「観察のみ」です。
コード内に「// ← 知らなくていい」コメントはまだ入れないでください。
```

```
第X章のS2（仮説を立て、関係者に確認する）を作成してください。
仮説テーブル → ヒアリング会話シーン → 確定テーブルの順で書いてください。
S2末尾に「〇〇を採用する」という設計の決断は書かないでください（決断はS7）。
```

```
第X章のS3（変更シミュレーション）を作成してください。
「痛みを確認しましょう」ではなく「変更を試みてみましょう」という中立的な書き方にしてください。
```

```
第X章のS4（原因分析）を作成してください。
末尾に「ケーブルで考える」セクションとImagePromptを入れてください。
S4の最後に「○○と△△は変わる理由が異なるため分けるべき」という判断文を入れてください。
```

```
第X章のS5（課題を定義する）を作成してください。
接続点の特定・非機能制約表・クライアント影響範囲・課題まとめ表（4列）を入れてください。
「ホットパス」は初出時に「（頻繁に呼び出されるコードパス）」の括弧説明を付けてください。
```

```
第X章のS6（対策案の検討）を作成してください。
案0〜案4（5案）を全て書いてください。
S6では結論を出さないでください（どの案を採用するかはS7で決めます）。
```

```
第X章のS7（対策選定・コスト天秤）を作成してください。
評価軸を先に宣言し、案0〜案4の全5案を比較表で示してから採用案を決めてください。
パフォーマンスが問題になる場合はVETO（足切り）として扱い、残りを3軸スコアリングしてください。
耐久テスト・過剰コード例・使う/使わない場面を含めてください。
```

```
第X章のS8（対策実施）を作成してください。
BatchApplication + main() を含む最終コード全体を書いてください。
「// ← 知らなくていい」コメントはここで初めて使えます。
変更シナリオ表（変わるクラス・変わらないクラス）と最終責任テーブルを入れてください。
```

```
整理・振り返り・パターン解説を作成してください。
整理セクションには「9ステップとこの章でやったことの対応表」を入れてください。
振り返りには「得られること1〜4がどのステップで示されたか」の確認表を入れてください。
```

## 品質チェック：生成後に確認すること

以下の点は Gemini が守れていないことが多い。生成されたテキストを確認して、問題があれば指摘して修正させる。

| チェック項目 | 確認方法 |
|---|---|
| パターン名がS6より前に出ていないか | S0〜S5の全文を検索 |
| S1のコードに `// ← 知らなくていい` が混入していないか | S1のコードブロックを確認 |
| S3が「痛みを確認しましょう」になっていないか | S3の冒頭文を確認 |
| S2末尾に採用決断が書かれていないか | S2の末尾を確認 |
| 各ステップ冒頭に「なぜやるか」の1文があるか | 各ステップの最初の文を確認 |
| 案0〜案4の5案が全て揃っているか | S6の見出しを確認 |
| S6に「〇〇を選びます」という結論が混入していないか | S6の末尾を確認 |
| 「局所化」の初出に括弧説明があるか | 全文検索 |
| ベタ褒め・激責めの表現がないか | 「完璧」「美しい」「ひどい」等を検索 |
| 技術用語の初出に説明があるか | ホットパス・vtable等の初出箇所を確認 |

## トラブルシューティング

- **出力が途中で切れた場合**：「続きを書いてください」と送る
- **パターン名が早出しされた場合**：「S6より前にパターン名を出さないでください。該当箇所を修正してください」と送る
- **橋渡し文がない場合**：「各ステップの末尾に次のステップへの橋渡し文を追加してください」と送る
- **案0〜案4が揃っていない場合**：「案0（現状維持）から案4（抽象×間接）の5案を全て書いてください」と送る
