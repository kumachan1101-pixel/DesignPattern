# Gemini への章執筆指示書

---

## 概要：この指示書の使い方

Gemini に各章を生成させるための手順書です。
第1〜8章（基本編）と第9〜12章（応用編）で添付ファイルが若干異なります。
各章は**独立したセッション**で実施してください（前章のファイルは不要です）。

---

## 共通：毎回必ず添付するファイル（6点固定）

| # | ファイル | 役割 |
|---|---|---|
| 1 | `agents/chapter-agent.md` | 執筆手順・生成ルール・自己採点チェック |
| 2 | `CLAUDE.md` | 品質ルール・コードルール・読者像・スタイル原則 |
| 3 | `rules/design-decision-guide.md` | フェーズ5〜7の思考モデル（接続点・案0〜案4・コスト天秤・対策実施） |
| 4 | `templates/chapter-template.md` | 章の構造テンプレート（第一部用） |
| 5 | `design-philosophy.md` | 設計思想カタログ（全設計問いへの回答定義） |
| 6 | `shared/skills/author-voice.md` | 著者の人格・文体・禁止表現 |

> **第9〜12章（応用編）を生成する場合は `templates/chapter-template2.md` も追加で添付してください。**

### ② フェーズ別指示（順番に送る）

```
第X章のフェーズ1（前半）：1-1〜1-5（システムの背景・仕様表・クラス構成図・責任配置テーブル・依存グラフ）を作成してください。
事実のみを書いてください。仮説・パターン名・評価はまだ不要です。
```

```
第X章のフェーズ1（後半）：1-6〜1-8（実装コード・実行結果・責任チェック表）を作成してください。
責任チェック表はこの段階では「観察のみ」です。❌/✅の判定はまだしないでください。
コード内に「// ← 知らなくていい」コメントはまだ入れないでください。
```

```
第X章のフェーズ2：2-1〜2-4（届いた変更要求・仮説テーブル・ヒアリング・確定テーブル）を作成してください。
仮説テーブル（2-2）→ ヒアリング会話シーン（2-3）→ 確定テーブル（2-4）の順で書いてください。
フェーズ2末尾に「〇〇を採用する」という設計の決断は書かないでください（決断はフェーズ6後半）。
ヒアリングで挙がった「将来のリスク」は必ず記録してください（フェーズ6-10の伏線になります）。
```

```
第X章のフェーズ3：3-1〜3-3（変更シミュレーション・変更影響グラフ・痛みの言語化）を作成してください。
「痛みを確認しましょう」ではなく「変更を試みてみましょう」という中立的な書き方にしてください。
変更影響グラフは Mermaid の graph LR 形式で書いてください。
```

```
第X章のフェーズ4：4-1〜4-3（観察→原因テーブル・変わるもの/変わらないものテーブル・ケーブルで考える）を作成してください。
4-3（ケーブルで考える）の末尾に「○○と△△は変わる理由が異なるため分けるべき」という判断文と ImagePrompt を入れてください。
```

```
第X章のフェーズ5：5-1〜5-4（接続点の特定・非機能制約表・クライアント影響範囲・課題まとめ表）を作成してください。
課題まとめ表は「接続点・分けた理由・非機能制約・クライアント影響」の4列で書いてください。
「ホットパス」は初出時に「（頻繁に呼び出されるコードパス）」の括弧説明を付けてください。
```

```
第X章のフェーズ6（前半）：6-1〜6-6（2×2マトリクス・案0〜案4）を作成してください。
案0〜案4の5案を全て連番で並列に書いてください。
この段階では結論を出さないでください（採用案はフェーズ6後半で決めます）。
パターン名はこのどこかの案で初登場させてください（フェーズ5以前には出さないでください）。
```

```
第X章のフェーズ6（後半）：6-7〜6-10（評価軸・コスト天秤・採用決定・耐久テスト）を作成してください。
評価軸（6-7）を先に宣言してから比較表（6-8）を示し、採用案（6-9）を決めてください。
耐久テスト（6-10）では、フェーズ2のヒアリングで挙がった将来リスクを「その変化が来た」として実演してください（伏線回収）。
過剰コード例と「使う/使わない場面」を含めてください。
```

```
第X章のフェーズ7：7-1〜7-3（最終コード・変更影響グラフ改善後・変更シナリオ表）を作成してください。
BatchApplication + main() を含む最終コード全体を書いてください。
「// ← 知らなくていい」コメントはここで初めて使えます。
変更影響グラフはフェーズ3（3-2）と同じシナリオで描き、変更が局所化された様子を示してください。
```

```
整理・振り返り・パターン解説を作成してください。
整理セクションには「7フェーズとこの章でやったことの対応表」を入れてください。
振り返りには「得られること1〜4がどのフェーズで示されたか」の確認表を入れてください。
```

## 品質チェック：生成後に確認すること

以下の点は Gemini が守れていないことが多い。生成されたテキストを確認して、問題があれば指摘して修正させる。

| チェック項目 | 確認方法 |
|---|---|
| パターン名がフェーズ6より前に出ていないか | フェーズ1〜5の全文を検索 |
| フェーズ1のコードに `// ← 知らなくていい` が混入していないか | フェーズ1（1-6）のコードブロックを確認 |
| フェーズ3が「痛みを確認しましょう」になっていないか | フェーズ3（3-1）の冒頭文を確認 |
| フェーズ2末尾に採用決断が書かれていないか | フェーズ2（2-4）の末尾を確認 |
| 各フェーズ冒頭に「なぜやるか」の1文があるか | 各フェーズの最初の文を確認 |
| 案0〜案4の5案が全て揃っているか | フェーズ6（6-2〜6-6）の見出しを確認 |
| フェーズ6（前半）に「〇〇を選びます」という結論が混入していないか | フェーズ6（6-6）の末尾を確認 |
| 「局所化」の初出に括弧説明があるか | 全文検索 |
| ベタ褒め・激責めの表現がないか | 「完璧」「美しい」「ひどい」等を検索 |
| 技術用語の初出に説明があるか | ホットパス・vtable等の初出箇所を確認 |

## トラブルシューティング

- **出力が途中で切れた場合**：「続きを書いてください」と送る
- **パターン名が早出しされた場合**：「フェーズ6より前にパターン名を出さないでください。該当箇所を修正してください」と送る
- **橋渡し文がない場合**：「各フェーズの末尾に次のフェーズへの橋渡し文を追加してください」と送る
- **案0〜案4が揃っていない場合**：「案0（現状維持）から案4（抽象×間接）の5案を全て書いてください」と送る
