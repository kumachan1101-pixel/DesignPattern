# 読者ペルソナ定義
# Kindle本・Note記事の両方で参照する共通定義

---

## ペルソナA：設計に悩む現役エンジニア（メインターゲット）

### 基本プロフィール

- 開発経験 3〜8年
- 動くコードは書ける。レビューで「設計を直して」と言われる
- 技術書を買う習慣はある。GoFの本も買ったが挫折した

### 日々の悩み

- 「リファクタリングしろと言われても、どこから手をつければいいか」
- 「パターンを読んで理解した気になるが、自分のコードに当てはめられない」
- 「先輩のコードを見て"なるほど"と思うが、同じように書けない」
- 「仕様変更のたびにコードが壊れる恐怖がある」

### このコンテンツに期待していること

「なぜそう設計するのかの思考プロセス」を知りたい。
答えではなく、考え方を学びたい。
でも「難しすぎる話」ではなく、「自分でもできそう」と感じたい。

### このペルソナへの語り方

「あなたも同じ悩みを持っていた著者」として書く。
解決策を押しつけない。「こういう考え方もある」という提示。
「簡単です」は禁句。「私もここで詰まりました」が刺さる。

---

## ペルソナB：チームをまとめるリーダー

### 基本プロフィール

- 開発経験 8〜15年
- 自分の設計判断には自信がある
- チームメンバーに「なぜこの設計の方がいいか」を説明するのが難しい

### 日々の悩み

- 「レビューで指摘できるが、理由を言語化できない」
- 「パターンは知っているが、いつ使うかの基準が曖昧」
- 「メンバーが設計を学べる場を作りたいが、いい教材がない」

### このコンテンツに期待していること

「チームへの説明に使える言葉」が欲しい。
費用対効果の判断軸が明確になれば、意思決定の根拠にできる。

### このペルソナへの語り方

「一緒に考える同僚」として書く。
答えを断言しない。「チームで話し合う価値がある」という表現が響く。

## KindleとNoteでのターゲット差

| | Kindle本 | Note記事 |
|:---|:---|:---|
| 主なペルソナ | A・B 両方 | ペルソナA 中心 |
| 読む状況 | じっくり・連続して読む | スキマ時間・SNSから流入 |
| 期待する価値 | 体験・理解の深化 | 共感・発見・気づき |
| 長さ | 1章 5,000〜8,000字 | 1記事 1,500〜3,500字 |
| 動機 | 「設計を本気で学びたい」 | 「自分と同じ悩みを持つ人がいる」 |

## 生成時のチェック問い

記事・章を生成したら、以下を自問する：

1. ペルソナAが読んで「自分のことが書いてある」と感じるか
2. 「簡単ですよ」という上から目線になっていないか
3. ペルソナAが「自分でもやってみよう」と思える終わり方か
4. ペルソナBが「これをチームで話し合おう」と使える内容か
