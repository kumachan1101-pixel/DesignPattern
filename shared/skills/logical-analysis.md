# skills/logical-analysis.md
# ロジカルに分析するスキル

---

## このスキルの役割

コード・設計問題・ビジネス要求を受け取ったとき、
「なんとなくこう思う」ではなく、
**観察 → 原因 → 対処** という論理的な順序で分析する手順を定義する。

このスキルは以下の場面で使う：
- 章の 〇.3〜〇.5（課題発見→原因特定→対策案の導出）を執筆するとき
- 新しいパターンや設計問題を分析するとき
- アイデアや課題を整理して、根拠のある複数案を導出したいとき

---

## フレームワーク：3段階の論理展開

### 第1段階：事実の観察
> 「何が起きているか」を感想なしに列挙する

**やること**：
- コードや状況を見て「起きていること」を動詞文で書く
- 「〇〇に手を入れると〇〇が壊れる」「〇〇が増えるたびに〇〇も増える」など
- 1つの観察に1つの事実だけ含む

**やらないこと**：
- 「悪いコードだ」→ 評価（事実ではない）
- 「リファクタリングが必要だ」→ 対処（まだ早い）
- 「明らかに設計が間違っている」→ 断定（根拠が必要）

> 品質チェック：「この観察は、コードを見せれば他の人も同意するか？」

---

### 第2段階：原因の特定
> 「なぜそうなるか」の構造的な根拠を1つ突き止める

**手順**：
1. 第1段階の観察を並べる
2. 「これらに共通する構造的な原因は何か？」と問う
3. 原因は「**〇〇と△△が同じ場所にいる**」という形で表現する

**設計問題での4つの切り口**：

| 切り口 | 原因の形 | 代表パターン |
|:---|:---|:---|
| A：変化の混在 | 変わるものと変わらないものが同じ場所にいる | Strategy / Command / Decorator / Observer など |
| B：依存の過多 | 使う側が知らなくていいことまで知っている | Facade |
| C：状態と振る舞いの混在 | 状態と、そのときの振る舞いが同じ場所にいる | State |
| D：生成と利用の混在 | 何を作るかの判断と、どう使うかが同じ場所にいる | Factory Method |

**確認の問い**：
「この原因が解消されれば、第1段階の観察はなくなるか？」

なくならないなら、原因の特定が浅い。もう一段「なぜ？」を掘る。

> 品質チェック：「原因を解消した世界を想像したとき、困難が消えているか？」

---

### 第3段階：対処の導出
> 「原因に対して何ができるか」を複数、論理的に列挙する

**原則**：
- 対処は「原因に対して何をするか」から導く
- 「なんとなく思いついたアイデア」を出発点にしない
- 導いた対処はどれも原因への正当な対応。状況で使い分ける

**手順**：

```
1. 第2段階の原因を一言で書く
   例：「変わるものと変わらないものが同じ場所にいる」

2. 「この原因を解消するには、どんなアプローチが考えられるか」を問う

3. アプローチごとに書く：
   - なぜこれが原因への対処になるか（1〜2文）
   - コスト（実装の手軽さ・学習コスト）
   - 適する状況（どんな条件でこれが合うか）
```

**注意**：アプローチAを「失敗する試み」として扱わない。
「状況Xにはアプローチが合う」「状況Yにはアプローチが合う」
という形で並列に提示する。

> 品質チェック：「このアプローチは原因を解消するか？思いつきではないか？」

---

## 比較の論理：基準を先に宣言する

複数の対処を比べるときは「何を重視するか」を先に言葉にする。

**正しい順序**：
1. **基準を宣言する**（何が重要か、なぜ重要か）
2. **各案を基準で測る**（それぞれどうか）
3. **判断する**（この状況ではどれが合うか）

**逆にしてはいけない**：
- 「結論を先に決めて、後から理由を探す」比較は論理的でない
- 「AよりBが優れている」という絶対評価は使わない
- 「この状況の基準に照らすとBが合っている」という相対評価にする

**比較基準の例**（設計問題向け）：

| 基準 | 問い |
|:---|:---|
| テストの独立性 | 各部品を、他の部品を知らずにテストできるか |
| 変更の局所性 | 1か所の変化が影響を広げないか |
| チームの分担 | 別の担当者が互いに干渉せず作業できるか |
| 変化の継続性 | 今後も同じ種類の変化が増え続けるか |
| 実装コスト | 今の締切と規模に見合うか |

## 論理の品質チェックリスト

### 観察の品質
- [ ] 観察は事実か（意見・評価・感情が混入していないか）
- [ ] 1観察に1事実か
- [ ] 「悪い」「間違っている」などの断定がないか

### 原因の品質
- [ ] 原因は観察から導かれているか（飛躍がないか）
- [ ] 「〇〇と△△が同じ場所にいる」という構造として表現されているか
- [ ] 原因が解消されれば観察もなくなるか（因果が通っているか）

### 対処の品質
- [ ] 各対処は原因から論理的に導かれているか
- [ ] 「失敗する試み」ではなく「適する状況が異なる案」として提示されているか
- [ ] 複数の対処が「切り口の違い」で区別されているか

### 比較の品質
- [ ] 基準は比較の前に宣言されているか
- [ ] 各案を同じ基準で測っているか
- [ ] 「この状況では〇〇が合う」という相対評価になっているか
- [ ] 「状況が変われば答えも変わる」ことを認めているか

## よくある論理の落とし穴

### 落とし穴1：観察と原因を混ぜる

❌ 「コードが複雑だから、設計を変えるべき」
　→ 「複雑」は評価。なぜ複雑になるのかを観察で分解してから原因へ。

✅ 「変更のたびに引数が増える（観察）。
　　変わるルールと変わらない骨格が同じ場所にいる（原因）。」

---

### 落とし穴2：対処を先に思いつく

❌ 「Strategyパターンを使えばいい」から逆算して原因を後付けする
　→ 答えが先にあると、原因の分析がそれに引っ張られる。

✅ 観察 → 原因 → 対処 の順序を崩さない。
　 パターン名は「辿り着いた構造のラベル」として最後に出る。

---

### 落とし穴3：基準なしで比較する

❌ 「アプローチAよりBの方が良い設計です」
　→ 何の基準で？どんな状況で？

✅ 「テストの独立性とチームの分担を重視するこの状況では、
　　アプローチBが合っています。
　　ルールが少なく1人で管理する場合はAが割に合います。」
