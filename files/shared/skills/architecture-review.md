# skills/architecture-review.md
# アーキテクチャ品質レビュースキル（全プロジェクト共通）

---

## このスキルの目的

生成されたコードや章の内容を「アーキテクチャ専門家」の視点でレビューする。
フォーマット・文体は他のエージェントが担当する。このスキルは**設計の質**だけを見る。

---

## 評価観点 A：単一責任原則（SRP）

**問い：各クラスの「変わる理由」は1つだけか？**

チェック方法：
1. コードに登場する全クラスを列挙する
2. 各クラスの「責任を一文で表せるか」を確認する（表せないなら複数責任の疑い）
3. 「このクラスを変更するとき、誰に話を聞きに行くか？」の答えが1人以下か確認する

問題パターン：
- `PayrollFacade` が `PayrollServiceImpl`・`LaborMgmtServiceImpl`・`PdfServiceImpl` の3つの具体クラスを持っている
  → 3サービスが変わるたびにFacadeが変わる = 責任が3つある
- 統括クラスが外部サービスの仕様（URL・認証方式・フィールド名）を知っている
  → 統括クラスの責任外の知識が混入している

修正方向：
- 変動する部分ごとにインターフェースを切る
- 各クラスが「1つのインターフェースに依存する」形にする

## 評価観点 B：依存性逆転原則（DIP）

**問い：呼び出し元は具体クラスではなく抽象（インターフェース）に依存しているか？**

チェック方法：
1. 各クラスのコンストラクタ引数・メンバ変数の型を確認する
2. 型が具体クラス名（`PayrollServiceImpl` 等）であれば DIP 違反
3. 型がインターフェース名（`IPayrollService` 等）であれば OK

問題パターン：
```cpp
// ❌ DIP違反：具体クラスを直接持っている
class PayrollFacade {
    PayrollServiceImpl* payroll_;  // ← 実装クラスに依存
};

// ✅ DIP準拠：インターフェースだけを知っている
class PayrollFacade {
    IPayrollService* payroll_;  // ← 抽象に依存
};
```

合わせて確認：
- Facade クラス自身もインターフェース（`IPayrollFacade`）を持っているか
- 統括クラス（MonthlyBatch相当）は Facade のインターフェースだけを知っているか

## 評価観点 C：インターフェース命名

**問い：インターフェース名はビジネス責任で命名されているか？実装手段が名前に入っていないか？**

| ❌ 実装手段の名前 | ✅ ビジネス責任の名前 | 問題 |
|---|---|---|
| `IPdfService` | `IPaySlipOutputService` | PDF → Excel に変えたとき名前が嘘になる |
| `IEmailNotifier` | `INotifier` | SMS に変えたとき名前が嘘になる |
| `IMySqlRepository` | `IOrderRepository` | DB変更で名前が嘘になる |
| `IHttpPayrollClient` | `IPayrollService` | 通信プロトコル変更で名前が嘘になる |

**原則**：インターフェース名は「何をする責任か」を表す。「どうやるか」は表さない。

## 評価観点 D：組み立て点（Composition Root）

**問い：具体クラスの生成・注入は1箇所に集約されているか？**

チェック方法：
1. コード全体で `new ConcreteClass()` または具体クラスのインスタンス化がどこに現れるかを確認する
2. BatchApplication（または同等クラス）以外の場所でインスタンス化されていれば問題

問題パターン：
```cpp
// ❌ 統括クラスが具体クラスを組み立てている
class MonthlyBatch {
    void run() {
        PayrollServiceImpl payroll;  // ← 業務フロー統括の責任外
        // ...
    }
};

// ✅ BatchApplicationだけが具体クラスを知る
class BatchApplication {
    void run(int year, int month) {
        PayrollServiceImpl payroll;   // ← ここだけが知っていればよい
        MonthlyBatch batch(&facade);
        batch.run(year, month);
    }
};
```

合わせて確認：
- `main()` は `BatchApplication` をキックするだけか（`new ConcreteClass()` が出ていないか）

## 評価観点 E：型の安定性

**問い：インターフェースのパラメータ型に「変わるリスク」があるか？その場合、選択肢を検討しているか？**

背景：
`int employeeId` という型が複数のインターフェースをまたいで使われている場合、
型が `double` や `string` に変わるとすべてのインターフェースが変更になる。
これは Facade パターンでは守れない。

確認ポイント：
1. 複数のインターフェースをまたぐ共通型（ID・金額・日付等）があるか
2. その型が将来変わるリスクについてストーリー内で言及があるか
3. 型が不安定な場合、以下の選択肢の検討があるか

| 選択肢 | 内容 | メリット | デメリット |
|---|---|---|---|
| 型を合意・固定 | 関係者と合意して型を決める | 型安全・可読性高 | 変更時に全インターフェースが変わる |
| DTO / データ型を別定義 | `EmployeeId` 型を独自に定義 | 変更箇所がDTO 1箇所に絞られる | 型定義が増える |
| void\* / 不完全型 | 型の知識をインターフェースに持たせない | 最大の変更耐性 | 型安全を失う・デバッグ困難 |
| 基底クラスで受け取る | 共通基底クラスを用意して渡す | OOP的アプローチ | 継承階層が増える |

**重要**：この選択はトレードオフであり、チームで合意して決める。正解はない。

## 評価観点 F：契約の合意プロセス

**問い：インターフェースの契約（型・シグネチャ・責任境界）が関係者との合意に基づいているか？**

確認ポイント：
- 変動/不変を確定する前（ステップ2）に、関係者ヒアリングの場面があるか
- 「このサービスのAPIは将来変わる予定はありますか？」という問いが示されているか
- ヒアリング結果が変動/不変テーブルの根拠として明記されているか

問題パターン：
- 「コードを読んだだけ」で変動/不変を断定している
- インターフェースの設計者が1人で決めたことが示唆される記述がある

---

## 評価観点 G：変更耐性の実証

**問い：「変更シナリオ表」で、各変更が本当に1クラスに収まっているか？**

確認方法：
1. 変更シナリオ表の各行を確認する
2. 「変わるクラス」が1〜2クラスに収まっているか確認する（多ければ設計の問題）
3. 業務フロー統括クラス（MonthlyBatch相当）が「変わらないクラス」に含まれているか確認する

合格基準：
- 外部サービスの仕様変更 → 対応する実装クラス1つだけが変わる
- 利用するサービスの差し替え → BatchApplication だけが変わる
- 業務フローの変更 → 統括クラスだけが変わる（サービス実装は変わらない）

## 出力形式

```json
{
  "agent": "architecture-review-agent",
  "chapter_file": "output/chapter02.md",
  "status": "ok|issues-found",
  "issues": [
    {
      "section": "〇.7",
      "severity": "error|warning|info",
      "principle": "SRP|DIP|命名|CompositionRoot|型安定性|契約合意|変更耐性",
      "description": "具体的な問題の説明",
      "evidence": "問題のあるコードや文章の引用",
      "suggestion": "修正案"
    }
  ],
  "summary": {
    "total_issues": 0,
    "by_severity": { "error": 0, "warning": 0, "info": 0 },
    "by_principle": {
      "SRP": 0,
      "DIP": 0,
      "命名": 0,
      "CompositionRoot": 0,
      "型安定性": 0,
      "契約合意": 0,
      "変更耐性": 0
    }
  }
}
```

severity の判定基準：
- `error`：設計として明確に問題がある（DIP違反・SRP違反・具体クラスへの直接依存等）
- `warning`：状況によっては問題になる・選択肢の検討が不十分
- `info`：問題ではないが、読者への説明として加えると価値がある
