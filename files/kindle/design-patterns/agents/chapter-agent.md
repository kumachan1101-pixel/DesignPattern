# agents/chapter-agent.md
# 1章分を担当するエージェント（design-patterns 固有版）

---

## 役割

1章分を担当して完成させる。
他の章に依存しない。第0章の構造だけを前提とする。

---

## 受け取る引数

- `chapter_number` : int（例：2）
- `pattern_file` : patterns/{name}.yaml（例：patterns/facade.yaml）

※ 前章ファイルへの参照は不要。各章は完全独立。

---

## 実行手順

### ステップ1：インプットを読む

1. `../../shared/skills/author-voice.md` を読む（最重要・最初に読む）
2. `CLAUDE.md` を読む（特に「執筆品質ルール」セクションを熟読する）
3. 指定された `pattern_file` を読む
4. `templates/chapter-template.md` を読む

### ステップ2：章を生成する

chapter-template.md の構造に従い、以下の順序で節を生成する：

```
── ステップ1：現状把握 ──
〇.1 今のシステムの仕様とコードの構造
      └ 核心一文「要するに〜するパターン」（〇.1見出し直後、システム概要の前）
      └ システム概要（2〜3文）
      └ 仕様テーブル（機能・入力・出力）
      └ 【起点コード】（コメントに実行結果例を含む）
〇.2 届いた変更要求
      └ 変更要求の対話
      └ 検索シーン＋grep根拠（「〇箇所」の根拠）

── ステップ2：課題の発見 ──
〇.3 変更しようとしたときに現れる困難
      └ 困難1、困難2（事実の観察、評価なし）
      └ 問い（「ここで答えは出しません」「ここで答えは出さない」は使わない）

── ステップ3：原因特定 ──
〇.4 困難の根本にあるもの
      └ 原因の説明（「〇〇と△△が同じ場所にいる」形式）
      └ 「知らなくていいこと」を行レベルのコメントで明示
      └ テーブル（本来知っていればよいこと vs 知らなくていいこと）
      └ 問いかけのブロック引用

── ステップ4：対策案の検討 ──
〇.5 最初の試み：【試行コード】
      └ 導入文（「最初の試みとして〜」：「まず〜」は使わない）
      └ 試行コード本体
      └ 依存注入コード（コンストラクタ定義）
      └ 試行コードのテスト（スタブ・モック・アサーションの説明付き）
      └ 残る課題の明示

〇.6 発想の転換：【解決コード】
      └ 橋渡し文（試行の残課題を受けた発想の転換）
      └ インターフェース
      └ 実装クラス
      └ 呼び出し元の変化
      └ 依存注入コード（コンストラクタ定義＋本番組み立て例）
      └ Mermaid図＋テキスト補足
      └ パターン名の紹介
      └ 解決コードのテスト

── ステップ5：天秤にかける・決断する ──
〇.7 比較の基準を先に宣言する
〇.8 試行コードと解決コードをテストで比較する
      └ 比較のまとめテーブル（基準・試行・解決・実装コスト）
〇.9 より難しい変化への耐久テスト
      └ 新たな変更要求
      └ 【深化コード】＋テスト
〇.10 使う場面・使わない場面
      └ 【過剰コード】
      └ 状況ごとの選択指針テーブル
      └ 「守る一線」のブロック引用

── 整理 ──
〇.11 5ステップとこの章でやったこと
```

### ステップ3：自己採点

CLAUDE.md の品質チェックリストを**全項目**確認する。
未達の項目があれば該当節を再生成する。

特に以下は必ず確認する：

**構造チェック**
- ステップ4が試行コード→解決コードの1方向になっているか（並列アプローチ禁止）
- 〇.1の直後に「要するに〜するパターン」の核心一文があるか
- コードラベルがCLAUDE.md定義のものだけ使われているか

**執筆品質チェック**
- 禁止表現がないか：
  - 文頭「まず〜」
  - 「まず率直に認めたい」「まず認めたい」等の上から目線表現
  - 「ここで答えは出しません」「まだ答えを出しません」「ここで答えは出さない」
  - 「次章では〜」「次章で〜を扱います」等の次章予告
- 仕様がテーブルで示されているか
- 数字（「N箇所」）に根拠が添えられているか
- 「知らなくていいこと」が行レベルで明示されているか
- 依存注入コードが含まれているか（コンストラクタ定義＋本番組み立て例）
- テスト文法（スタブ・モック・アサーション）の説明が初出箇所にあるか

**著者の人格チェック**
- 他章への言及がないか（章の独立性）
- 次章予告がないか
- 著者の口癖が3〜5箇所、不自然でなく入っているか
- 執筆指示コメント（`<!-- -->`）が出力に残っていないか

### ステップ4：ファイルを書き出す

`output/chapter{nn}.md` に書き出す。
（例：chapter_number=2 なら `output/chapter02.md`）
