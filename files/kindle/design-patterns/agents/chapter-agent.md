# agents/chapter-agent.md
# 1章分を担当するエージェント（design-patterns 固有版）

---

## 役割

1章分を担当して完成させる。
他の章に依存しない。第0章の構造だけを前提とする。

---

## 受け取る引数

- `chapter_number` : int（例：2）
- `pattern_file` : patterns/{name}.yaml（例：patterns/facade.yaml）

※ 前章ファイルへの参照は不要。各章は完全独立。

---

## 実行手順

### ステップ1：インプットを読む

1. `../../shared/skills/author-voice.md` を読む（最重要・最初に読む）
2. `CLAUDE.md` を読む（特に「執筆品質ルール」「コードルール」「読者視点チェック」を熟読する）
3. 指定された `pattern_file` を読む
4. `templates/chapter-template.md` を読む

---

### ステップ2：章を生成する

chapter-template.md の構造に従い、以下の順序で節を生成する。
ステップ0〜7のH2見出しを必ず使う。順序を崩さない。

```
── ステップ0：視点のチューニング ──
〇.0 共通の問いと変動・不変の仮説
      └ 全パターン共通の問い（固定文。パターン固有にしない）
      └ 変動・不変の仮説テーブル（コードを読む前に立てる）

── ステップ1：現状把握 ──
〇.1 今のシステムの仕様とコードの構造
      └ 核心一文「要するに〜するパターン」（〇.1見出し直後、システム概要の前）
      └ システム概要（2〜3文）
      └ 仕様テーブル（機能・入力・出力）
      └ 変更前クラス図（classDiagram）
      └ 【起点コード】（❌ ラベル・基本OOP文法のみ・専門知識排除）
      └ main()実行結果（「このコードは正しく動く。問題は構造にある」を示す）
〇.2 届いた変更要求
      └ 変更要求の対話
      └ 依存グラフ graph TD（依存の広がりをマクロで示す）
      └ 「〇箇所」の根拠（ファイル名列挙またはgrep例）

── ステップ2：変動と不変の峻別 ──
〇.3 ステップ0の仮説をコードで検証・確定する
      └ 仮説との照合（一致・修正が必要だった点）
      └ 変動/不変の確定テーブル（分類・具体的内容・変わるタイミング・設計への影響）
      └ 設計の決断（不変を抽象として固定する根拠）

── ステップ3：課題分析 ──
〇.4 変更しようとしたときの困難と痛み
      └ 痛みのシナリオ（変更要求を実装しようとしたときの困難）
      └ 変更影響グラフ graph LR（改善前：変更が飛び火する様子）
      └ 問いかけのブロック引用（なぜこうなるのか、という問い）

── ステップ4：原因分析 ──
〇.5 困難の根本にあるもの
      └ なぜなぜ分析テーブル
      └ 「〇〇と△△が同じ場所にいる」という構造的原因の言語化
      └ 「知らなくていいこと」を行レベルのコメント（`// ← 知らなくていい`）で明示
      └ 知っていればよいこと vs 知らなくていいこと のテーブル

── ステップ5：対策案の検討 ──
〇.6 最初の試み：【試行コード】
      └ 導入文（「最初の試みとして〜」：「まず〜」は使わない）
      └ 試行コード本体（基本OOP文法のみ）
      └ 依存注入コード（コンストラクタ定義）
      └ 試行コードのテスト（スタブ・モック・アサーションの説明付き）
      └ 残る課題の明示（次の発想の転換への橋渡し）

〇.7 発想の転換：【解決コード】
      └ 橋渡し文（試行の残課題を受けた発想の転換）
      └ インターフェース（基本OOP文法：純粋仮想クラス）
      └ 実装クラス（基本OOP文法）
      └ 呼び出し元の変化
      └ 依存注入コード（コンストラクタ定義＋本番組み立て例）
      └ 変更後クラス図（classDiagram）＋変更前との対比説明（矢印の数の変化）
      └ パターン名の紹介

── ステップ6：天秤にかける ──
〇.8 比較の基準を先に宣言する
      └ 評価軸の宣言（何を・なぜ重視するか）
〇.9 比較と判断
      └ 比較テーブル（評価軸・改善前・パターン適用後）
      └ 適用判断フローチャート flowchart TD
〇.10 より難しい変化への耐久テスト
      └ 新たな変更要求
      └ 【深化コード】＋テスト
〇.11 使う場面・使わない場面
      └ 【過剰コード】
      └ 使う場面・使わない場面のテーブル

── ステップ7：決断と、手に入れた未来 ──
〇.12 解決後のコード
      └ ✅ 解決コード（基本OOP文法のみ）
      └ 変更前のmain()実行結果と同じ結果をテストで保証する（動作が変わっていないことを証明）
      └ 変更影響グラフ graph LR（改善後：変更が局所化された様子）

── 整理 ──
〇.13 8ステップとこの章でやったこと
      └ 各ステップでやったことの1行サマリーテーブル
      └ 「このプロセスの結果に〇〇パターンという名前がついた」でしめる
```

**コード生成時の必須ルール（CLAUDE.mdより）：**
- 使ってよい文法：class / virtual / コンストラクタ / メソッド呼び出し / if / for / 生ポインタ
- 使わない文法：lambda・スマートポインタ・templateメタプログラミング・C++17以降の機能
- コード例に専門知識（ファイルI/O・通信・特定フレームワーク）を混入しない
- 脳内の声は `// 💭 「…」` で示す（🧠 は使わない）

---

### ステップ3：自己採点

CLAUDE.md の品質チェックリストを**全項目**確認する。
未達の項目があれば該当節を再生成する。

**構造チェック（全てtrue必須）：**
- ステップ0〜7のH2見出しが揃っているか
- ステップ0で全パターン共通の問いを使っているか（パターン固有にしていないか）
- 変更前クラス図（ステップ1）と変更後クラス図（ステップ5）が両方あるか
- 依存グラフ・変更影響グラフ2枚・判断フローチャートが揃っているか
- コードにlambda・スマートポインタ・templateが混入していないか
- ステップ1の起点コードにmain()と実行結果があるか
- ステップ7に「変更前と同じ結果をテストで保証」するコードがあるか
- 1コード＋テストで完結してから次に進んでいるか
- 〇.1の直後に「要するに〜するパターン」の核心一文があるか

**執筆品質チェック：**
- 禁止表現がないか（文頭「まず〜」「ここで答えは出しません」「次章では〜」「〜すべき」等）
- 仕様説明がテーブルまたはコード実行例で示されているか
- 数字（「〇箇所」等）に根拠が添えられているか
- 「知らなくていいこと」が行レベルのコメントで明示されているか
- 依存注入コードが含まれているか（コンストラクタ定義＋本番組み立て例）
- テスト文法（スタブ・モック・アサーション）の説明が初出箇所にあるか
- 次章予告が書かれていないか

**読者視点チェック：**
- 列挙・比較で「なぜこれだけ？」と思わせる不完全な例示がないか
- 見出しと本文の内容が一致しているか
- クラス図・コード・本文でクラス名・メソッド名・用語が統一されているか
- 変更前クラス図とステップ1のコードが同じ構造を表しているか
- 変更後クラス図とステップ7のコードが同じ構造を表しているか
- ステップ2の変動/不変の仕分けとステップ5のインターフェース設計が矛盾していないか
- 前章の知識を前提にしていないか
- 専門用語（インターフェース等）の初出時に説明があるか

**著者の人格チェック：**
- 他章への言及がないか（章の独立性）
- 著者の口癖が3〜5箇所、不自然でなく入っているか
- 執筆指示コメント（`<!-- -->`）が出力に残っていないか

---

`output/chapter{nn}.md` に書き出す。
（例：chapter_number=2 なら `output/chapter02.md`）
