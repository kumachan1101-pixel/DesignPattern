# 【ソフトウェア設計】if文を条件分岐ではなく責任の境界として見る

結論：if文が複雑になったときは「誰が判定するべきか」を先に見るようにしている

if文が増えてきたときは、個人的にはまず「この判定を、この場所でやる必要があるのか」を見るようにしています。

昔は、条件分岐は仕様を実現するための普通の書き方だと思っていました。

もちろん、if文自体が悪いわけではありません。

ただ、最近はif文の中に、責任の混在や密結合の兆候が出ていることがあると感じるようになりました。

特に気にしているのは、次のような場合です。

* 判定に使っているデータの責任者が別にいる
* 判定条件が何度も変わりそう
* if文の中で複数の意味の条件をまとめて見ている
* 判定する人と、振る舞う人が同じ場所に混ざっている

こういう場合は、私はその場で条件を頑張って書くよりも、判定する責任を別の関数やクラスに寄せられないかを考えます。

## 困ったこと：仕様通りに動くif文なのに、あとから読むと判断の責任が見えなくなる

例えば、画面にボタンを表示するかどうかを判定する場面を想定します。

```java
if (userType == 1 && itemStatus != 3 && inputCount > 0) {
    showButton();
}
```

動きとしては問題ないかもしれません。

ただ、あとから見ると少し迷います。

この判定は、画面が決めることなのでしょうか。

それとも、ユーザー状態を管理している側が決めることなのでしょうか。

商品状態を管理している側が決めることなのでしょうか。

if文が増えると、仕様は満たしていても「その判断を誰が持つのか」が見えづらくなります。

## 迷った理由：if文は自然に書けるので、責任の混在に気づきにくい

if文はとても自然に書けます。

だからこそ、設計上の違和感に気づきにくいです。

「この条件ならこうする」と書けば動きます。

ただ、その条件に使っている情報が、別の部品の管理データだった場合、その場所は他人の事情を詳しく知りすぎているかもしれません。

最近は、ここを少し疑うようにしています。

## 試したこと：複雑な条件をその場で読まず、判定名に置き換えてみた

複雑なif文を見たときは、まず条件の中身をそのまま理解しようとせず、名前を付けられないか考えます。

```java
if (canShowButton(userType, itemStatus, inputCount)) {
    showButton();
}
```

これだけでも、読む側は少し楽になります。

ただ、ここで終わりにはしません。

次に考えるのは、canShowButton をどこに置くかです。

画面側に置くのか。

状態を管理する側に置くのか。

判定専用の部品に寄せるのか。

ここが設計判断だと思っています。

## どう整理したか：判定する人と振る舞う人を分けて考える

今は、if文を見たときに次のように分けて考えています。

| 見る観点 | 判断 |
|:---|:---|
| 判定に使うデータを誰が管理しているか | その責任者側に判定を寄せられないか考える |
| 判定条件が変わりやすいか | 条件を直接書かず、名前を付けて閉じ込める |
| 振る舞いだけが違うのか | 判定と実行を分ける |
| その場限りの単純な条件か | 無理に分けず、その場に残す |

判断フローにすると、こうです。

```
if文が複雑
  ↓
判定に使うデータの責任者は誰か
  ↓
自分ではない
  ↓
責任者側に「判定結果」を聞けないか考える
  ↓
自分は結果を見て振る舞うだけにする
```

## 今の判断：条件の意味が業務判断に近いときは、if文をその場に置かない

今の自分の判断としては、条件の意味が単なる表示都合ではなく、何かの状態判断に近い場合は、その場に直接if文を書かないようにしています。

例えば、

```java
if (isReadyToSubmit(form)) {
    enableSubmitButton();
}
```

のように、画面側は「送信できるか」という結果だけを見る形にします。

細かい条件は、フォームの状態を知っている側に寄せます。

逆に、単なる表示上の分岐で、その場で完結するものなら、無理に分けません。

```java
if (isVisible) {
    showLabel();
}
```

この程度なら、その場にあってもよいと考えています。

## 小さな学び：if文は処理ではなく、責任の境界を見る入口になる

if文を見るときに、以前は「条件が正しいか」だけを見ていました。

今はそれに加えて、「この判定をここで持つのは自然か」を見るようになりました。

複雑なif文は、単に読みづらいだけではなく、責任が混ざっているサインかもしれません。

すべてのif文を消したいわけではありません。

ただ、判定する人と振る舞う人が混ざっていると感じたときは、そこを分けるだけで構造が少し見えやすくなることがあります。

今も迷いながらですが、if文は設計の違和感を見つける入口として、かなり意識するようになりました。
