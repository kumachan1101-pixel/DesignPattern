# 【ソフトウェア設計】インターフェースの粒度、どこで区切るか毎回迷う

## インターフェースの粒度って、結局どこで区切ればいいのか迷う

インターフェースを設計していると、毎回かなり迷います。

「この責務で分けるべきなのか」
「まとめた方が使いやすいのか」
「分けすぎて逆に読みにくくなってないか」

設計本を読むと、
「依存を抽象化する」
「変更に強くする」
みたいな話はよく出てきます。

もちろんそれはそうなんですが、実際の実装だと、
"じゃあどの粒度でインターフェース切るの？"
がかなり難しい。

当時は、インターフェースを細かく分ければ設計が良くなると思っていた時期もありました。

でも実際にやってみると、今度はインターフェースだらけになって、何がしたいのか逆に読み取りづらくなることがありました。

## 「分ければ良い」でもなかった

例えば、ファイル保存を考えます。

最初はこういう感じにしたくなることがあります。

```csharp
public interface IFileNameProvider
{
    string GetFileName();
}
public interface IFileContentProvider
{
    string GetContent();
}
public interface IFileSaver
{
    void Save(string fileName, string content);
}
```

一見、責務が分かれていて綺麗に見えます。

でも実際に使う側を書くと、

```csharp
var fileName = fileNameProvider.GetFileName();
var content = contentProvider.GetContent();
fileSaver.Save(fileName, content);
```

みたいに、
呼び出し側が細かい組み立てを知ってしまう。

これ、変更が入ると結構つらいことがあります。

ファイル名ルールが変わる。
保存前に加工が必要になる。
保存形式が変わる。

そうなると、結局呼び出し側がいろいろ知り始める。

当時は、
「分離できているはずなのに、なぜか使いづらい」
みたいな感覚がありました。

## 「使う側が何を知るか」がかなり重要だった

そこから少しずつ意識するようになったのが、

「利用側に何を知ってほしいか」

でした。

例えば、

```csharp
public interface IReportExporter
{
    void Export(Report report);
}
```

こうすると、

```csharp
reportExporter.Export(report);
```

だけで済みます。

呼び出し側は、

* ファイル名
* 保存形式
* 保存場所
* 変換処理

みたいな細かいことを知らなくて良い。

この感覚はかなり大きかったです。

インターフェースって、
「機能を分ける」
よりも、

「利用側に余計なことを考えさせない」

の方が大事なのかもしれないと思うようになりました。

## ただ、隠しすぎると今度は変更しづらい

ただ、これも単純ではありませんでした。

全部ブラックボックスにすると、
今度は細かい制御が必要になった時につらくなる。

例えば、

```csharp
Export(report);
```

だけだと、

* PDFだけ変えたい
* 保存先だけ変えたい
* ファイル名ルールだけ変えたい

みたいな変更に弱くなることがあります。

なので結局、

* どこが今後変わりそうか
* どこを差し替えたいか
* どこを固定したいか

を考えないといけない。

ここがかなり難しい。

## 「抽象化したい」のに、未来が分からない

特に難しいのが、
未来の変更なんて読めないことです。

今はシンプルで良いと思っても、
後から仕様変更が来ると崩れる。

逆に、
将来を考えすぎて細かく分けると、
今度は複雑になる。

この辺りは今でもかなり迷います。

なので最近は、

「まずは利用側が読みやすいか」

をかなり重視しています。

例えば、

```csharp
reportExporter.Export(report);
```

を見た時に、

「これでレポート出力するんだな」

がすぐ分かる。

まずそこを優先する。

その上で、

* 本当に変更されそうか
* 差し替え需要があるか
* 呼び出し側が知る必要があるか

を見ながら、必要なら分割する。

最初から全部抽象化しきろうとしない。

今はそんな感じで考えることが多いです。

## インターフェースは「接続部分」だから迷う

インターフェースって、
結局「つなぐ場所」なんですよね。

だから、
ここがガチガチに結合すると、
変更時に影響が広がる。

逆に、
抽象化しすぎると、
何をしているか見えなくなる。

このバランスが本当に難しい。

今でも、
「この粒度で良かったのか」
は後からよく悩みます。

ただ、以前より少しだけ意識するようになったのは、

* 利用側が何を知るべきか
* 何を隠したいか
* どこを差し替えたいか
* どこを固定したいか

を考えながら、
"インターフェースを道具として使う"

という感覚でした。

綺麗に分割すること自体が目的ではなくて、

「変更時に困りにくいか」
「呼び出し側が理解しやすいか」

を見ながら、
ちょうど良い落とし所を探していく。

今のところは、その感覚で少しずつ整理しています。
