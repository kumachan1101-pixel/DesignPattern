# 【ソフトウェア設計】非同期処理を分離するというより、"責任"を分けたかった

## 同期と非同期をどう設計するかで、ずっと迷っていました

同期処理だけなら、まだ追いやすいです。

ボタンを押したら処理が動いて、
結果が返ってきて、
画面が変わる。

流れが一直線なので、
コードを読んでも追いやすいです。

ただ、実際には非同期が入ってきます。

ファイル保存後に通知が飛ぶとか、
別の処理があとで動くとか、
状態があとから変わるとか。

このあたりから、
「どこで何が変わるのか」が急に見えづらくなりました。

特にしんどかったのは、

「この処理は誰が責任を持っているのか」

が曖昧になることでした。

---

## 非同期を意識し始めると、全部が壊れ始める感覚があった

以前の自分は、
非同期を意識しすぎていました。

「あとで来るかもしれない」
「まだ終わってないかもしれない」
「順番が前後するかもしれない」

みたいなことを、
呼び出し側が全部考え始める。

するとコードが急に複雑になります。

```csharp
if (task.IsCompleted)
{
    ShowResult();
}
else
{
    ShowLoading();
}
```

最初はこれくらいでも、
だんだん条件が増えていきます。

```csharp
if (task.IsCompleted && notificationArrived)
{
    ShowResult();
}
else if (task.IsRunning)
{
    ShowLoading();
}
else if (task.IsFailed)
{
    ShowError();
}
```

さらに別の通知が増える。

状態が増える。

タイミング差も増える。

そのうち、

「何を条件にこの画面が変わるんだっけ？」

が整理できなくなっていきました。

---

## 非同期そのものを見せない方が整理しやすかった

途中から少し考え方が変わりました。

「非同期であること」を、
利用側があまり知らない方がいいのでは、
と思うようになりました。

利用側は、

- 状態が変わった
- 通知が来た
- 結果が確定した

みたいなことだけを見る。

「それが同期なのか非同期なのか」は、
なるべく隠す。

イメージとしては、

「こうなったら、こう振る舞う」

だけに寄せる感覚です。

---

## 状態変化を中心にすると少し整理しやすかった

例えばファイルアップロードを考えます。

最初はこういう感じでした。

```csharp
public class UploadService
{
    public void Upload(File file)
    {
        Save(file);
        SendNotification();
        CreateThumbnail();
        UpdateScreen();
    }
}
```

同期で全部つながっています。

ただ実際には、
サムネイル生成だけ時間がかかったりします。

すると途中から、
「あとでやる」
が入り始めます。

その時に、
呼び出し側まで非同期を意識し始めると、
かなり崩れやすかったです。

なので途中から、
状態変化だけを扱うように寄せました。

```csharp
public class UploadService
{
    public void Upload(File file)
    {
        Save(file);
        ChangeStatus(file, FileStatus.Uploaded);
    }
}
```

別の側が状態を見て動く。

```csharp
public class ThumbnailWorker
{
    public void OnUploaded(File file)
    {
        CreateThumbnail(file);
        ChangeStatus(file, FileStatus.ThumbnailCreated);
    }
}
```

画面側も、
「非同期かどうか」ではなく、
状態だけを見る。

```csharp
if(file.Status == FileStatus.ThumbnailCreated)
{
    ShowThumbnail();
}
```

こうすると、
責務の境界が少し整理しやすかったです。

---

## 「非同期を頑張る人」と「結果を使う人」を分ける

自分の中では、

- 非同期で頑張る側
- 結果だけ利用する側

を分けるイメージがかなり大きいです。

非同期側は、

- 後で実行する
- 通知する
- 状態を変える
- リトライする

みたいな責任を持つ。

利用側は、

- 状態を見る
- 通知を受ける
- 今どうなっているか判断する

だけに寄せる。

この分離をしないと、
利用側に非同期の都合が漏れてきて、
条件分岐がどんどん増える感覚がありました。

---

## 条件分岐が増え始めたら、設計が崩れ始めている気がする

最近は、

「この条件分岐は、何を隠しきれていないんだろう」

をかなり見るようになりました。

特に、

```csharp
if (isAsyncCompleted)
```

みたいな条件が増え始める時。

もちろん必要なこともあります。

ただ、
いろんな場所でそれを見始めると、
非同期の事情がシステム全体に漏れ始めている感覚があります。

この状態になると、
同期・非同期・タイミング差が全部混ざって、
かなり追いづらくなりました。

---

## まとめ

- 利用側が「非同期かどうか」を意識しすぎると、条件分岐が増えて崩れやすい
- 状態変化・通知・結果に責任を寄せると、境界が整理しやすくなる
- 「非同期で頑張る側」と「結果を使う側」を分けることが、設計の出発点かもしれない

---

## おわりに

正直、
同期と非同期の設計は今でも難しいです。

どこまで状態を持つか、
通知を使うか、
状態参照に寄せるか、
どこで同期を待つか。

毎回迷います。

ただ、

「利用側が非同期をどれだけ意識しているか」

を見ると、
少し整理しやすくなる感覚があります。

非同期そのものを広げるのではなく、
状態変化・通知・結果に責任を寄せる。

非同期で頑張る側は、
その役割だけに集中する。

最近はそういう分け方を意識することが増えました。

まだ試行錯誤中ですが、
少なくとも、
「みんなが非同期を意識し始める」
状態よりは、
かなり読みやすくなった感覚があります。
